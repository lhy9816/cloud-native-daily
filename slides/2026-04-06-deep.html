📦 K8s v1.36 Breaking Changes 详解

Breaking2026-03-30
Kubernetes 官方博客 · v1.36 beta.0 已于 3/20 发布
⚠️ Ingress NGINX 退役（3/24）
SIG Network + 安全响应委员会联合宣布停止维护。不再发布任何版本、不再修复任何安全漏洞。现有部署可继续运行，但 Helm Chart 和镜像不再更新。
→ 官方建议迁移到 Gateway API 或其他 Ingress Controller
⚠️ Service .spec.externalIPs 废弃（KEP-5707）
该字段自 K8s 早期就存在，长期被标记为安全隐患（CVE-2020-8554：中间人攻击）。v1.36 起使用会触发废弃警告，v1.43 将完全移除。
→ 替代方案：LoadBalancer Service / Gateway API / NodePort
⚠️ gitRepo Volume 永久禁用（KEP-5040）
自 v1.11 起已废弃。存在严重安全漏洞：攻击者可通过 gitRepo 以 root 权限在节点执行代码。v1.36 起插件永久禁用且无法重新启用。
→ 替代方案：Init Container + git-sync 工具

📦 K8s v1.36 值得关注的新特性

Enhancements

⚡ SELinux 卷挂载加速 — GA

用 mount -o context=XYZ 替代递归文件重新标记，显著减少 Pod 启动延迟。自 v1.28 beta，v1.32 增加指标和退出选项，v1.36 默认对所有卷生效。

⚠️ 注意：混合 privileged 和 unprivileged Pod 共享卷时可能导致冲突，Pod 作者需正确设置 seLinuxChangePolicy

🔑 ServiceAccount Token 外部签名 — GA

kube-apiserver 可将 token 签名委托给外部系统（云 KMS、HSM 等）。对依赖集中式密钥管理基础设施的集群提升安全性和密钥管理便利性。

🎯 DRA 设备 Taint/Toleration 支持

v1.33 引入 DRA 设备 Taint 机制。v1.36 进一步增强驱动支持，允许设备管理器为特定设备设置 Taint，调度器可据此实现精细化的设备级调度控制。

→ 这对 GPU 调度意义重大：可将故障 GPU 或专用 GPU 标记 Taint，防止普通工作负载调度上去

📦 Ingress NGINX 退役：为什么 & 怎么办

深度分析

📢 退役原因

  • 长期存在未修复的安全漏洞
  • SIG Network 和安全响应委员会联合决定
  • 维护资源不足，无法满足安全标准
  • Ingress API 本身设计有局限性（单一入口、缺少高级路由）

📊 影响评估

  • 全球使用最广泛的 Ingress Controller
  • 大量 Helm Chart、教程、CI/CD 模板依赖
  • 不影响现有部署运行，但无安全更新

🔄 推荐迁移方案

  • Gateway API — 官方标准，Ingress2Gateway 工具辅助迁移
  • Envoy Gateway — 基于 Envoy，Gateway API 原生实现
  • Higress — 刚加入 CNCF，内置 AI 推理路由能力
  • Cilium — eBPF 数据面，高性能 + 安全
  • Istio — 服务 Mesh，适合微服务场景

⏰ 时间窗口

  • 现有部署：继续运行，但无安全补丁
  • 建议：尽快评估迁移方案
  • Gateway API 已有成熟迁移工具
这是一次教科书级的社区治理案例。Ingress NGINX 是 K8s 生态中使用最广泛的组件之一，但安全委员会宁可"强制退休"也不妥协。这说明社区对安全问题的态度越来越零容忍。对调度层的影响：新的 Gateway 方案（特别是 Higress 的 AI 路由能力）会改变流量入口的资源需求模型。

🤖 云原生模型分发：OCI-Native 全链路方案

解决方案

核心理念：用软件交付的最佳实践管理模型 — 版本化、不可变、GitOps、可审计。

CI/CD Pipeline │ ├─ 1️⃣ modctl modelfile generate → 自动生成 Modelfile ├─ 2️⃣ modctl build → 打包为 OCI Artifact（含 ARCH/FAMILY/FORMAT 元数据） ├─ 3️⃣ modctl push → Harbor → 推送到 Artifact Registry ├─ 4️⃣ Dragonfly P2P → 集群内节点互相分发 └─ 5️⃣ K8s OCI Volume / CSI Driver → 模型挂载到推理容器

📦 ModelPack + modctl

CLI 工具，将模型打包为标准 OCI Artifact。生成的 Manifest 包含模型元数据（media type: application/vnd.cncf.model.manifest.v1+json），包含 ARCH、FAMILY、FORMAT 等字段，与容器镜像共用同一套 Registry 基础设施。

🐘 ORAS

OCI Registry as Storage — 将模型作为 OCI Artifact 存储。复用现有 Registry（Harbor），支持版本管理、签名、RBAC。

🐉 Dragonfly

P2P 分发，集群内节点互相传输模型数据。带宽利用率接近线性增长，避免所有节点同时从 Registry 拉取造成带宽瓶颈。

这个方案的精妙之处在于完全复用了容器交付的基础设施。模型不再是"放在 S3 上的大文件"，而是与容器镜像享有同等生命周期管理的"一等公民"。对调度器来说，这意味着模型拉取可以像镜像拉取一样被优化和缓存。

🖥️ KubeVirt v1.8：VM 场景的 AI 性能突破

SIG Compute 对齐 K8s v1.35

🔥 PCIe NUMA 拓扑感知（AI/HPC 重点）

KubeVirt v1.8 引入 PCIe NUMA 拓扑感知，使 VM 中的 AI/HPC 工作负载获得接近裸金属的性能。GPU 通过 PCIe 总线连接 CPU 和内存，跨 NUMA 节点访问的延迟高 2-3 倍。在多 GPU 服务器上（如 8×A100/H100），如果 VM 的 vCPU 和内存不在 GPU 所在的 NUMA 节点上，推理延迟会显著增加。

🔐 Intel TDX 机密计算认证

Confidential Computing Working Group 引入 Intel TDX Attestation 支持。VM 可证明自己运行在可信的机密硬件上。对 AI 模型保护场景意义重大 — 模型权重在 VM 内运行时对云厂商不可见。

🧩 Hypervisor 抽象层

新增 Hypervisor Abstraction Layer，KubeVirt 可集成 KVM 以外的虚拟化后端（如 QEMU、Firecracker），同时保持 KVM 为默认行为。为未来多云虚拟化场景铺路。

💾 增量备份（CBT）+ ContainerPath Volume

基于 QEMU/libvirt 的 Changed Block Tracking 增量备份，存储无关，不依赖特定 CSI 驱动。只捕获修改数据，大幅减少备份窗口和存储占用。ContainerPath Volume 允许将容器路径映射为 VM 存储，提升可移植性。

📊 性能基准数据（SIG Scale）

KWOK 测试扩展至 8000 VMI。virt-api 内存从 100→170 VMI 增长 30MB（3.89 KB/VMI），virt-controller 增长 1335MB（173 KB/VMI）。控制面在 VMI 规模增长时表现稳定。

🛡️ Kubescape 4.0：运行时安全 GA + AI Agent 安全

安全 CNCF

🔍 运行时威胁检测 GA

基于 CEL（Common Expression Language）规则引擎，实时监控三类事件：系统交互（进程、Linux capabilities、系统调用）、网络连接（TCP/UDP/HTTP 事件）、存储访问（文件系统活动）。规则和 RuleBinding 作为 K8s CRD 管理，告警可导出到 AlertManager、SIEM、Syslog、HTTP Webhook。

💾 Storage GA — Aggregated API

利用 K8s Aggregated API 作为安全元数据的集中存储层。Application Profiles、SBOM、漏洞 Manifest 等数据不存入 etcd，避免压垮核心数据库。已在大规模高密度集群验证。

🤖 AI Agent 安全：双面盾牌

赋能 AI Agent：通过 KAgent-native 插件，AI Agent 可直接扫描集群安全态势 — 漏洞清单、RBAC 配置、运行时行为（系统调用、文件访问、网络通信）。

监管 AI Agent：为 KAgent（CNCF Sandbox 项目）引入 42 个安全配置检查点 + 15 个 Rego 规则，检测空安全上下文、缺失 NetworkPolicy、过度权限等问题。

🏗️ 架构简化

移除 host-sensor（DaemonSet 弹出式扫描），将 host-agent 能力直接集成到 node-agent。从两个高权限组件合并为一个，审计更清晰，运维更简单。

🤖 Agent Sandbox：K8s 官方 AI Agent 托管方案

SIG Apps 重磅发现
Kubernetes 官方博客 · SIG Apps 正在开发中
核心问题：AI Agent 正从"短命无状态推理"转向"长驻有状态自主执行"。它们需要持久身份、安全沙箱（执行不可信代码）、长时间空闲后快速唤醒。传统 K8s 原语（StatefulSet+PVC+Headless Service）无法优雅地表达这种工作负载模式。

📦 Sandbox CRD — 三大核心能力

  • 强隔离运行时 — 原生支持 gVisor 和 Kata Containers，为不可信代码提供内核级和网络级隔离
  • 生命周期管理 — 支持缩放到零（Scale to Zero），空闲时释放资源，需要时快速恢复到之前的状态
  • 稳定网络标识 — 每个 Sandbox 有稳定的 hostname 和网络身份，支持多 Agent 协同发现和通信

⚡ SandboxWarmPool — 消除冷启动

传统 Pod 启动约 1 秒开销，对 Web 服务可接受，但对 AI Agent 的交互连续性是致命的。SandboxWarmPool 维护预配置的 Sandbox Pod 池，用户通过 SandboxClaim + SandboxTemplate 即刻获取预热环境。冷启动时间从 ~1s 降至 ~0。

这是本期最重要的发现之一。SIG Apps 专门为 AI Agent 设计了新的 CRD，说明 K8s 社区已经认可 AI Agent 是一种独立的工作负载类型，而不仅仅是"特殊的 Pod"。SandboxWarmPool 的设计思路（预热池 + 即时分配）与 GPU 推理的 Model Serving 有异曲同工之妙。

🔄 Image Promoter 重写：9 阶段渐进式工程

工程实践 42 位贡献者 · 3500+ commits

kpromo 负责将镜像从 staging 推到 registry.k8s.io，签名 + 复制到 20+ 区域镜像。如果它坏了，K8s 无法发布。

"没人注意到"的重写才是最好的重写。9 个阶段、每个阶段独立 review 和验证，最终在生产环境无缝切换。这种渐进式重写策略值得借鉴 — 不追求"大爆炸"式重写，而是让每一步都可以独立回滚。

共同特征：这些淘汰都源于安全问题，而非功能不足。社区正在用"安全不达标就淘汰"的标准重新审视每个组件。

🍒 樱桃的深度总结

个人观点
洞察 1：模型分发是 AI 调度的"第一公里"瓶颈
不是调度算法不够好，而是模型还没到位。140GB 的模型文件让 Pod 在 Init 阶段干等 30 分钟。ModelPack + Dragonfly 的 OCI-native 方案是最务实路径 — 复用容器基础设施，零学习成本。建议任何在做 GPU 推理的团队立即评估 Dragonfly。
洞察 2：NUMA 感知正在从"优化"变成"必需"
KubeVirt v1.8 的 PCIe NUMA 拓扑感知 + DRA 设备 Taint，说明 K8s 核心正在内化硬件感知调度能力。对 8×GPU 服务器的在线推理场景，跨 NUMA 访问的延迟差异（2-3 倍）直接决定用户体验。调度器必须原生支持 NUMA 拓扑。
洞察 3：AI Agent 是 K8s 的新工作负载类型
Agent Sandbox 不是又一个 CRD，而是 SIG Apps 对"AI Agent = 新工作负载类型"的正式认可。长驻、有状态、单例、可缩放到零 — 这些特征与 Web Server 完全不同。未来 K8s 调度器可能需要为 Agent 工作负载设计专门的调度策略。
洞察 4：安全零容忍正在重塑生态
Ingress NGINX 退役、gitRepo 永久禁用、host-sensor 移除 — 社区在用行动表明"安全不达标就不留"。这种文化对 AI 推理场景尤其重要，因为模型文件本身就是高价值攻击目标。
