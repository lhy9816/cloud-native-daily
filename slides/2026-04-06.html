📦 Ingress NGINX 退役意味着什么？

樱桃分析

📢 影响范围

  • 全球数百万集群直接受影响
  • 大量 Helm Chart / manifests 需要改写
  • 社区文档、教程、培训材料需要更新
  • Ingress2Gateway 1.0 提供迁移工具

🔄 迁移方向

  • Gateway API — 官方推荐的下一代标准
  • Higress — 刚加入 CNCF 的 AI 网关
  • Envoy Gateway / Istio — 服务 mesh 方案
  • Cilium — eBPF 数据面，性能优势
Ingress NGINX 的退役是社区安全治理的一个标志性事件。一个广泛使用的组件因为安全问题被"强制退休"，说明 K8s 社区在安全问题上越来越果断。这对调度层也有间接影响——新的 Gateway 方案会带来不同的资源需求模型，调度器需要适应。

核心问题：模型拉取速度直接卡住 GPU 推理节点的启动时间。
再好的调度策略，如果 Pod 在 Init 阶段干等 30 分钟拉模型，调度器的亲和性优化毫无意义。
  • 传统方案：Git LFS（慢）、S3/MinIO（带宽瓶颈）、NFS/CephFS（运维复杂）
  • 问题本质：模型分发没有沿用软件交付的最佳实践

🤖 云原生模型分发方案

解决方案
CI/CD Pipeline → ModelPack (打包) → ORAS (推送到 Registry) → Harbor (存储 & 版本管理) → Dragonfly (P2P 分发到各节点) → Init Container (挂载模型) → 推理引擎加载
核心理念：将软件交付的最佳实践引入模型管理
→ 版本化、不可变、GitOps、声明式、可审计
  • ModelPack — 模型打包标准（OCI 兼容）
  • ORAS — OCI Registry as Storage，模型用 OCI 格式管理
  • Dragonfly — P2P 分发，集群内节点互相传输，带宽利用率接近线性

🖥️ KubeVirt v1.8 — PCIe NUMA 感知

AI 性能 CNCF

🔧 新功能

  • PCIe NUMA 拓扑感知 — AI/HPC VM 获近原生性能
  • Hypervisor 抽象层 — 支持 KVM 以外的虚拟化后端
  • Intel TDX 机密计算认证
  • 增量备份（CBT）
  • passt 网络绑定晋升为核心组件

⚡ 为什么重要？

  • GPU 通过 PCIe 总线连接 CPU 和内存
  • CPU/GPU 跨 NUMA 节点访问延迟高 2-3 倍
  • 推理延迟敏感场景（在线 serving）影响显著
  • 多 GPU 服务器问题更严重
PCIe NUMA 感知是 GPU 推理性能的基础设施级优化。调度器在分配 GPU 资源时必须考虑 NUMA 拓扑，否则再好的推理引擎也无法发挥硬件的全部性能。KubeVirt 把这个能力带到 VM 场景，对需要 SR-IOV 或直通 GPU 的场景特别实用。

🍒 樱桃的总结与思考

个人观点
1. 模型分发是 K8s AI 调度的"第一公里"问题
我们往往关注调度算法的优化，但忽略了模型拉取这个更基础的瓶颈。Dragonfly + ORAS 的 OCI-native 方案可能是最务实的路径——复用现有容器基础设施，不引入新的系统。
2. NUMA 感知正在从"可选项"变成"必需品"
随着 GPU 推理对延迟的要求越来越高，NUMA 亲和性不再是高级优化，而是基本要求。调度器需要原生支持 NUMA 拓扑感知，而不是依赖外部插件。
3. AI Agent 操作 K8s 是不可逆的趋势
AI Agent 被授权操作集群正在成为范式。安全、审计、权限管理需要跟上。我们不仅是"在 K8s 上跑 AI"，还在"让 AI 管理 K8s"。
