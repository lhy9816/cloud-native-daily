### 1. 📦 Pod 真的是 AI Agent 的正确部署单元吗？kagent 团队的反思

**原文标题:** Is a Pod the right deployment unit for an AI agent?
**来源:** CNCF Blog（作者 Lin Sun, Solo.io）| 2026-07-14
**链接:** https://www.cncf.io/blog/2026/07/14/is-a-pod-the-right-deployment-unit-for-an-ai-agent/

🎯 **一句话精髓**: kagent 项目反思「每个 AI agent 一个 Pod + Service + ServiceAccount」的模型，提出在 K8s 之上再加一层 agent-substrate 控制面，把 agent 的生命周期/放置与 Pod 执行环境解耦。

🔧 **技术栈**: kagent / Kubernetes Pod / ServiceAccount / agent-sandbox / agent-substrate / WorkerPool / ActorTemplate

📐 **架构/方法亮点**:
- Pod 作为部署单元一度解决了隔离、身份、网络策略、可观测与调度——但 agent 是短生命周期、bursty 的（秒到分钟级），常驻 Pod 极度浪费资源
- agent-substrate 引入 Pod 之上的控制面：WorkerPool≈NodePool、Worker≈Node、ActorTemplate≈Pod 的声明式 spec；K8s 继续管 Pod/网络/存储/计算，substrate 管 AI actor 的生命周期与放置
- agent 行为不像长运行服务：可动态创建子 agent 并行执行、可暂停等待人工审批、可代用户执行——这些都超出了传统微服务的生命周期模型

💡 **关键数据**: 文章以架构论证为主，未给出具体性能数字；核心价值在于提出 WorkerPool/Worker/ActorTemplate 三层抽象对标 K8s 原生概念

🔗 **樱桃的调度视角**: 这直接戳中 Master 的方向——当 agent 成为 K8s 调度对象，调度器面对的不再是长运行 Pod，而是「秒级 bursty actor」。这要求调度延迟极低、支持 worker pool 快速弹性扩缩（类似 serverless 冷启动优化），传统基于节点资源快照的调度周期可能跟不上。

### 2. 📦 AI 推理成本首次算清：GPU 利用率不到 52%，千万别自建！

**原文标题:** AI 推理成本首次算清：GPU 利用率不到 52%，千万别自建！
**来源:** InfoQ 中文 | 2026-07-13
**链接:** https://www.infoq.cn/article/hocbRjZumiBA6MN4s7bu

🎯 **一句话精髓**: 揭穿「自建比 API 便宜」的逻辑漏洞——成立前提是利用率≥API 厂商（90%+），而中型 AI 公司往往只有 30%-50%，52% 才是盈亏平衡临界点。

🔧 **技术栈**: H100 / 五维成本拆解模型 / 蒙特卡洛敏感性分析 / GPU 折旧 / NVIDIA 官方数据校验 / Python 可运行代码

📐 **架构/方法亮点**:
- 五维成本拆解 + 利用率盈亏平衡点的数学推导（52% 临界点），附可运行 Python 代码（含蒙特卡洛敏感性分析，NVIDIA 官方数据校验，偏差 4.7%）
- 真实案例：月流水 $50 万的 AI 客服公司，API 调用成本占 36%（$18 万）；8×H100 在 80% 利用率月成本 $16.5 万，但利用率只有 40% 时反成 $19.5 万——比继续用 API 还贵
- API 厂商靠超大规模调度实现 90%+ 利用率，定价策略利用了客户「不知道自己利用率多少」的认知不对称

💡 **关键数据**: 盈亏平衡利用率临界点 52%；自建 40% 利用率月成本 $195,000 vs API $180,000；模型校验偏差 4.7%

🔗 **樱桃的调度视角**: 这正是 GPU 调度的真金白银价值——通过混部、请求排队、continuous batching、PD 分离把利用率从 40% 拉到 80%+，才能让自建跑赢 API。Master 做的每一项调度优化，都在直接决定盈亏线喵～

### 3. 📦 面向 Agentic 负载的下一代 LLM 推理引擎：从「极限吞吐」到「低延迟 + GPU Token 效率」

**原文标题:** 面向 Agentic 负载的下一代 LLM 推理引擎设计实践｜AICon深圳
**来源:** InfoQ 中文（阿里云 TokenSpeed 团队，讲师尚旭春）| 2026-07-14
**链接:** https://www.infoq.cn/article/fjkG0xhaV42S8sfCayFy

🎯 **一句话精髓**: Agentic 推理负载（高并发、短请求、多轮工具调用、逐 Token 生成）正逼迫推理引擎从「追求极限吞吐」转向「保证低延迟的同时实现更高 GPU Token 输出效率」，阿里云从 Runtime 与 Kernel 双层重构。

🔧 **技术栈**: TokenSpeed / SGLang / Mooncake / PD 分离（Prefill-Decode disaggregation）/ EPLB 弹性流水线负载均衡 / KVStore / HiCache / PP 流水线并行

📐 **架构/方法亮点**:
- Runtime 层：PD 分离（Prefill-Decode 解耦）+ EPLB（弹性流水线负载均衡）+ KVStore；Kernel 层做算子级调优，双层协同
- 讲师身兼三大社区 Maintainer：Mooncake Store（KV Cache 池化，GitHub 4k+ star）、SGLang Committer（贡献 PD 分离/PP 并行/HiCache 长文本高并发）、TokenSpeed Maintainer（PD 分离/EPLB/KVStore）
- Agentic 负载特征驱动新设计目标：高并发短请求 + 多轮工具调用 + 逐 Token 生成，与传统聊天场景的吞吐优先截然不同

💡 **关键数据**: Mooncake Store GitHub 4k+ star；AICon 大会聚焦 50+ 头部企业（腾讯/阿里/华为/百度/蚂蚁）实战案例

🔗 **樱桃的调度视角**: PD 分离 + EPLB 是当前 AI 推理调度的前沿形态——把 prefill（计算密集）和 decode（访存密集）调度到不同 GPU 池，再按负载弹性均衡流水线。这正是 Master 在 K8s 上做 GPU 调度要落地的目标架构：两类异构负载、两套调度策略。

### 5. 📦 CNCF 云原生 AI 数据存储白皮书：按训练/推理/Agentic 三阶段刻画存储画像

**原文标题:** The CNCF Data Storage in Cloud Native AI White Paper
**来源:** CNCF Blog（TAG Infrastructure）| 2026-07-08
**链接:** https://www.cncf.io/report-whitepaper/2026/07/08/the-cncf-data-storage-in-cloud-native-ai-white-paper/

🎯 **一句话精髓**: CNCF TAG Infra 发布白皮书，把云原生 AI 数据生态拆成数据湖仓/向量库、缓存与数据本地性、CSI/COSI 标准接口、现代数据管道四层，并按训练/推理/Agentic 三阶段刻画截然不同的存储画像。

🔧 **技术栈**: CSI / COSI / Fluid（K8s 内分布式缓存编排）/ Apache Parquet / Iceberg / Milvus（向量库）/ Kafka / CDC / FUSE CSI / KV Cache / Prefix Cache

📐 **架构/方法亮点**:
- 计算存储分离虽可扩展，但引入重 API 调用开销和低 GPU 利用率；Fluid 在 K8s 内编排分布式缓存以消除数据传输延迟
- 三阶段存储画像：训练=吞吐导向+checkpoint 同步写突发；推理=延迟敏感+spiky 流量+KV/Prefix Cache；Agentic=短/长期记忆+artifact 仓库+闭环迭代推理
- 小文件陷阱：百万级小文件压垮存储元数据服务器，是数据湖训练的常见瓶颈

💡 **关键数据**: 白皮书为结构性指南，未给单一性能数字；核心是三阶段存储画像的差异化设计

🔗 **樱桃的调度视角**: 存储画像直接反哺调度决策——训练任务要调度到缓存命中高的节点（数据本地性/拓扑感知），推理要快速加载模型（镜像/缓存感知调度）。这与 Master 的拓扑感知调度方向高度契合：让数据跟着算力走，而不是算力等数据。

## 📓 樱桃的今日学习总结

1. **AI Agent 正在催生新的 K8s 工作负载抽象** - Pod 太重（常驻浪费资源）、也太慢（秒级 bursty actor 难以快速弹性）。kagent 的 agent-substrate 与 serverless 化是同一趋势的两面，调度器要面向「短生命周期 actor」重塑调度延迟与扩缩模型。
2. **推理优化的重心从「极限吞吐」转向「低延迟 + GPU Token 效率」** - PD 分离 + EPLB + KV Cache 池化（Mooncake）已成标配。调度决策必须区分 prefill（计算密集）和 decode（访存密集）两类异构负载，分池调度、按负载弹性均衡。
3. **自建 vs API 的胜负手在 GPU 利用率（52% 临界点）** - 调度/混部/batching 是把利用率从 40% 拉到 80%+ 的关键杠杆。Master 做的每项调度优化，都在直接决定自建能否跑赢 API 的盈亏线——技术价值即商业价值喵～
