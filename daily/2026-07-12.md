## 🚀 重点速览

### 1. 📦 HAMi-DRA：将 GPU 虚拟化调度从 Device Plugin 迁入 DRA 声明式模型

**原文标题:** 从 HAMi 到 HAMi-DRA：异构环境的算力资源管理实践｜AICon深圳
**来源:** InfoQ 中文 | 2026-07-11
**链接:** https://www.infoq.cn/article/fRMquPuKMsj6zkyFmO6P

🎯 **一句话精髓**: 范式智能（原第四范式）系统研发专家杨守仁（HAMi 社区 maintainer）分享了 HAMi-DRA 的设计实践--在 Kubernetes v1.34 将 DRA 推进至 GA 的背景下，将现有基于 Device Plugin 的 GPU 虚拟化调度体系平滑迁移至 DRA 资源模型，在不牺牲核心调度能力的前提下显著提升调度性能和精细化管理水平。

🔧 **技术栈**: HAMi / HAMi-DRA / Kubernetes DRA (Dynamic Resource Allocation) / ResourceClaim / Device Plugin / GPU 虚拟化 / 异构算力 / NVIDIA / 国产 GPU

📐 **架构/方法亮点**:
- Device Plugin 的根本局限：传统 Device Plugin 接口仅能暴露简单容量信息（如 GPU 数量），无法在 Pod 创建前完成资源的表达与约束，导致调度性能和精细化分配能力面临瓶颈
- DRA 的声明式资源模型：通过引入 ResourceClaim，设备资源可以在 Pod 创建之前完成表达与约束（Consumable Capacity 特性），从「以节点为中心的分配」演进为「以资源对象为核心的声明式管理」
- HAMi-DRA 平滑迁移策略：将现有 GPU 虚拟化调度体系（HAMi 已有的 vGPU 切分/隔离能力）映射到 DRA 资源模型上，保持核心调度能力不丢失，同时获得 DRA 带来的调度性能提升和完善的设备生命周期管理
- 落地痛点：依赖高版本 Kubernetes（v1.34+），生产环境推动困难；DRA API 仍在持续演进，部分特性不够稳定，集成成本高；开箱即用的 DRA Driver 数量较少，支持的异构设备有限

💡 **关键数据**: Kubernetes v1.34 将 DRA 正式推进至 GA；HAMi 是范式智能参与发起的异构算力管理项目；HAMi-DRA 在调度性能和精细化管理方面带来显著提升（具体数据待 AICon 演讲披露）；范式实践中国产算力从单一 NVIDIA 设备逐步走向多厂商异构集群。

🔗 **樱桃的调度视角**: 这篇对 Master 的价值极高--HAMi-DRA 是目前看到的第一个将生产级 GPU 虚拟化体系实际迁移到 DRA 模型的公开实践。DRA GA 意味着 Device Plugin 终将被替代，但迁移路径的痛点（高版本依赖、Driver 生态匮乏）是 Master 必须提前规划的。HAMi-DRA 的「不牺牲核心调度能力的平滑迁移」策略值得关注：它没有推翻 HAMi 原有的 vGPU 切分逻辑，而是用 DRA 的 ResourceClaim 替换了底层的资源声明层，这种「接口替换、能力保留」的迁移思路适用于任何从 Device Plugin 迁向 DRA 的场景喵。

### 2. 📦 vLLM 多模态推理优化：Encoder 多级缓存 + ViT 数据并行 + Cuda Graph

**原文标题:** vLLM 针对多模态模型的推理优化实践｜AICon深圳
**来源:** InfoQ 中文 | 2026-07-10
**链接:** https://www.infoq.cn/article/ItOOYNuWf6t9KtfgeH7F

🎯 **一句话精髓**: vLLM 核心开发者莫梓峰（Inferact）分享了 vLLM 针对多模态大语言模型（如 Kimi-K2.6、Qwen3.6、cosmos3）的推理优化实践--核心挑战是 Continuous Batching 框架在处理多模态输入（图片/视频 token）时，Encoder 计算（如 ViT）成为推理流水线的瓶颈，vLLM 通过 Encoder Cache、Processor Cache 两级缓存和 ViT DP、Encoder Cuda Graph 两项优化显著缓解了这一问题。

🔧 **技术栈**: vLLM / PagedAttention / Continuous Batching / Encoder Cache / Processor Cache / ViT Data Parallelism / Encoder Cuda Graph / Kimi-K2.6 / Qwen3.6 / cosmos3

📐 **架构/方法亮点**:
- 多模态推理在 Continuous Batching 下的挑战：传统 Continuous Batching 针对文本 token 的 decode 优化，但多模态模型需要先经过 Vision Encoder（如 ViT）处理图片/视频输入，Encoder 计算密集且耗时，打破了 batching 的均匀性假设
- 两级缓存设计：Encoder Cache 缓存已计算的 Encoder 输出（避免相同图片重复编码），Processor Cache 缓存预处理结果（如图片预处理后的 tensor），减少重复计算
- ViT DP（数据并行）：将 Vision Encoder 的计算分布到多个 GPU 上并行执行，解决单 GPU encoder 计算瓶颈
- Encoder Cuda Graph：将 Encoder 的固定计算图预编译为 Cuda Graph，减少 kernel launch 开销，提升单次编码效率

💡 **关键数据**: vLLM 截至 2026 年 6 月 GitHub 获 82k star；被 AWS、Meta、Digital Ocean 等大型厂商用于生产环境；多模态大语言模型已成为发展趋势（Kimi-K2.6、Qwen3.6、cosmos3 均支持原生多模态）。

🔗 **樱桃的调度视角**: 多模态推理优化直接关系到 Master 的 GPU 调度策略。Encoder 计算密集意味着多模态推理 Pod 的 GPU 资源使用模式和纯文本推理完全不同--Encoder 阶段是 compute-bound（需要高 FLOPS），Decode 阶段是 memory-bound（需要高 HBM 带宽），调度器需要感知这种阶段差异。ViT DP 将 encoder 计算分布到多 GPU，本质上是「推理内部的并行调度」，和 Prefill/Decode 分离一样都是把推理流水线拆分为可独立调度的阶段。Encoder Cache 的思路也值得借鉴：如果多个请求引用同一张图片（如 RAG 场景），缓存 encoder 输出可以大幅降低 GPU 负载喵。

### 3. 📦 Airbnb K8s 动态配置 Sidecar Sitar-agent：S3 快照引导 + SQLite 本地存储 + 拉取模型

**原文标题:** Airbnb 分享 Kubernetes 动态配置 Sidecar Sitar-agent 的架构
**来源:** InfoQ 中文 | 2026-07-11
**链接:** https://www.infoq.cn/article/fO5byVPuZwwlBPosijBV

🎯 **一句话精髓**: Airbnb 详细介绍了其 Kubernetes Sidecar Sitar-agent 的架构设计--该 sidecar 能将动态配置更新分发到数万个 Pod 中，每分钟处理多次配置变更，通过 Java 重写、Amazon S3 快照引导启动、从 Sparkey 迁移至 SQLite 三项关键改造，实现了高可靠、低启动开销、去中心化的配置分发。

🔧 **技术栈**: Kubernetes Sidecar / Sitar-agent / Amazon S3 / SQLite / Sparkey / RocksDB / 共享文件系统 / 内存缓存 / 功能标志 / 影子读取 / Helm

📐 **架构/方法亮点**:
- Sidecar 模式 vs 库模式：Airbnb 选择 sidecar 而非将配置分发逻辑嵌入应用程序库（库模式资源消耗更低），理由是 sidecar 简化运维、减少跨语言重复实现（Java/Python/Go/TypeScript/Ruby 五种语言）、确保行为一致性
- S3 快照引导启动：系统定期生成配置快照存入 S3，新 Pod 启动时 Sitar-agent 先拉取最新快照（而非等待轮询），然后从后端同步增量更新，之后应用才开始处理流量--降低启动开销且确保中央配置服务不可用时仍可启动
- SQLite 替换 Sparkey：评估了 SQLite 和 RocksDB 后选择 SQLite，原因是并发模型简单、操作简便、对语言生态系统支持广泛；迁移采用影子读取验证 + 功能标志控制的分阶段部署策略降低风险
- 拉取模型（Pull-based）：Sitar-agent 每约 10 秒轮询后端服务，配合服务器端缓存和增量变更跟踪减轻后端负载

💡 **关键数据**: Sitar-agent 服务数万个 Pod；每分钟处理多次配置变更；更新在数十秒内传播到整个微服务集群；配置轮询间隔约 10 秒；支持 Java/Python/Go/TypeScript/Ruby 五种语言。

🔗 **樱桃的调度视角**: S3 快照引导启动的模式对推理服务有直接借鉴价值--模型加载也可以用类似策略：新 Pod 启动时先从对象存储（S3/MinIO）拉取模型快照，再同步增量权重更新，避免每次从中央模型仓库全量拉取。SQLite 作为 sidecar 本地存储的选择也值得注意：在 K8s 环境中，SQLite 比 RocksDB 更轻量，不需要额外的编译依赖，适合作为 sidecar 的嵌入式存储。拉取模型的 10 秒轮询间隔对配置分发是合理的，但对推理服务的权重热更新可能需要更短间隔或推送模型喵。

### 4. 📦 HubSpot VaaS：200 亿向量的语义搜索平台从 Helm 迁移到 K8s Operator

**原文标题:** HubSpot 如何将语义搜索扩展至 200 亿个向量
**来源:** InfoQ 中文 | 2026-07-10
**链接:** https://www.infoq.cn/article/eRl25z5ewrRHjReWhOUX

🎯 **一句话精髓**: HubSpot 详细介绍了其语义搜索平台 VaaS（向量即服务）如何从概念验证发展为内部平台服务--管理 200+ 亿向量、200+ 索引、140+ 集群、5 个区域，核心改造是从 Helm 迁移到内部 Kubernetes Operator 框架，通过 Translators 组件每 60 秒对账目标状态与实际状态，实现集群创建/退役、分片迁移、副本恢复的全面自动化。

🔧 **技术栈**: Qdrant / VaaS (Vector as a Service) / Kubernetes Operator / Helm / 命名向量 / 混合搜索 / 多阶段查询 / 加权重排 / 量化处理 / 磁盘存储 / Translators 对账模型

📐 **架构/方法亮点**:
- Helm 到 Operator 迁移动因：Helm 无法进行 API 调用、无法根据外部指标自动扩展、无法处理状态感知的生命周期任务（如分片迁移、副本恢复），随着集群规模扩大手动管理不可持续
- Translators 对账模型：每 60 秒将系统目标状态与实际状态同步一次，自动化处理水平扩展、分片再平衡和复制恢复，同一套对账逻辑覆盖三种运维场景
- 选择 Qdrant 的理由：支持本地部署（数据主权）、命名向量、混合搜索、多阶段查询、加权重排、量化处理和磁盘存储（成本控制），能与内部追踪、成本监控、速率限制、安全工具集成
- 分片放置与内存均衡：数十亿数据点的场景下，即使一个分片失衡也会迫使集群提前扩展，因此分片放置策略至关重要

💡 **关键数据**: 200+ 亿向量；38+ 团队使用；200+ 索引；140+ 集群；5 个区域；2 个环境；峰值写入 10 万次/秒；集群启动时间从数小时缩短至数分钟；消除对备用集群的需求。

🔗 **樱桃的调度视角**: 「从 Helm 迁移到 Operator」是 AI 基础设施在 K8s 上的必经之路。向量数据库（Qdrant）、推理引擎（vLLM/TGI）、模型仓库都需要有状态感知的生命周期管理，Helm 的模板化部署无法胜任。HubSpot 的 Translators 对账模型本质上是 K8s Controller 的 reconcile loop 思路推广到了运维场景。对 Master 来说，如果要在 K8s 上运营向量搜索或推理服务，自定义 Operator + 对账循环是标准架构。10 万次/秒的峰值写入和 140+ 集群的规模也说明向量搜索的基础设施复杂度不亚于传统数据库喵。

### 5. 📦 AI 工作负载应该跑在哪里？主权视角下的 K8s 多环境部署策略

**原文标题:** Where should AI workloads run? A sovereign and sensible approach
**来源:** CNCF Blog | 2026-07-10
**链接:** https://www.cncf.io/blog/2026/07/10/where-should-ai-workloads-run-a-sovereign-and-sensible-approach/

🎯 **一句话精髓**: KubeOps 的 Johannes Hemminger 和 Martin Hafner 提出从「数字主权」视角审视 AI 工作负载部署--不是每个 AI 任务都需要前沿模型，常规/重复/窄域任务可用开源权重模型在私有云或本地运行，关键是「构建选择而非猜测完美目的地」，通过 K8s 可移植性避免建造死路。

🔧 **技术栈**: Kubernetes / 数字主权 (Digital Sovereignty) / 开源权重模型 / 本地部署 / 私有云 / 可移植性 / AI Readiness Check / CNCF 生态

📐 **架构/方法亮点**:
- 数字主权五要素：① 运营自主性（控制系统所有元素）② 合规性（法律确定性，如数据存储位置）③ 可审计性（安全性和主权可验证）④ 可移植性（避免技术/合同/组织依赖，防止突然涨价）⑤ 弹性（冗余、恢复机制、稳健流程）
- AI 工作负载分层策略：前沿模型用于需要最高智能的任务（可外包为 "bring your own model"）；常规/重复/窄域任务用开源权重模型或旧版本模型，涉及敏感数据/内部流程/受监管操作时部署到本地或私有云
- AI Readiness Check 清单：在将 AI 工作负载部署到任何平台前，检查加速器容量、存储性能、数据本地性、网络隔离、身份集成、监控、备份恢复、软件供应链、漏洞管理和策略执行
- 「构建选择」原则：工作负载应可移植、操作可复现、安全可执行、迁移可行、成本可见，平台无论下一个工作负载落在本地、私有环境、边缘还是多云都应保持有用

💡 **关键数据**: 文章来自 KubeOps（德国公司），服务于公共部门和关键基础设施客户；K8s 因资源管理、自动化、可移植性和操作一致性成为 AI 基础设施的通用基座。

🔗 **樱桃的调度视角**: 这篇的「AI Readiness Check」对 Master 的推理平台建设有实操价值--在部署 GPU 推理集群前，需要系统检查加速器容量（GPU 型号/数量/驱动版本）、存储性能（模型权重 IO 带宽）、数据本地性（训练/推理数据位置）。「构建选择」原则也意味着不要把推理服务绑死在单一云厂商或单一 GPU 架构上，DRA 的 ResourceClaim 抽象正是实现这种可移植性的关键--通过声明式资源需求而非硬编码设备 ID，工作负载可以在不同 GPU 供应商间迁移喵。

### 6. 📦 Cloudflare Meerkat：用 QuePaxa 共识算法替代 Raft 实现全球强一致

**原文标题:** Introducing Meerkat: an experiment in global consensus
**来源:** Cloudflare Blog | 2026-07-08
**链接:** https://blog.cloudflare.com/meerkat-introduction/

🎯 **一句话精髓**: Cloudflare Research 团队正在构建名为 Meerkat 的全球共识服务，采用 2023 年发表的 QuePaxa 共识算法替代 Raft--核心差异是 QuePaxa 无 Leader（所有副本随时可写）且不依赖超时判断进度，从根本上解决了 Raft 在广域网中因 Leader 故障和超时配置困难导致的不可用问题，这将是 QuePaxa 在全球规模的首次工业部署。

🔧 **技术栈**: QuePaxa 共识算法 / Raft / Meerkat / 线性一致性 (Linearizability) / KV Store / 分布式租约/锁 / 2f+1 容错 / Cloudflare 330+ 数据中心

📐 **架构/方法亮点**:
- Raft 在广域网的致命缺陷：Raft 依赖 Leader 和超时--Leader 是唯一可写副本，崩溃或网络退化时系统不可用直到其他副本超时并选出新 Leader；超时值在不可预测延迟的网络中难以配置，Cloudflare 已因 Leader 不可用导致多次事故
- QuePaxa 的无 Leader 设计：所有副本随时可执行写入，进度永不因超时而停滞；只要多数副本存活且可通信（2f+1 系统容忍 f 个故障），任何数据中心的客户端都可读写
- Meerkat 架构：开发者为服务请求 Meerkat 副本集群，每个副本参与共识算法且可接收读写请求；开发者可指定哪些数据中心托管副本，Meerkat 自动放置；上层构建事务性 KV 存储和租约系统
- 应用场景：管理控制面状态（如 AI 模型实例的位置信息、数据库的 Leader 信息），初始仅限 Cloudflare 内部使用

💡 **关键数据**: Cloudflare 运营 330+ 全球数据中心；Meerkat 是 QuePaxa 在全球规模的首次工业部署；QuePaxa 发表于 2023 年；Meerkat 目前仍为实验性内部服务。

🔗 **樱桃的调度视角**: Meerkat 对 K8s 调度有间接但深远的启示。K8s 的 etcd 基于 Raft，在多区域部署时面临和 Cloudflare 完全相同的 Leader 超时问题。如果 QuePaxa 在全球规模验证成功，可能催生 etcd 的替代共识层或影响 K8s 多集群控制面的设计。更直接的是，Meerkat 管理的「AI 模型实例位置信息」本质上就是调度元数据--在多区域推理集群中，哪个区域有哪个模型的副本、哪个实例可用，这些控制面状态需要强一致性保证。一个无 Leader 的共识服务可以让任何区域都能立即写入调度决策，而不必等待 Leader 切换喵。

## 📓 樱桃的今日学习总结

1. **GPU 调度正从「设备插件时代」迈向「声明式资源时代」，但迁移成本是真实痛点** - HAMi-DRA 的实践表明，DRA GA 并不意味着一夜之间迁移，而是「接口替换、能力保留」的渐进式迁移。Device Plugin 的容量暴露能力太粗糙，DRA 的 ResourceClaim 允许在 Pod 创建前表达资源约束，这是质的飞跃。但高版本 K8s 依赖、DRA API 持续演进、Driver 生态匮乏三大痛点意味着 2026 年下半年仍将是 Device Plugin 和 DRA 并存的过渡期，调度器需要同时支持两种模型。

2. **多模态推理打破了 Continuous Batching 的均匀性假设，推理流水线正在被拆分为可独立优化的阶段** - vLLM 的 Encoder Cache/Processor Cache 两级缓存和 ViT DP/Encoder Cuda Graph 优化表明，多模态推理不再是单一的 decode 流水线，而是「Encoder 计算 + Decode 生成」两阶段流水线，每个阶段有不同的计算特征（compute-bound vs memory-bound）和不同的优化策略。这和 Prefill/Decode 分离的思路一致--推理正在被拆分为更细粒度的可独立调度和优化的阶段。

3. **AI 基础设施的「Operator 化」是规模化运营的必经之路** - HubSpot 从 Helm 迁移到 K8s Operator 的故事，和 Airbnb 的 Sitar-agent sidecar 架构，共同指向一个趋势：当 AI 基础设施（向量数据库、推理引擎、配置分发）需要管理 100+ 集群或数万个 Pod 时，Helm 模板和手动运维不再可行，必须有状态感知的 Operator + 对账循环来自动化生命周期管理。K8s Controller 的 reconcile loop 模式正在从「编排 Pod」扩展到「编排 AI 基础设施」。
