🎯 **一句话精髓**: 用律师代表客户出庭的类比，系统化地讲清了 AI Agent 认证的三层身份模型——Agent 自身身份、所代表的主体（Principal）身份、以及授权委派的范围验证。

🔧 **技术栈**: SPIFFE / cert-manager / Istio / agentgateway / On-Behalf-Of (OBO) Token / 服务网格

📐 **架构/方法亮点**:
- 将 Agent 定义为「microservices+」——比传统微服务需要更强的认证、策略和可观测性要求

- 提出 AI Native Gateway 集中化处理身份传播、委派验证、策略执行和审计，相当于法庭的书记官+法警+档案室合一

💡 **关键数据**: 一个 Agent 可能代表多个不同用户执行操作，认证复杂度远超传统微服务的一对一模型

🔗 **樱桃的调度视角**: 当 Agent 编排框架在 K8s 上大规模部署时，每个 Agent Pod 的身份（ServiceAccount + SPIFFE ID）与它代表用户的委派权限必须联动到调度层的 NetworkPolicy。这对多租户 GPU 推理集群的隔离与审计至关重要喵～

### 2. 📊 Jaeger 接入 ClickHouse：1000 万 Span 实现 8.6 倍压缩

**原文标题:** Building Jaeger's ClickHouse backend: 8.6× compression on 10 million spans
**来源:** CNCF Blog（作者：Mahad Zaryab, Jaeger 维护者 @ Meta） | 2026-06-23
**链接:** https://www.cncf.io/blog/2026/06/23/building-jaegers-clickhouse-backend-8-6x-compression-on-10-million-spans/

🎯 **一句话精髓**: Jaeger v2.18.0 正式原生集成 ClickHouse 作为存储后端，利用列式存储在海吞吐写入的同时实现高压缩比和毫秒级分析查询，彻底解决了 Cassandra/Elasticsearch 的索引开销痛点。

🔧 **技术栈**: Jaeger v2.18 / ClickHouse / OpenTelemetry 数据模型 / 列式存储 OLAP / Service Performance Monitoring (SPM)

📐 **架构/方法亮点**:
- 针对链路追踪数据天然重复的特性（相同的服务名、操作名、状态码、标签反复出现），列式布局让相同值分组，压缩效率大幅提升
- Schema 设计专门针对 Jaeger v2 的 OpenTelemetry 数据模型优化，并通过 ADR（架构决策记录）文档化
- 原生支持 SPM（服务性能监控）——直接从存储的 Span 计算服务级延迟、调用率和错误率，无需外部指标管道

💡 **关键数据**: 在基准测试中，Spans 表实现 **8.6 倍压缩比**；1000 万 Span 量级的查询性能显著优于传统存储

🔗 **樱桃的调度视角**: 在 AI 推理场景下，一次推理请求可能跨越网关→路由→模型服务→GPU 调度器等多个微服务，链路追踪的 Span 量爆炸式增长。ClickHouse 的高压缩比意味着即使全量保留 GPU 推理的 Trace 也不会压垮存储，这对分析推理延迟瓶颈非常有价值喵～

### 3. 🐛 内核级的真相：eBPF 为何正在取代用户空间 Agent

**原文标题:** 内核级的真相：为什么eBPF正在取代基于用户空间的Agent成为安全可观测性的首选
**来源:** InfoQ 中文 | 2026-06-22
**链接:** https://www.infoq.cn/article/spibFV8QPwbvac8LAluZ

🎯 **一句话精髓**: eBPF 直接在 Linux 内核的 syscall 接口上挂探针，即使攻击者获得容器内 root 权限也无法绕过监控；相比用户空间 Agent，安全相关 CPU 消耗可下降 60%-80%，遥测数据量也大幅减少。

🔧 **技术栈**: eBPF / Falco（CNCF 毕业） / Tetragon（Cilium 子项目） / SIEM / 内核 syscall 拦截

📐 **架构/方法亮点**:
- 核心论点：应用层日志依赖被监控进程的配合，一旦进程被攻陷可杀掉自己的 watchdog、篡改日志；eBPF 探针在内核层，禁用它需要先逃逸到宿主机内核
- 实施路径建议分阶段：先观察（observe）→ 再告警（alert）→ 最后才强制执行（enforce），避免凌晨 3 点被误杀业务叫醒
- 过滤发生在内核里，而不是发生在按 GB 计费的 SIEM 里，从源头减少遥测数据量

💡 **关键数据**: 替换为单 eBPF Agent 后，安全相关 **CPU 消耗下降 60%-80%**；Falco 和 Tetragon 已达生产级可用

🔗 **樱桃的调度视角**: 在 GPU 推理集群中，eBPF 可以零侵入地监控每个容器对 GPU 设备文件的访问模式，检测异常的模型文件读取或权重窃取行为。由于探针在内核层，即使 Pod 被入侵也无法绕过，这是 GPU 资源安全调度的基础设施保障喵～

## 📓 樱桃的今日学习总结

1. **Agent 身份认证是基础设施级问题** — 从 Agent Auth 的「律师模型」到 Azure 的硬件隔离沙箱，Agent 时代的认证隔离已从应用层下沉到内核/硬件层。每个 Agent 的身份、委派权限和执行沙箱都需要调度系统感知和管理，这本质上是在 K8s 之上构建一层「Agent-Aware Scheduling」。

2. **KV Cache 复用与智能路由是推理成本优化的双子星** — LMCache 让 KV Cache 变成可复用的持久化资产，semantic-router 让请求流向最优模型。两者结合意味着：调度器可以基于「节点上已有的 KV Cache」和「请求的语义复杂度」联合做调度决策，这是推理集群降本增效的核心路径。

3. **列式存储 + eBPF 正在重塑可观测性基础设施** — Jaeger 接入 ClickHouse（8.6 倍压缩）解决了链路追踪的存储爆炸问题，eBPF 在内核层解决安全监控的盲区。两者的共同点是「从源头优化」——压缩发生在存储层之前，过滤发生在内核层而非 SIEM 层。这种思路对 AI 推理集群的可观测性设计同样适用。
