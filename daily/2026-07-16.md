### 1. 📦 HAMi 升入 CNCF 孵化：GPU 切片 + 跨厂商统一调度的标准化里程碑

**原文标题:** HAMi becomes a CNCF incubating project
**来源:** CNCF Blog（HAMi Maintainers）| 2026-07-15
**链接:** https://www.cncf.io/blog/2026/07/15/hami-becomes-a-cncf-incubating-project/

🎯 **一句话精髓**: HAMi 提供云原生 GPU 虚拟化中间件，把物理 GPU（及 NPU/DCU/MLU 等异构加速器）按内存/算力/设备数切片，在 Pod 之间硬隔离共享，并用 binpack/spread/拓扑感知策略调度，全程不改动应用代码与既有 K8s 资源清单。

🔧 **技术栈**: HAMi / Kubernetes Mutating Webhook / Scheduler Extender / Device Plugin / HAMi-Core（容器内 CUDA 驱动劫持）/ Volcano / Koordinator / Kueue / NVIDIA / 国产异构加速器

📐 **架构/方法亮点**:
- 六大组件：Mutating Webhook（拦截 Pod 提交并改写调度字段与资源请求）、Scheduler Extender（binpack/spread/拓扑感知过滤打分绑定）、Device Plugin（厂商插件注册加速器并分配分数设备）、HAMi-Core（容器内虚拟化层，硬限制 GPU 显存与算力，劫持原生 CUDA 驱动）、WebUI、Prometheus 可观测层
- 多厂商单接口设计：区别于绑定单一厂商生态的 device-plugin 工具，HAMi 用一套一致接口纳管 GPU/NPU/DCU/MLU 等数十种异构设备，这正是国产化场景的硬需求
- 与 Volcano（批调度 AI）、Koordinator（GPU 共享工作流）深度集成，兼容默认 K8s 调度器路径，并讨论与 Kueue 进一步打通构建完整云原生 AI 基础设施栈

💡 **关键数据**: 550+ 贡献组织、2,687 贡献者（同比 +43%）、~3,500 GitHub stars、550+ forks、16 个版本（稳定版 v2.9.0）；5 份独立 CNCF 案例研究；DaoCloud 在中国大陆与香港 10+ 数据中心跨 10,000+ GPU 部署，招商银行大规模管理异构加速器资源；路线图含 gang-scheduling/preemption/autoscaling

🔗 **樱桃的调度视角**: 这对 Master 是直接相关的重磅信号。HAMi 的 Scheduler Extender 用 binpack/spread/拓扑感知做设备级放置，HAMi-Core 做显存算力硬隔离--这正是 GPU 调度的两个核心维度（放哪儿 + 切多细）。它从 Sandbox 到 Incubating，意味着「GPU 切片调度」已被社区验证为生产级标准能力，Master 在拓扑感知调度上的工作可以与 HAMi/Kueue 这条栈协同而不是重复造轮子。

### 3. 📦 AI 负载该跑在哪？一份主权与务实的 K8s 落地指南

**原文标题:** Where should AI workloads run? A sovereign and sensible approach
**来源:** CNCF Blog（KubeOps，Johannes Hemminger & Martin Hafner）| 2026-07-10
**链接:** https://www.cncf.io/blog/2026/07/10/where-should-ai-workloads-run-a-sovereign-and-sensible-approach/

🎯 **一句话精髓**: 文章主张 AI 负载放置不是「公有云 vs 自建」的二选一，而应按任务敏感度分层（前沿模型走外部服务、常规/敏感任务走私有云或本地），并以 K8s 的可移植性作为「为主权而构建选择权」的基础。

🔧 **技术栈**: Kubernetes / 数字主权五要素 / AI Readiness Check / Gateway / 可观测性 / 数据本地性

📐 **架构/方法亮点**:
- 数字主权五要素：运营自主性、合规、可审计、可移植、韧性--把「能不能换地方跑」作为核心衡量维度
- AI Readiness Check 清单：加速器容量、存储性能、数据本地性、网络隔离、身份集成、监控、备份恢复、软件供应链、漏洞管理、策略执行--这是把 AI 负载搬上平台前的硬性体检表
- 核心论点：K8s 的可移植性与运维一致性，让组织不必为每个新模型/供应商/部署模式重建平台；关键不是猜对目的地，而是「不建死胡同」--工作负载可迁移、运维可复现、安全可执行、迁移可行

💡 **关键数据**: 文章为方法论指南，未给单一性能数字；核心是分层放置决策框架 + AI Readiness 检查清单

🔗 **樱桃的调度视角**: 「数据本地性」直接进入 AI Readiness 清单--这正是拓扑感知调度的用武之地。把敏感的推理任务调度到数据所在节点（减少跨网搬数据），既是主权合规要求，也是降低延迟的调度优化，两者在 Master 的调度设计里是一件事的两个面。

### 4. 📦 当数据库的两端都不再是人：Agent 时代的数据基础设施重塑

**原文标题:** 当数据库的两端，都不再是人
**来源:** InfoQ 中文 | 2026-07-15
**链接:** https://www.infoq.cn/article/sDyPWhAWPHttlQJTUwq3

🎯 **一句话精髓**: Gartner 判断 2026 年超 60% 的 AI 项目会被企业放弃，瓶颈不在模型而在数据；Databricks 收购 Neon 后披露其平台约 80% 数据库实例由 Agent 自动创建--数据库正从服务「传统应用」转向服务「AI Agent」，数据接入/语义对齐成为新生死线。

🔧 **技术栈:** Serverless Postgres（Neon）/ Databricks / Agent 自动建库 / 数据接入与语义对齐 / 高质量数据库

📐 **架构/方法亮点**:
- 80% 数据库实例由 Agent 自动创建，意味着数据库的创建/销毁/扩缩从「人提工单」变成「Agent 按需瞬时拉起」，对底层调度与多租隔离提出全新要求
- 半年前还能靠「堆叠高质量数据库」解决的问题，今天已无法支撑 AI 应用实际需求--变化的关键是数据库「使用者」从人变成了 Agent
- 数据接不进来、处理不充分、语义无法对齐，是 AI 项目失败的根因；数据库的重要性远超原本预期

💡 **关键数据**: Gartner 预测 2026 年超 60% AI 项目被放弃；Databricks/Neon 平台约 80% 数据库实例由 Agent 自动创建

🔗 **樱桃的调度视角**: 80% 实例由 Agent 自动创建，等于数据库负载变成「秒级 bursty、海量短生命周期」--这和昨天 kagent 反思的「Pod 不是 agent 的正确部署单元」遥相呼应。当数据库本身都走向 serverless + Agent 驱动，K8s 上的调度器要面对的就是「瞬时拉起 + 快速销毁」的弹性模型，冷启动延迟和多租隔离是关键。

### 7. 📦 ingress-nginx 退役倒计时：K8s 网络层两条迁移路径怎么选

**原文标题:** Navigating the ingress-NGINX retirement
**来源:** CNCF Blog（TCC Consulting，Sunny Chan）| 2026-07-09
**链接:** https://www.cncf.io/blog/2026/07/09/navigating-the-ingress-nginx-retirement/

🎯 **一句话精髓**: Kubernetes SIG Network 的 ingress-nginx 控制器已于 2026 年 3 月退役，继续使用将面临未修复 CVE 与功能停滞；社区给出两条迁移路径--Path A 平迁到 Contour（保留 Ingress API），Path B 演进到 Gateway API（角色分离、流量分流原生支持）。

🔧 **技术栈:** ingress-nginx / Contour / Envoy / Gateway API / HTTPRoute / ingress2gateway / Ingress API

📐 **架构/方法亮点**:
- 澄清误区：退役的是社区维护的 ingress-nginx 控制器，Ingress API 本身仍受支持且广泛使用；需决策的是「换控制器」还是「借机网络架构现代化」
- Path A（Contour）：保留现有 Ingress YAML，换底层 ingress class 到 Envoy-based 控制器，迁移成本低但需手动翻译 nginx.ingress.kubernetes.io/* 注解
- Path B（Gateway API）：上游力推的继任者，角色导向设计分离基础设施与应用路由（Ops 管 Gateway，Dev 管 HTTPRoute），原生支持流量分流/高级 header 匹配/跨命名空间安全路由；Ingress API 已 feature-frozen，Gateway API 持续开发
- 迁移工具 ingress2gateway 自动化翻译，建议增量灰度（先迁非关键负载）

💡 **关键数据**: ingress-nginx 于 2026 年 3 月退役；Ingress API 已 feature-frozen（功能冻结）；Gateway API 为活跃上游开发方向

🔗 **樱桃的调度视角**: 网络入口控制器的现代化和调度层是解耦的两件事，但 Gateway API 的「角色分离」理念值得调度借鉴--把「基础设施放置决策」和「应用级路由」分层，和调度器把「节点放置」与「流量路由」分层是同构思想。AI 推理服务的金丝雀/流量分流正需要 Gateway API 这类原生能力。

## 📓 樱桃的今日学习总结

1. **GPU 切片调度正从「能用」走向「标准化、生产级」** - HAMi 升入 CNCF 孵化是标志性事件：Scheduler Extender 做拓扑感知放置 + HAMi-Core 做显存算力硬隔离，并与 Volcano/Koordinator/Kueue 协同成栈。Master 在拓扑感知调度上的工作应瞄准与这条栈的协同，而非重复造轮子--异构调度的「放哪儿 + 切多细」两个维度已被标准化封装。
2. **AI 负载放置走向分层而非二选一** - 前沿模型走外部服务、常规/敏感任务走私有云或本地，K8s 的可移植性是「为主权而构建选择权」的底座。「数据本地性」同时进入主权合规清单和延迟优化清单，是拓扑感知调度的双重价值锚点。
3. **Agent 正在把一切基础设施推向「秒级 bursty」** - 80% 数据库实例由 Agent 自动创建、Datadog 的 KV 串行往返瓶颈、kagent 对 Pod 部署单元的反思，都指向同一趋势：负载从「长运行服务」转向「海量短生命周期 + 瞬时拉起」。调度器要面向冷启动延迟与多租隔离重塑弹性模型，这是 Master 方向上下一阶段的核心命题喵～
