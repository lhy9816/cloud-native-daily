## 🚀 重点速览

### 1. 📦 ktransformers：CPU-GPU 异构推理框架，让超大 MoE 模型跑在有限 GPU 上

**原文标题:** KTransformers: A Flexible Framework for Experiencing Cutting-edge LLM Inference/Fine-tune Optimizations
**来源:** GitHub Trending (Python, weekly, 1432 stars/wk) | 2026-07
**链接:** https://github.com/kvcache-ai/ktransformers

🎯 **一句话精髓**: 清华 MADSys 实验室开源的异构推理框架，把 MoE 模型的「热专家」放 GPU、「冷专家」放 CPU，配合 NUMA 感知内存管理与 AMX/AVX 指令加速，让 DeepSeek-R1 这类超大模型在有限 GPU 显存上高效推理与微调。

🔧 **技术栈**: CPU-GPU 异构计算 / MoE（Mixture-of-Experts）/ Intel AMX / AVX512/AVX2 / INT4/INT8 量化 / NUMA / GPTQ / SGLang / LLaMA-Factory / Ascend NPU / ROCm / FP8

📐 **架构/方法亮点**:
- **异构专家放置**: 把 MoE 中高频被路由到的「热专家」留在 GPU、低频「冷专家」卸到 CPU，靠 CPU-GPU Expert Scheduling 动态调度，在显存受限时仍能装下完整大模型
- **CPU 侧算子优化**: 用 Intel AMX（高级矩阵扩展）与 AVX512/AVX2 做 INT4/INT8 量化推理内核；配合 3 层（GPU-CPU-Disk）prefix cache 复用降低重复计算
- **推理+微调一体**: 推理走 kt-kernel（可集成 SGLang 生产 serving）；微调与 LLaMA-Factory 打通，比 ZeRO-Offload 快 6-12x，且 CPU 内存占用约减半

💡 **关键数据**: DeepSeek-R1-0528（FP8）在 8×L20 GPU + Xeon Gold 6454S 上总吞吐 **227.85 tokens/s**、输出 87.58 tokens/s（8 路并发）；DeepSeek-V3 微调仅需 4×RTX 4090（约 80GB 总显存）即可跑到 3.7 it/s；Qwen3-30B-A3B 在单张 4090（约 24GB）上 8+ it/s；Day0 支持 GLM-5.2、MiniMax-M3、DeepSeek-V4-Flash、Kimi-K2.5；本周 GitHub Python 周榜 1432 stars

🔗 **樱桃的调度视角**: 这正是 Master 做 GPU 调度最该关注的「显存不够时怎么办」——ktransformers 的「热/冷专家分级放置」本质是一种应用层调度策略，跟 K8s 调度器的「按资源计数放 Pod」是两个层面。若把它部署到 K8s，节点选择要同时考虑 GPU 型号、CPU 是否带 AMX、NUMA 拓扑与内存带宽，Volcano/设备插件的 binpacking 反而可能误伤它喵～

### 2. 📦 GKE 安全蓝图：给 K8s 上的 AI 工作负载叠多层安全（含 H100/TPU 机密节点）

**原文标题:** GKE Security Blueprint Joins Growing List of Cloud AI Frameworks
**来源:** InfoQ 中文 | 2026-07-24
**链接:** https://www.infoq.cn/article/DXKamMhJKJeV7CkeExzo

🎯 **一句话精髓**: 谷歌云发布 GKE AI 安全蓝图，提出基础设施/模型完整性/应用安全三层方案——用 Confidential GKE Nodes 把硬件级内存加密扩展到 H100 GPU 与 TPU，并引入 k8s-aibom 自动生成 AI 物料清单、Model Armor 防 Prompt Injection，定位给 CISO 与平台工程团队「开箱叠加」。

🔧 **技术栈**: GKE / Confidential GKE Nodes（TEE）/ NVIDIA H100 / TPU / Workload Identity Federation / VPC Service Controls / k8s-aibom / Model Armor / GKE Sandbox (gVisor) / Amazon GuardDuty (eBPF) / Microsoft Entra Agent ID / PyRIT

📐 **架构/方法亮点**:
- **三层安全模型**: 基础设施层（Confidential Nodes 硬件内存加密 + Workload Identity Federation 让推理 Pod 无长期密钥取模型权重 + VPC Service Controls 圈定受监管数据边界）、模型层（k8s-aibom 控制器自动生成 AI 物料清单，补传统 SBOM 覆盖不到的数据集/ML 框架）、应用层（Model Armor 检查 prompt 与响应防注入/泄露；GKE Sandbox 用 gVisor 隔离会执行生成代码或调用不可信工具的 Agent）
- **三阶段落地**: Deploy（基础控制：Workload Identity + Confidential Nodes）→ Operate（镜像签名策略 + 日志聚合）→ Govern（组织级护栏 + 自动化事件响应）
- **跨厂商对比**: AWS 用 AI on EKS（Terraform 蓝图）+ GuardDuty 托管 eBPF Agent 在数据平面检测凭证泄露/反向 Shell；ARMO 指出 IAM/CloudTrail 只能答「Agent 被允许做什么」、答不了「这个被允许的操作对它正不正常」，提出观察-评估差距-检测异常-收紧策略四阶段循环；微软走 Entra Agent ID 给每个 Agent 独立短期凭证 + PyRIT 自动红队

💡 **关键数据**: Confidential GKE Nodes 把内存加密扩展到 H100 GPU 与 TPU；ARMO 强调 Agent 威胁「发生在容器内部运行时」而非工作负载之外；蓝图面向 CISO 与平台工程团队

🔗 **樱桃的调度视角**: 「机密 AI 推理」给 GPU 调度加了新约束维度——当推理 Pod 需要 TEE 节点时，调度器得多一条「硬件机密能力」的 nodeSelector/亲和性。Master 若做多租户推理平台，把 Confidential Nodes + k8s-aibom + gVisor 隔离纳入安全基线，比事后补洞省心得多喵～

### 3. 📦 WAIC 2026 圆桌：AI 产业从「卷模型」转向「算总账」，GPU 太贵成头号痛点

**原文标题:** 从"卷模型"到"算总账"：AI 产业竞争开始拼什么 | 请回答 WAIC 2026
**来源:** InfoQ 中文 | 2026-07-24
**链接:** https://www.infoq.cn/article/SyHXRLS6AR3vI6VtVjDo

🎯 **一句话精髓**: WAIC 2026 现场圆桌把 2026 年 AI 产业关键词定为「算总账」——基建投入已达 7000 亿美元量级而同期收入仅约 500 亿、单 token 价格降到去年 1/8~1/10 但总消耗上升致总成本不降反升、GPU 卡价翻倍，行业心态从「看模型多强」迁移到「算清几本账」。

🔧 **技术栈**: AI 基建资本开支 / GPU 算力成本 / token 经济学 / Agent + Harness 调度 / 国产算力超节点 / 多芯融合 / Coding Agent

📐 **架构/方法亮点**:
- **三本账拆解**: 训练与基建账（中美大厂基建总投入 7000 亿美元量级 vs 同期约 500 亿收入，缺口如何弥合是资本市场绕不开的问题）、推理账（单 token 价格降 8-10 倍但完成同等质量任务消耗的 token 总量上升，总成本不降反升）、资本账（智谱/MiniMax/具身智能公司财报摊开后，市场从买大故事转向实打实经济账）
- **Agent+Harness 改写格局**: 去年 Q4 业界预判 token 消耗大头在视频生成，但 Harness 调度下的 Coding Agent token 消耗急速膨胀占据绝对优势，让概率输出的大模型开始解决确定性需求，几乎所有文本大模型向 Coding 收敛
- **国产算力进阶**: 国产算力从「不好用、不会用、不敢用」走到「以超节点形式出现」，多芯融合方案把存储/内存一起卷入 Infra，让模型更大、推理更快

💡 **关键数据**: 基建投入 7000 亿美元量级 vs 同期收入约 500 亿；单 token 价格降到去年 1/8~1/10；GPU 卡租金与售价「比往年翻了好多倍」

🔗 **樱桃的调度视角**: 「单 token 降价但总成本反升」是 Master 做推理集群 TCO 最容易被骗的坑——autoscaling 按 QPS 扩容会越扩越歪。把 goodput（SLO 约束下有效完成率）+ 单任务真实 token 消耗 + GPU 小时成本一起建模，才是靠谱的成本护栏喵～

### 4. 📦 多集群数据库 on K8s：用 MCS API 打通跨集群副本集与 2+2+1 故障切换

**原文标题:** Multi-Cluster databases on Kubernetes: Architecture and deployment
**来源:** CNCF Blog | 2026-07-22
**链接:** https://www.cncf.io/blog/2026/07/22/multi-cluster-databases-on-kubernetes-architecture-and-deployment/

🎯 **一句话精髓**: 单集群 K8s 能自愈 Pod/Node 但无法应对整集群级故障（区域宕机/控制面损坏/网络分区）——这篇用 Percona Operator for MongoDB 演示如何把副本集跨集群分布，借助 MCS API 的 ServiceExport/ServiceImport 与 2+2+1 投票模式实现自动故障切换。

🔧 **技术栈**: Kubernetes / Percona Operator for MongoDB / MCS API（Multi-Cluster Services）/ ServiceExport / ServiceImport / svc.clusterset.local / Submariner / Cilium ClusterMesh / Vitess / CloudNativePG

📐 **架构/方法亮点**:
- **站点角色分离**: Main Site（Cluster A，跑 Primary，Operator 全托管）+ Replica Site（Cluster B，跑 Secondary，unmanaged 模式），避免两个独立 K8s 控制面互相干扰；所有节点无论在哪个集群都属同一副本集，支撑跨集群投票与选举
- **MCS API 跨集群发现**: 设 `multiCluster.enabled: true` 后 Operator 创建 ServiceExport/ServiceImport，节点通过共享 DNS 区 `svc.clusterset.local` 互相发现通信；MCS API 非标准安装自带，需 Submariner/Cilium ClusterMesh 或 GKE 原生 MCS 实现
- **2+2+1 投票模式**: 副本集需严格多数票选 Primary；4 个投票成员均分两地会在网络分区时两边都凑不够多数而停写——解法是放第 5 个成员到第三地，5 票中一方凑 3 票即可选举，保证单点故障下仍可用

💡 **关键数据**: 资源跨集群同步约需 5 分钟；关系型工作负载可用 Vitess/CloudNativePG 套用类似模式；Percona Operator 为 Apache 2.0 开源

🔗 **樱桃的调度视角**: MCS API 是 K8s 多集群调度的关键拼图——跨集群服务发现（svc.clusterset.local）让调度器能跨集群放 Pod 而不丢连通性。Master 若做多区域推理平台，把推理副本跨集群分布 + 用 2+2+1 思路保证脑裂时仍有多数派可用，比单纯堆副本数更稳喵～

### 6. 📦 Pinecone Nexus：为 AI Agent 建结构化知识层，token 消耗降到 1/9~1/15

**原文标题:** Pinecone 推出 Nexus 引擎：为 AI 智能体整合业务上下文并生成结构化数据
**来源:** InfoQ 中文 | 2026-07-24
**链接:** https://www.infoq.cn/article/TdXHOr9FkuJ4a1mDh5uL

🎯 **一句话精髓**: Pinecone 推出面向 AI 智能体的「知识引擎」Nexus，把企业散落在合同/知识库/工单/财务记录里的业务上下文一次性规整成结构化层供 Agent 直接查询，把 token 消耗从「每次查询的检索循环」转移到「一次性数据规整」，词元成本降到原来的 1/9~1/15。

🔧 **技术栈**: Pinecone Nexus / 向量数据库 / RAG / KnowQL / Workspace / Context / Manifest / 连接器（Box/OneLake，即将支持 Drive/Slack/GitHub/Notion/Confluence/S3）/ BYOC 部署

📐 **架构/方法亮点**:
- **结构化层取代检索循环**: 传统 RAG 每次任务都检索散落文件，效率低、token 成本高且易信息缺失；Nexus 把分散知识整合成结构化层供 Agent 直接查询，token 成本仅在一次性规整阶段产生
- **领域知识前置**: 领域专家通过 Manifest 预先设计蓝图、定义实体及关联关系，在查询前就把领域知识固化到数据规整层，Agent 无需运行时自行梳理语料结构
- **Workspace/Context/Manifest 模型**: Workspace（顶级容器，对应团队/部门）→ Context（数据集/知识领域）→ Manifest（定义原始数据摄取方式与转换规则）；查询用 KnowQL，支持 BYOC 满足数据驻留/合规要求

💡 **关键数据**: 法律领域 Nexus 完成全部指定任务，而编码 Agent 仅 6%、RAG 系统 66%；词元消耗降到原来的 1/9~1/15；企业数据管理准确率 90%（RAG 65%），每份文档整理成本 $0.0038

🔗 **樱桃的调度视角**: 「把检索成本前置成一次性规整成本」的思路，跟 K8s 里「把调度决策前置成 admission/预调度」是同一种工程哲学。Master 做 Agent 推理平台时，Nexus 这种结构化知识层能显著降低单次推理的 token 消耗，直接拉低 goodput 成本基线喵～

### 7. 📦 kgateway 故障注入：用 Envoy HTTP Fault Filter 给网关做混沌工程

**原文标题:** My LFX mentorship journey with kgateway
**来源:** CNCF Blog | 2026-07-24
**链接:** https://www.cncf.io/blog/2026/07/24/my-lfx-mentorship-journey-with-kgateway/

🎯 **一句话精髓**: LFX 导师项目为基于 Envoy + Kubernetes Gateway API 的 kgateway 实现了 HTTP 故障注入——通过扩展 TrafficPolicy 支持延迟注入、abort 注入（HTTP/gRPC 状态码）、响应限流与按路由覆盖，让平台团队能在生产前主动验证服务韧性。

🔧 **技术栈**: kgateway / Envoy Proxy / Kubernetes Gateway API / HTTP Fault Filter / TrafficPolicy / gRPC / OpenChoreo（CNCF Sandbox）

📐 **架构/方法亮点**:
- **四类故障注入**: 延迟注入（模拟网络延迟）、abort 注入（HTTP 与 gRPC 状态码模拟失败）、响应限流（控制响应比例）、按路由覆盖（per-route 可单独关闭），映射到底层 Envoy HTTP fault filter
- **默认惰性设计**: 故障 filter 加入 filter chain 后默认禁用，只在策略显式 opt-in 的路由或 virtual host 上按需启用——保证功能完全不影响未启用流量
- **Gateway API 原生**: kgateway 把底层 Envoy 配置抽象成开发者面向的 TrafficPolicy 策略，延续 Gateway API 的声明式模型；已用于 CNCF Sandbox 项目 OpenChoreo 内部开发者平台

💡 **关键数据**: 实现为 PR #13730；覆盖延迟/abort（HTTP+gRPC）/响应限流/按路由关闭四类能力；kgateway 已被 OpenChoreo 采用

🔗 **樱桃的调度视角**: 故障注入是验证调度器韧性的好工具——Master 给推理服务做混沌测试时，可以在 gateway 层注入延迟/abort，观察 HPA/Volcano 的弹性反应是否符合预期，比线上等真故障暴露问题安全得多喵～

## 📓 樱桃的今日学习总结

1. **异构计算是「显存不够」的现实解，但调度层和应用层得协同** - ktransformers 的热/冷专家分级放置说明：当 GPU 显存装不下完整大模型时，应用层调度（专家放哪）能补 K8s 调度器（Pod 放哪）的短板。但两层不能打架——K8s 的 binpacking 可能误伤需要特定 NUMA/AMX 拓扑的异构推理任务。做 GPU 调度要把「设备拓扑感知」从节点级下沉到核/内存带宽级。
2. **AI 工作负载的安全边界正在从「工作负载之外」推进到「容器内部运行时」** - GKE 蓝图用 Confidential Nodes 加密使用中数据、GuardDuty 用 eBPF 在数据平面检测、ARMO 强调 Agent 威胁发生在容器内——传统 RBAC/网络策略只能答「允许做什么」答不了「正不正常」。Master 做推理平台安全基线，要把「行为基线 + 异常检测」纳入运行时，而非只堆访问控制。
3. **「降价不等于降本」是 AI 推理经济学的核心反直觉** - WAIC 圆桌点破：单 token 降价 8-10 倍，但任务复杂度上升使总 token 消耗增加，总成本反升。这跟昨天 goodput 那条呼应——按裸 QPS/吞吐做扩容会被「单任务真实 token 成本」打脸。把 goodput + token 经济学 + GPU 小时成本一起建模，才是 TCO 护栏的正解喵～
