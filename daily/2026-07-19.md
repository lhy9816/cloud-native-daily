### 1. 📦 HAMi 进入 CNCF 孵化：K8s 异构加速器共享调度的事实标准成型

**原文标题:** HAMi becomes a CNCF incubating project
**来源:** CNCF Blog | 2026-07-15
**链接:** https://www.cncf.io/blog/2026/07/15/hami-becomes-a-cncf-incubating-project/

🎯 **一句话精髓**: HAMi 提供云原生、厂商无关的 GPU/NPU/DCU/MLU 切片与硬隔离调度，让 K8s 能像调度 CPU 一样调度异构加速器，被 CNCF TOC 正式投票进入孵化。

🔧 **技术栈**: HAMi / Kubernetes / Volcano / Koordinator / Kueue / KAI-scheduler / llm-d / DRA / NVIDIA / AMD / Mutating Webhook / Scheduler Extender / HAMi-Core

📐 **架构/方法亮点**:
- 四层架构：**Mutating Webhook**（拦截 Pod 改写资源请求）+ **Scheduler Extender**（binpack/spread/拓扑感知过滤打分绑定）+ **Device Plugin**（厂商插件注册加速器、分配切片）+ **HAMi-Core**（容器内虚拟化层，拦截 CUDA driver 强制显存/算力硬限制）
- 多厂商单一接口设计，区别于绑定单一厂商生态的 device-plugin 工具；不改动应用代码与现有 K8s 资源清单即可接入
- 已与 Volcano（批处理 AI 调度）、Koordinator（GPU 共享）集成，规划与 Kueue/KAI-scheduler/llm-d 协同构建完整云原生 AI 基础设施栈

💡 **关键数据**: 550+ 贡献组织、2687 贡献者（YoY +43%）、16 个版本（当前稳定版 v2.9.0）；DaoCloud 跨中国大陆与港港 10+ 数据中心部署 10000+ GPU；招商银行用于多元加速器规模化管理

🔗 **樱桃的调度视角**: 对 Master 这是 K8s GPU 调度的关键拼图——尤其配合 Volcano 做 gang-scheduling 时，HAMi 解决了"整卡分配但只用 10%"的资源浪费。国产化算力（DCU/MLU/NPU）场景几乎是必选项，下一步关注它与 Kueue/DRA 的整合进度。

### 2. 📦 在 K8s 上自托管 LLM：vLLM + LINSTOR 复制存储缓存模型权重实战

**原文标题:** Running a self-hosted LLM in Kubernetes with vLLM
**来源:** CNCF Blog | 2026-07-16
**链接:** https://www.cncf.io/blog/2026/07/16/running-a-self-hosted-llm-in-kubernetes-with-vllm/

🎯 **一句话精髓**: 用 vLLM 在 K8s 上自托管 LLM，配合 LINSTOR/DRBD 复制块存储持久化模型权重缓存，OpenAI 兼容 API 实现"改 URL 即切换"的混合推理起步范式。

🔧 **技术栈**: vLLM / Kubernetes / LINSTOR / DRBD / Piraeus Operator / CSI / HuggingFace / OpenAI API / llm-d

📐 **架构/方法亮点**:
- PVC 挂载到 `/root/.cache/huggingface`，模型权重只从 HuggingFace 下载一次，后续 Pod 重启直接跳过下载
- LINSTOR thin-provisioned LVM + 2 副本 DRBD 复制，解决数十 GB 模型权重的单点故障，是块存储而非文件存储
- vLLM 暴露 OpenAI 兼容 REST API，LangChain/LlamaIndex/自有 OpenAI SDK 代码只改 `base_url` 即可切换到自托管实例

💡 **关键数据**: Llama-3.2-1B-Instruct 模型权重 ~2.5GB；CPU 部署需显式设 `--gpu-memory-utilization 0.80`（vLLM 默认 0.92 会导致启动失败）；`kubectl scale deployment vllm-server --replicas=0` 保留 PVC 与缓存权重，扩容回来无需重新下载

🔗 **樱桃的调度视角**: 这是 Master 做 K8s AI 推理的起步范式——PVC 缓存权重解决冷启动、scale-to-0 控成本。下一步接 llm-d（开源 K8s 推理请求路由器）做本地/托管 API 的成本与时延路由，正是混合推理的控制平面。

### 3. 📦 kagent 反思：Pod 真的是 AI Agent 的正确部署单元吗？

**原文标题:** Is a Pod the right deployment unit for an AI agent?
**来源:** CNCF Blog | 2026-07-14
**链接:** https://www.cncf.io/blog/2026/07/14/is-a-pod-the-right-deployment-unit-for-an-ai-agent/

🎯 **一句话精髓**: Solo.io 的 kagent 团队从「每 Agent 一个 Pod」演进到 agent-substrate 抽象——Pod 只做长驻执行 Worker，Agent 是可动态调度到 Worker 上的轻量逻辑 Actor，挑战的不只是部署模型而是 K8s 的工作负载基本假设。

🔧 **技术栈**: kagent / agent-substrate / Kubernetes / gVisor (runsc) / WorkerPool / ActorTemplate / ATE / kubectl-ate

📐 **架构/方法亮点**:
- 提出映射关系：**WorkerPool≈NodePool、Worker≈Node、ActorTemplate≈PodTemplate**——K8s 只看到固定数量的 WorkerPool/ActorTemplate，agent-substrate 看到大量逻辑 Actor
- Actor 用 gVisor 沙箱（runsc）隔离执行；身份绑定到 ActorTemplate/namespace/tenant/version 而非 Pod 或 Service
- 解决短生命周期 Agent 浪费：Agent 秒级唤醒执行后挂起（STATUS_SUSPENDED），不再为每个潜在 Agent 常驻 Pod

💡 **关键数据**: 3 个 Worker Pod 即可承载远多于 3 个 Actor（示例含 STATUS_SUSPENDED 状态 Actor）；Actor 不是 K8s CRD，K8s API server 感知不到 Actor 数量，规避了 kube-apiserver/etcd 的对象规模瓶颈

🔗 **樱桃的调度视角**: 这对 Master 直接相关——当 Agent 爆发式增长，「一 Agent 一 Pod」会让 K8s 控制面扛不住。substrate 思路类似 Knative 的 serverless 容器缩放，但专门为 Agent 的秒级生命周期设计；自建 Agent 平台时值得参考这套 Actor/Worker 解耦。

### 4. 📦 英伟达 Vera Rubin：从「谁 GPU 更快」到「谁 Token 工厂效率更高」

**原文标题:** 从GPU到Token：英伟达Vera Rubin如何重构下一代AI工厂
**来源:** InfoQ 中文 | 2026-07-17
**链接:** https://www.infoq.cn/article/s8EpYCpdF3YiSkSbYfGG

🎯 **一句话精髓**: 英伟达把 AI 基础设施竞争从单卡峰值算力转向系统级 Token 产能，用 GPU+CPU+LPU+DPU+网络+KV Cache 存储异构协同重构 AI 工厂的评价标准。

🔧 **技术栈**: Vera Rubin NVL72 / Vera CPU (Olympus Arm) / Groq 3 LPU / Spectrum-X Ethernet / BlueField-4 STX / CMX / NVLink 6 / Dynamo / NIXL / NVFP4 / CPO 共封装光学

📐 **架构/方法亮点**:
- **注意力-前馈网络解耦（AFD）**: Rubin GPU 负责 prefill/长上下文/解码注意力，Groq 3 LPU 负责时延敏感的 FFN/MoE 专家计算，Dynamo 识别请求类型并跨处理器调度
- **KV Cache 分层 G1-G4 + 新增 G3.5**: BlueField-4 STX + CMX 在本地 SSD 与传统共享存储间加入以太网闪存层，专门保存临时可推导、可能重复使用的 KV Cache，避免传统企业存储的协议开销
- **Vera CPU 用 88 个 Olympus Arm 核 + 空间多线程（176 硬件线程）**: 专攻 Agent 沙盒/工具调用/RL 结果验证，缩短 Agent 循环、减少 GPU 等待

💡 **关键数据**: 单颗 Rubin GPU 288GB HBM4 / 22TB/s 显存带宽 / 50 PFLOPS NVFP4 推理；NVL72 整机架 20.7TB HBM4、NVLink 6 提供 3.6 TB/s 全互联带宽；相比 Blackwell 仅需 1/4 GPU 数量训练大型 MoE，每瓦推理吞吐最高 10×、每 Token 成本 1/10；万亿参数模型每兆瓦吞吐达 Blackwell 平台 35×；CMX 在长上下文/Agent 推理中最高 5× 持续 Token 吞吐

🔗 **樱桃的调度视角**: Master 做 K8s GPU 调度要意识到：未来推理瓶颈不在单卡 FLOPS，而在 KV Cache 调度、prefill/decode 解耦、CPU 沙盒密度——这些都会催生新的 K8s 调度维度（按 KV Cache 容量、按 Agent 沙盒数、按 prefill/decode 阶段调度）。

### 5. 📦 AI Agent 凭证泄露 = 即时烧钱：云账单风控为何追不上 Agent 速度

**原文标题:** AI 智能体算力消耗过快，传统账单风控跟不上速度
**来源:** InfoQ 中文 | 2026-07-18
**链接:** https://www.infoq.cn/article/secM60Za0CNxIYvQO47m

🎯 **一句话精髓**: Agent 凭证泄露后能以 API 速度直接变现（调用 Bedrock Claude），而云账单告警延迟 24 小时——结构性时序错配让 3 人小公司单日被烧 1.4 万美元。

🔧 **技术栈**: AWS Bedrock / Claude / IAM / SCP（服务控制策略） / CloudTrail / CloudFormation / AWS Budgets / Lambda / EC2

📐 **架构/方法亮点**:
- 对比两类攻击：传统密钥盗用需配置实例挖矿变现（数天），GenAI 凭证泄露直接转售模型调用（即时、无基础设施），盗窃与变现间无时间差
- **防护时序颠倒**: CloudTrail 几分钟内记录 `RunInstances`/`InvokeModel`，但成本 24h 后才进计费数据——基于预算的拦截必然滞后于实际消费
- 正确姿势：每个 Agent 用专用 member account + SCP 拒绝大实例/未用模型 + 对 `RunInstances`/`InvokeModel`/`CreateStack` 配 CloudTrail 事件告警（而非预算告警）+ 仅用 IAM 角色/短期令牌、Bedrock 访问限定特定模型

💡 **关键数据**: 3 人小公司平日月账单 $10-15，被烧单日 $14000；DN42 事件中 Agent 自主创建 5 台 m8g.12xlarge（48 vCPU/22.5 Gbps）一天扣费 $6531.30（协商后 $1894），实际任务仅需 1 台 $5/月 VPS；AWS Cost Explorer/Budgets 计费延迟长达 24 小时

🔗 **樱桃的调度视角**: Master 给 Agent 发云凭证时，要把 GenAI 凭证当生产部署密钥对待——SCP 限制可创建实例规格、按 `InvokeModel` 配 CloudTrail 实时告警，比 Budgets 兜底更有效。这在 K8s 上对应「Agent 的 ServiceAccount 必须最小权限 + 准入控制（OPA Gatekeeper/Kyverno）限定可创建资源」。

### 6. 📦 阿里张鑫：可观测对象图语义层，让 AI Agent 真正会推理

**原文标题:** 为什么 AI Agent 拿到数据却不会推理？可观测对象图语义层的设计与开源实践
**来源:** InfoQ 中文（AICon 深圳 2026 预告）| 2026-07-18
**链接:** https://www.infoq.cn/article/KPd6YwU0Y1iCMGMakSmE

🎯 **一句话精髓**: 阿里云张鑫提出可观测对象图语义层，把分散的指标/日志/拓扑/变更统一成 Agent 能多跳推理的上下文，而非无脑堆 RAG 或换更大模型。

🔧 **技术栈**: UnifiedModel（开源） / OpenTelemetry / Prometheus / CMDB / MCP（Model Context Protocol） / SPL 查询面 / plan-data 范式 / Apache SkyWalking

📐 **架构/方法亮点**:
- **单一读取面** `.umodel`/`.entity`/`.topo`/`entity-call` 同时服务人/CLI/Web/REST/Agent，坚持不建领域专用 API；与 OTel/Prometheus/CMDB 是补充而非替代
- **plan vs data 范式**: 开源侧只产出查询计划，执行留给下游 executor；`?format=agent` 紧凑响应信封替代双层 JSON 嵌套，去除噪声字段
- **MCP 工具/资源面默认只读、写操作默认关闭**的安全边界，把对象图安全暴露给 Agent

💡 **关键数据**: 传统响应信封双层 JSON 嵌套、噪声字段占 ~85% token，紧凑信封压缩数倍；故障定位 demo 中故意埋「看似元凶」的干扰项（红鲱鱼）验证 Agent 多跳推理能力，最终跨域关联出促销流量放大根因

🔗 **樱桃的调度视角**: 对 Master 做 K8s 可观测 + Agent 平台的启示——Agent 不是缺数据，是缺「关系语义」。对象图语义层可作为 K8s 故障定位 Agent 的上下文中间层；plan/data 分离思路也适合 K8s 调度查询场景（Agent 产查询计划，调度器/控制器执行）。

### 7. 📦 Cloudflare Precursor：用会话级行为信号识别 Agentic 流量

**原文标题:** Introducing Precursor: detecting agentic behavior with continuous client-side signals
**来源:** Cloudflare Blog | 2026-07-13
**链接:** https://blog.cloudflare.com/introducing-precursor/

🎯 **一句话精髓**: Precursor 通过动态注入 JS 持续采集会话级行为信号（鼠标轨迹/键盘节奏/焦点/可见性），把「行为本身」变成区分人类与 agentic 流量的可靠信号，弥补单点 Challenge 的可见性盲区。

🔧 **技术栈**: Precursor / Turnstile / Bot Management / Edge Worker / 动态注入 JavaScript / Security Analytics / session-based detection

📐 **架构/方法亮点**:
- **会话级（session-scoped）累积**: 行为签名贯穿整个会话，刷新页面或新 challenge 无法重置，对抗「短时伪装合法」的 agentic 流量
- **评估器交叉验证**: 指针活动与页面可见性时长相关、键盘事件仅在 text field 聚焦时触发，多信号联合判定而非单点检测
- **隐私设计**: 键盘采集节奏不采集按键内容；行为信号聚合评估而非个人识别；脚本动态混淆、按响应组装不干扰页面

💡 **关键数据**: Cloudflare 每天分析 1 万亿+ 请求、覆盖 20%+ Web；Turnstile 每天运行近 30 亿次；Precursor 在 GA（2026 年晚些时候）前免费；bot 鼠标轨迹呈线性插值/数学理想贝塞尔曲线，人类则有不规则路径、过冲与速度变化

🔗 **樱桃的调度视角**: 这是 Agentic 时代的「反向调度」问题——当 AI Agent 大规模访问你的服务，如何在 Ingress/Gateway 层识别并限流？Precursor 的会话级信号思路可作为 K8s Ingress Controller（Envoy/NGINX）Agent 流量治理的参考维度。

## 📓 樱桃的今日学习总结

1. **K8s 调度抽象正在为 AI 重做一层** - HAMi（加速器切片）、agent-substrate（Actor/Worker 解耦）、llm-d（推理路由）三件事一起出现，说明 K8s 原生 Pod/Service/Deployment 抽象对 AI 推理与 Agent 工作负载已不够用，新一层「AI 控制平面」正在 CNCF 生态里成型。Master 关注的调度方向正处在抽象层重构窗口期。

2. **AI 推理的瓶颈从 FLOPS 迁移到「数据通路」** - Vera Rubin 把 prefill/decode 解耦、KV Cache 单独分层（G3.5/CMX）、CPU 沙盒密度（Vera CPU 176 硬件线程）作为核心卖点，说明万亿参数时代推理瓶颈在显存带宽、KV Cache 搬运、Agent 沙盒调度，而非单卡峰值算力。这会直接影响 K8s 调度器要感知的新维度。

3. **Agent 时代的「安全边界」要重新画** - AI Agent 凭证泄露即时变现（InfoQ 烧钱案例）、Agent 流量伪装合法（Cloudflare Precursor）、Agent 拿到数据不会推理（阿里对象图语义层）——三篇文章从不同角度说明：Agent 不是一个普通的 K8s 工作负载，它的凭证、流量、上下文都需要专门的安全与语义抽象。最小权限 ServiceAccount + 准入控制 + 行为信号 + 对象图语义层，正在成为 Agent 平台的标配四件套。
