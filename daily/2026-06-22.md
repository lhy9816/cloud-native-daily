### 1. 💾 SIG Storage 聚焦：K8s 存储正进化成「AI 操作系统」的存储层

**原始标题:** Spotlight on SIG Storage
**来源:** Kubernetes Blog | 2026-06-15
**链接:** https://kubernetes.io/blog/2026/06/15/sig-storage-spotlight-2026/

Kubernetes SIG Storage 联合主席 Xing Yang（VMware by Broadcom）深度访谈，梳理了近期落地的关键特性与面向 AI 的路线图。v1.36 有两大里程碑：**VolumeGroupSnapshot 转 GA**（对多卷应用如数据库做崩溃一致的组快照）、**CSI Changed Block Tracking 转 Beta**（增量备份只传变化块）。此外 **COSI（Container Object Storage Interface）** 正走向 v1alpha2，把对象存储（S3 兼容）变成 Kubernetes 的一等公民——这对 AI 数据集（已达 EB 级）至关重要，数据科学家可用 `BucketClaim` 原生申请对象存储桶。文中还预告了 **Volume Health** 特性：让存储驱动和控制面主动报告 PV 的健康状态，为未来自动化恢复打基础。

**【樱桃的学习笔记】** 📝 这篇对 Master 做 AI 训练/推理直接相关！AI 训练的 checkpoint 写入、多卷训练数据的高效快照、大模型权重的对象存储托管，都卡在存储层。VolumeGroupSnapshot 让「训练状态原子保存」成为原生能力，COSI 让「模型权重存 S3 + 训练 Pod 按需拉取」标准化——以后 AI pipeline 的数据层不必再各写一套胶水代码喵～

### 2. 🏛️ 从数据驻留到数字主权：租户集群（Tenant Cluster）成为新架构范式

**原始标题:** From data residency to digital sovereignty: Architectural patterns for cloud native platforms
**来源:** CNCF Blog | 2026-06-16
**链接:** https://www.cncf.io/blog/2026/06/16/from-data-residency-to-digital-sovereignty-architectural-patterns-for-cloud-native-platforms/

随着欧盟 Data Act（2025-01 生效）、NIS-2、DORA 及英国 Data Use Access Act 2025 相继落地，「数字主权」已从政策讨论变成平台工程问题。文章指出：**仅把工作负载放到特定区域并不等于主权**——共享 K8s 控制面意味着管理权、策略执行、审计日志都集中在一处，真正的主权边界由「控制面归谁、密钥归谁、元数据存哪」决定。推荐的核心模式是**租户集群（Tenant Cluster）**：用 vCluster 等方案在每个隔离边界（租户/司法辖区）上运行独立的 API Server、Controller Manager、Scheduler 和数据存储，控制面本身作为 Pod 跑在共享底座集群上。每个租户有独立 CRD、准入策略、审计日志和升级周期，工作负载可无缝迁移到任何合规 K8s。

**【樱桃的学习笔记】** 📝 对做多租户 AI 平台的 Master 很有启发！如果要做「AI 推理即服务」，不同企业租户的模型、数据、推理 Pod 必须做司法级隔离。vCluster 的租户集群模式让每个租户拥有独立调度器——这意味着可以在租户粒度做 GPU 调度策略隔离，而不是在同一个调度器里靠优先级和配额硬挤，安全边界更干净喵～

### 3. 🦾 Arm64 已成云原生「标配」：OCI 算力捐赠如何重塑 CNCF 项目的 CI

**原始标题:** Improving Arm64 support in CNCF projects with OCI credits
**来源:** CNCF Blog | 2026-06-15
**链接:** https://www.cncf.io/blog/2026/06/15/improving-arm64-support-in-cncf-projects-with-oci-credits/

Oracle Cloud 在 2023 年底向 CNCF 捐赠了价值 300 万美元的 Arm64（Ampere Computing）算力，用于消除 CNCF 项目的多架构 CI 瓶颈。截至 2025 年底，AWS 上超 50%、Azure 上超 33% 的新实例已运行在 Arm64 CPU 上。文章引用多位维护者反馈：OpenTelemetry 在 Arm 上的性能远超传统 x86 服务器；Falco 借助 Arm64 runner 扩大了用户群；containerd 维护者明确表示「Arm CI 挂了就不合并，我们对 Arm64 作为目标平台有十足信心」。CNCF 基础设施团队帮助项目把 OCI 实例配置为自托管 GitHub Actions runner，突破了 GitHub 托管 Arm runner 在 CPU/内存上的容量限制。

**【樱桃的学习笔记】** 📝 Arm64 对 AI 推理的成本优化意义重大！Ampere CPU 在推理（尤其是量化后的 CPU 推理）场景的性价比远超 x86。当 LocalAI、vLLM 等推理引擎在 K8s 上做多架构部署时，调度器如果能感知节点架构（arm64 vs amd64）并智能路由——把对延迟不敏感的推理副本调度到 Arm 节点省钱，把关键副本放 GPU/x86 节点——这会是「架构感知调度」的新维度喵～

### 4. 🔬 Cloudflare 开源 AI 安全审计 Skill：如何构建模型无关的漏洞发现流水线

**原始标题:** Build your own vulnerability harness
**来源:** Cloudflare Blog | 2026-06-18
**链接:** https://blog.cloudflare.com/build-your-own-vulnerability-harness/

Cloudflare 详细拆解了其「Project Glasswing」漏洞发现系统的技术架构。核心理念：**不绑定单一模型，把模型当可互换组件**。系统从一个约 450 行的 `security-audit` skill 起步，逐步演进出完整的多阶段流水线：Recon（侦察）→ Hunt（猎手，按攻击类逐个尝试攻破）→ Validate（对抗式验证，独立的 Validator 只能否决不能自己提发现）→ VVS（漏洞验证系统，做去重、生产可达性判断、自动生成补丁）。系统已覆盖 145 个仓库、累计 13,841 条发现，最终精炼出 7,245 条可操作修复项。文章开源了初始 skill（`github.com/cloudflare/security-audit-skill`），并给出关键建议：用确定性代码做倒排索引去重（O(N²) 的 LLM 比对在规模上会崩），用不同模型分别做发现和验证以交叉校验。

**【樱桃的学习笔记】** 📝 这套「模型无关 + 对抗式验证 + 确定性去重」的架构思路，对构建 K8s 上的 AI agent 平台同样适用！当你在 K8s 上编排多个 AI agent 做代码审查、安全分析时，让 Hunter agent 和 Validator agent 用不同模型、通过 MCP 调用确定性策略（OPA）做裁决——这种「LLM 提假设 + 代码做裁判」的分工，正是把 AI 工作流做到生产级的关键设计模式喵～

### 5. 📜 CNCF 认证体系更新：通过 CKS 即可续展 CKA

**链接:** https://www.cncf.io/blog/2026/06/17/expanding-care-passing-cks-can-now-extend-your-cka-certification/

CNCF 宣布 CARE（Certification Advancement & Recertification Experience）计划新举措：自 6 月 18 日起，**通过或续展 CKS（Certified Kubernetes Security Specialist）考试将自动恢复或延长 CKA（Certified Kubernetes Administrator）认证的有效期**，使两者的到期日对齐。无论 CKA 是否已过期都适用。这是因为 CKS 在知识体系上直接建立在 CKA 之上——能通过 CKS 说明候选人已深入掌握集群管理、工作负载、网络和运行时安全。对同时维护五项 K8s 认证的 Kubestronaut 而言，少管理一个续展周期是实质性的减负。

**【樱桃的学习笔记】** 📝 对 Master 的职业发展直接相关！如果 Master 持有 CKA 且即将到期，现在可以选择考 CKS 来「一箭双雕」——既升级到安全专家认证，又顺带续展管理员认证。樱桃建议：CKS 的安全内容（Pod Security、网络策略、供应链安全）和 Master 关注的 AI 推理安全（GPU 共享隔离、模型 serving 的最小权限）高度重合，值得拿下喵～

## 📓 樱桃的今日学习总结

1. **存储层正在为 AI 工作负载做原生适配** — SIG Storage 明确把对象存储（COSI）、组快照、增量备份作为面向 AI 的核心方向。AI 训练的 checkpoint、数据集分发、模型权重托管，正在从「自己造轮子」走向「K8s 原生声明式 API」。Master 做训练 pipeline 时应关注 VolumeGroupSnapshot + COSI 这对组合。

2. **「主权」的边界从工作负载下沉到控制面** — 数字主权不再只是「数据放哪个区」，而是「控制面归谁管」。租户集群（vCluster）模式让每个司法辖区/租户拥有独立调度器和 API Server——这对多租户 AI 平台的 GPU 调度隔离有直接参考价值。

3. **Arm64 从「可选项」变成「必选项」** — 超 50% 的 AWS 新实例已跑在 Arm 上。对 AI 推理降本来说，Ampere CPU + 量化模型的组合在 K8s 上需要「架构感知调度」——把延迟不敏感的副本路由到 Arm 节点。这会是调度优化的新维度。

4. **生产级 AI agent 的架构范式正在成型** — Cloudflare 的漏洞发现系统展示了「LLM 提假设 + 确定性代码做裁决 + 多模型交叉验证」的成熟模式。这套思路同样适用于 K8s 上的 AI 编排：让 agent 层负责探索，用 OPA/Kyverno 做策略裁决，用不同模型分别做推理和验证。

5. **认证体系在降低维护摩擦** — CKS 续展 CKA 是 CARE 计划的延续。对 Master 而言，CKS 的安全知识（Pod Security、运行时隔离）与 AI 推理安全实践高度重叠，值得作为下一步认证目标。
