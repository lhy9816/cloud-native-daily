### 1. 📦 HAMi 升级为 CNCF 孵化项目——K8s 异构算力调度走向标准化

**原文标题:** HAMi becomes a CNCF incubating project
**来源:** CNCF Blog | 2026-07-15
**链接:** https://www.cncf.io/blog/2026/07/15/hami-becomes-a-cncf-incubating-project/

🎯 **一句话精髓**: CNCF TOC 投票通过 HAMi 进入孵化阶段，这是首个面向 K8s 多厂商 GPU/NPU 虚拟化与调度的中立开源中间件，直接解决"贵价 GPU 被整卡独占导致碎片化浪费"的核心痛点。

🔧 **技术栈**: HAMi / Kubernetes Mutating Webhook / Volcano / Koordinator / Kueue / DRBD / GPU 虚拟化 / NPU / DCU / MLU

📐 **架构/方法亮点**:
- 通过 Mutating Webhook 拦截 Pod 创建，按显存/算力核数/设备数对物理 GPU（及 NPU、DCU、MLU 等异构加速器）进行切片，无需改动应用代码或现有资源清单
- 支持硬隔离运行时（共享同一卡的工作负载互不干扰），并提供 binpack / spread / 拓扑感知三种调度策略
- 多厂商统一接口设计——区别于绑定单一厂商生态的 device-plugin，一个平面管所有异构加速器；已与 Volcano（批处理 AI 调度）、Koordinator（GPU 共享）集成，正推进与 Kueue 的对接

💡 **关键数据**: 550+ 贡献组织、2,687 名贡献者（同比 +43%）、约 3,500 GitHub Stars；DaoCloud 在中国大陆及港澳 10+ 数据中心跨 10,000+ GPU 部署；招商银行用于异构算力规模化管理；已发布 16 个版本，当前稳定版 v2.9.0

🔗 **樱桃的调度视角**: 这条对 Master 价值最高——HAMi 把"GPU 调度"从整卡分配细化到了显存/算力核粒度，正好补上 K8s 默认调度器在异构加速器上"只见设备不见容量"的短板。配合 Volcano 做 AI 批处理调度、Kueue 做队列管理，几乎是 K8s 上 GPU 池化的标准组合拳，值得深入研究其 Webhook 切片与拓扑感知策略的实现细节。

### 2. 📦 英伟达 Vera Rubin——从"比谁 GPU 快"转向"比谁 Token 工厂效率高"

**原文标题:** 从GPU到Token：英伟达Vera Rubin如何重构下一代AI工厂
**来源:** InfoQ 中文 | 2026-07-17
**链接:** https://www.infoq.cn/article/s8EpYCpdF3YiSkSbYfGG

🎯 **一句话精髓**: 英伟达不再只卖更快的 GPU，而是把 CPU/GPU/LPU/DPU/网络/上下文存储整合为异构"Token 工厂"，用每瓦 Token 产出和单 Token 成本重新定义 AI 基础设施竞争力。

🔧 **技术栈**: Vera Rubin NVL72 / Vera CPU (Olympus Arm) / Groq 3 LPU / NVLink 6 / Spectrum-X Ethernet / BlueField-4 STX / CMX 上下文存储 / NVIDIA Dynamo / NVFP4 / Attention-FFN Disaggregation

📐 **架构/方法亮点**:
- NVL72 机架将 72 颗 Rubin GPU + 36 颗 Vera CPU 通过第六代 NVLink 铜质主干连成一个"超大 GPU"，机架间用 Spectrum-X 或 Quantum-X800 横向扩展；对上层软件而言 72 颗 GPU 即一个统一加速器
- 引入 Groq 3 LPU 做"注意力-前馈网络解耦（AFD）"：Rubin GPU 负责预填充与注意力计算，LPU 负责时延敏感的前馈/MoE 专家网络，NVIDIA Dynamo 调度器决定任务在 GPU 还是 LPU 上执行——把推理流水线按 prefill/decode 拆到不同处理器
- KV Cache 升级为独立基础设施：在 GPU HBM(G1)、CPU 内存(G2)、本地 SSD(G3) 与远端共享存储(G4) 之间新增 G3.5 层（BlueField-4 STX + CMX），用以太网闪存专门缓存可复用 KV Cache，减少重复预填充与 GPU 空等

💡 **关键数据**: 单颗 Rubin GPU 288GB HBM4、22 TB/s 显存带宽、50 PFLOPS NVFP4 推理；NVL72 整机架 20.7TB HBM4；对比 Blackwell 仅需 1/4 GPU 数量训练大型 MoE，每瓦推理吞吐最高 10×、单 Token 成本降至 1/10；Vera CPU 88 核/176 硬件线程，单机架 256 颗 CPU 可承载 22,500+ 强化学习/智能体沙盒；LPX 机架 256 颗 LPU、128GB 片上 SRAM、315 PFLOPS FP8；万亿参数模型每兆瓦吞吐最高达 Blackwell 35×；英伟达 Q1 2026 首次登顶数据中心以太网交换机营收第一（21 亿美元、21.5% 份额、同比 +192.7%）

🔗 **樱桃的调度视角**: Vera CPU 单机架跑 2.25 万个 Agent 沙盒、AFD 把推理按 prefill/decode 拆到不同处理器——这直接影响 K8s 上 AI 推理的调度粒度。未来调度器不能只看"有没有 GPU"，还要区分 prefill 池与 decode 池、管理 KV Cache 的 G3.5 共享层。Master 做推理调度时可以提前关注 Dynamo 的调度接口与 CMX 的 API 形态。

### 3. 📦 用 vLLM 在 K8s 上自托管大模型——OpenAI 兼容 API + 复制存储实践

**原文标题:** Running a self-hosted LLM in Kubernetes with vLLM
**来源:** CNCF Blog | 2026-07-16
**链接:** https://www.cncf.io/blog/2026/07/16/running-a-self-hosted-llm-in-kubernetes-with-vllm/

🎯 **一句话精髓**: 一篇手把手的 K8s 自托管推理实战——用 vLLM 暴露 OpenAI 兼容 API，用 LINSTOR（基于 DRBD 的复制块存储）通过 CSI 持久化模型权重，实现"下载一次、Pod 重启免下载"的混合推理架构。

🔧 **技术栈**: vLLM / Kubernetes / LINSTOR / DRBD / CSI 驱动 / Piraeus Operator / Hugging Face / Llama-3.2-1B-Instruct

📐 **架构/方法亮点**:
- 三件 K8s 资源搞定：PVC（linstor-csi-lvm-thin-r2 存储类，2 副本精简配置 LVM）挂载到 /root/.cache/huggingface 缓存模型权重；Secret 存 HF Token；Deployment + Service 跑 vLLM 推理服务器
- 关键设计：模型权重（动辄数十 GB）只从 Hugging Face 下载一次，落盘到 LINSTOR 复制卷后，后续 Pod 重启直接读本地缓存，跳过网络下载；DRBD 双副本保证节点故障时权重不丢
- 暴露 OpenAI 兼容 REST API——LangChain/LlamaIndex/任何用 OpenAI SDK 的代码只需改 URL 即可指向自托管实例，这是"混合架构（本地跑高频/敏感负载 + 托管 API 兜底）"成立的根基

💡 **关键数据**: 示例模型 Llama-3.2-1B-Instruct（1B 参数，可在 CPU 上跑）；PVC 申请 50Gi；使用 thin-provisioned LVM 双副本卷

🔗 **樱桃的调度视角**: 虽然示例是小模型 + CPU，但"用复制持久卷缓存模型权重避免重复下载"这个模式对 Master 的 GPU 推理集群同样适用——大模型权重几十上百 GB，Pod 调度到新节点时若每次都拉取会严重拖慢弹性扩容。结合 HAMi 的 GPU 切片，可以做出"权重共享卷 + GPU 分片"的高效推理 Deployment。

### 4. 📦 AI Agent 该不该住进 Pod？kagent 引入 agent-substrate 控制面

**原文标题:** Is a Pod the right deployment unit for an AI agent?
**来源:** CNCF Blog（作者 Lin Sun, Solo.io / CNCF Ambassador）| 2026-07-14
**链接:** https://www.cncf.io/blog/2026/07/14/is-a-pod-the-right-deployment-unit-for-an-ai-agent/

🎯 **一句话精髓**: Solo.io 团队在构建 kagent 过程中发现"给每个 AI Agent 独占一个 Pod"在规模化后很浪费——Agent 是短时突发型负载而非长驻服务，于是引入 agent-substrate 控制面，用 WorkerPool/Worker/Actor 三层抽象把 Agent 生命周期从 Pod 生命周期中解耦。

🔧 **技术栈**: kagent / agent-substrate (ate.dev) / Kubernetes / gVisor (runsc) / ServiceAccount / NetworkPolicy

📐 **架构/方法亮点**:
- 痛点分析：Pod 提供进程隔离、ServiceAccount 给身份、网络策略/准入策略可用——解决了隔离与治理；但 Agent 可能只醒来几秒到几分钟就空闲，给每个潜在 Agent 常驻独占 Pod 极度浪费；且 Agent 会动态生成子 Agent、会暂停等人审批、生命周期以秒/分钟计
- agent-substrate 在 K8s 之上加一层控制面：WorkerPool（类似 NodePool）管理一批执行 Worker（类似 Node），ActorTemplate（类似 PodTemplate）声明式描述 Agent 如何执行；K8s 仍管 Pod/Service/网络/存储/计算，上层只管 Agent 的生命周期与放置
- 用 gVisor (runsc) 做沙箱隔离，ActorTemplate 里直接内嵌 runsc 配置（amd64/arm64 双架构 hash + 下载地址）和快照存储位置（gs://ate-snapshots），Worker 复用而非每个 Agent 一个 Pod

💡 **关键数据**: WorkerPool 默认 replicas=3；gVisor runsc 使用 2026-06-02 nightly 版本；镜像以 digest 精确锁定（sha256）

🔗 **樱桃的调度视角**: 这篇直击 Master 关注的"调度抽象"——传统 K8s 工作负载模型（Deployment/Job）假设负载是持续可用的或一次性批处理，但 AI Agent 是"事件驱动 + 可能动态分裂子 Agent + 可能阻塞等人"的新形态。agent-substrate 的 WorkerPool/Actor 模式很像 K8s 里的"二级调度"，和 HAMi 的 GPU 切片思路异曲同工：都是在 K8s 原生调度之上加一层针对特定负载语义的资源管理层。

### 5. 📦 AWS 推出 Claude 应用网关——AI 代码工具的控制层从第三方走向模型厂商原生

**原文标题:** 亚马逊云科技推出 Claude 应用网关：面向 Claude Code 与 Claude Desktop 的自托管控制平面
**来源:** InfoQ 中文 | 2026-07-17
**链接:** https://www.infoq.cn/article/j59f7KU3djH4Ex4YhqxF

🎯 **一句话精髓**: Anthropic 联合 AWS/谷歌云/微软推出 Claude 应用网关——一个无状态容器部署的自托管控制平面，把企业身份认证、策略管控、成本限额、遥测监控、多平台故障转移集中到一个 YAML 配置里，标志着 AI 代码工具的治理层正从第三方网关回归模型厂商原生基础设施。

🔧 **技术栈**: Claude App Gateway / Amazon ECS·EKS·EC2 / Amazon RDS PostgreSQL / OpenID Connect / OpenTelemetry / Amazon Bedrock / IAM Roles Anywhere / Anthropic Messages API

📐 **架构/方法亮点**:
- 部署极简：一个无状态容器即可，跑在 ECS/EKS/EC2 上，内网 ALB 对外，RDS PostgreSQL 存短期登录态与速率限制计数器；配置仅一份 YAML，启动时加载
- 五大能力闭环：OIDC 单点登录（开发者浏览器登录获取短期令牌，IdP 移除用户后会话自动失效）→ 服务端托管策略（按 IdP 用户组划分模型/工具权限，开发者无法本地覆盖）→ OpenTelemetry 遥测（每请求打标转发到 CloudWatch/Prometheus）→ 统一保管上游凭证代为转发（支持跨区域跨账号故障转移）→ 按组织/组/用户设日/周/月成本限额，超额即拦截
- 协议层做 Anthropic Messages API 转换，统一对接 Bedrock、AWS 上的 Claude 平台、谷歌云智能体平台、微软 Foundry 及原生 Anthropic API；Anthropic 承诺开放网关通信协议，便于中立网关厂商兼容

💡 **关键数据**: 对接 Bedrock 时直接用容器 IAM 任务角色，无需静态密钥；AWS 安全负责人给出 Private CA + IAM Roles Anywhere 方案，用 X.509 证书兑换绑定 IAM 角色的临时凭证，全程无长期密钥

🔗 **樱桃的调度视角**: 虽然不是 K8s 调度本身，但"无状态容器 + YAML 配置 + OpenTelemetry 遥测 + 成本限额"这套模式正是 Master 在 K8s 上治理 AI 推理工作负载时可以借鉴的范式——把模型调用的治理（谁、用什么模型、花了多少钱、超限拦截）下沉到网关层，比在每个应用里各自实现要干净得多。

### 7. 📦 Cloudflare Precursor——用"整段会话行为"检测 AI Agent 流量

**原文标题:** Introducing Precursor: detecting agentic behavior with continuous client-side signals
**来源:** Cloudflare Blog | 2026-07-13
**链接:** https://blog.cloudflare.com/introducing-precursor/

🎯 **一句话精髓**: Cloudflare 推出 Precursor——在页面中动态注入轻量 JS，贯穿整个会话持续采集鼠标轨迹、点击节奏等行为信号，用"人类受物理与认知约束、机器人走数学理想曲线"这一本质差异，把 bot 检测从"单次挑战"升级为"整段会话行为画像"。

🔧 **技术栈**: Cloudflare Precursor / Turnstile / Enterprise Bot Management / 动态 JS 注入 / 客户端行为信号 / 会话级验证

📐 **架构/方法亮点**:
- 核心洞察：现代自动化能执行 JS、用真实浏览器、通过单次 CAPTCHA，但很难在整段会话中持续模拟人类行为——人类鼠标受腕关节旋转约束走弧线、有认知延迟、有生理手抖频率；机器人多走线性插值或理想贝塞尔曲线、点击精度人类无法复现
- 架构三层：注入采集层（HTML 响应经过 Cloudflare 网络时自动注入脚本，无需额外配置/网络连接/第三方嵌入）→ 会话级行为画像 → 实时接入 Bot Management 决策；作为 Turnstile 的可选补充，把检测从"关键端点"扩展到"整个应用全旅程"
- 对合法用户意味着更少不必要的打断；对 bot 开发者意味着必须模拟完整会话——成本显著上升

💡 **关键数据**: Cloudflare 网络层日均分析超 1 万亿请求、覆盖超 20% 的 Web；Turnstile 每天在敏感端点运行近 30 亿次；Precursor 属 Enterprise Bot Management 功能

🔗 **樱桃的调度视角**: 这条偏安全侧，但和 AI Agent 主题强相关——当 Agent 越来越多地代替人访问 Web 应用，"区分人类流量与 agentic 流量"会成为基础设施层的刚需。Master 若在做 AI Agent 平台，需要关注这类检测对 Agent 自动化流程的影响，以及如何在合规前提下让 Agent 携带可信身份访问 API。

## 📓 樱桃的今日学习总结

1. **GPU 调度正从"整卡分配"走向"显存/算力核切片 + 拓扑感知"** — HAMi 进入 CNCF 孵化是个信号：异构加速器调度需要设备级上下文，K8s 默认调度器只看到"设备数量"看不到"显存容量/核利用率"。未来 GPU 池化的标准范式很可能是 HAMi（切片+隔离）+ Volcano（批处理调度）+ Kueue（队列管理）的组合，Master 应重点关注其 Webhook 拦截机制与拓扑感知策略。

2. **AI 推理基础设施的衡量标准从"峰值算力"转向"Token 工厂效率"** — Vera Rubin 的 AFD（注意力-前馈网络解耦）把 prefill 和 decode 拆到不同处理器、KV Cache 升级为独立 G3.5 存储层、用每瓦 Token 产出和单 Token 成本衡量系统——这意味着 K8s 上的推理调度器未来要管理的是"多类型计算池 + 多级 KV Cache 存储 + 异构网络"，而非简单的"给 Pod 分一张 GPU"。

3. **AI Agent 需要新的工作负载抽象，而非套用 Pod/Deployment** — kagent 的 agent-substrate 和腾讯 Cube Sandbox 从不同角度得出同一结论：Agent 是短时突发、可能动态分裂子 Agent、可能阻塞等人的新形态负载，常驻独占 Pod 极度浪费。这预示着 K8s 之上会出现一层"Agent 调度控制面"（类似二级调度），Master 在设计 Agent 平台时应把"按需唤醒 + 沙箱隔离 + 生命周期与 Pod 解耦"作为核心架构原则。
