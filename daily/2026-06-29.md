### 1. 🧩 Moebius：MoE 推理引擎运行时动态切换 EP↔TP，8×H200 上 RL 吞吐提升 1.25×

**原文标题:** Moebius: Serving Mixture-of-Expert Models with Seamless Runtime Parallelism Switch
**来源:** arXiv (cs.DC) | 2026-06-25
**链接:** http://arxiv.org/abs/2606.26607

🎯 **一句话精髓**: 首次实现 MoE 模型在张量并行(TP)与专家并行(EP)之间不重启引擎、不丢请求地热切换，让单一推理系统横跨低并发与高并发两种最优工作点。

🔧 **技术栈**: MoE 架构 / 专家并行(EP) / 张量并行(TP) / 融合 GPU-GPU 传输内核 / KV-Cache 重分片 / Qwen3-235B-A22B / NVIDIA H200

📐 **架构/方法亮点**:
- **核心洞察**：EP 和 TP 是同一个模型的两种布局，计算的是字节级相同的专家权重与 KV cache，因此切换只需改变「每个 rank 拥有哪一片」，而不是搬运整个模型
- **切换成本最小化**：只迁移 owner 发生变更的切片，借助高带宽 GPU 互连，可在两个 decode 步之间完成切换，无需排空在途请求
- **常驻双布局**：同时保留两种并行策略的 runtime resident，专家权重与 KV cache 单副本、固定地址重分片，仅 2.4% 额外显存开销

💡 **关键数据**: 8×H200 服务 Qwen3-235B-A22B，RL rollout 吞吐提升 **1.16–1.25×**；每次切换耗时仅 **215–434ms**；内存开销仅 **2.4%**

🔗 **樱桃的调度视角**: 这对 Master 是「GPU 调度粒度」的范式启发——并行策略不再是部署期的静态决策，而是可随负载波动的运行时调度变量。RL 训练里 rollout 的「突发→长尾」曲线正是 K8s + Volcano 弹性队列的理想场景，Moebius 的热切换思路可与 Volcano 的 gang scheduling 结合，实现算力拓扑与模型布局的协同调度喵～

### 2. 🧊 CrossPool：把 FFN 权重与 KV-Cache 拆成两个显存池，冷 MoE 模型 P99 延迟降 10.4×

**原文标题:** CrossPool: Efficient Multi-LLM Serving for Cold MoE Models through KV-Cache and Weight Disaggregation
**来源:** arXiv (cs.DC) | 2026-06-23
**链接:** http://arxiv.org/abs/2606.24506

🎯 **一句话精髓**: 针对「大量冷门 MoE 模型共驻一机」的显存困境，将静态权重池与动态 KV-Cache 池物理解耦，按聚合实际需求而非单模型峰值来供给显存。

🔧 **技术栈**: MoE 多模型服务 / KV-Cache 池化 / 显存虚拟化 / 层流水线调度 / 持久化内核 / 控制下沉(control lowering)

📐 **架构/方法亮点**:
- **双池分离**：权重池跨冷模型合并 FFN 权重，KV-Cache 池按活跃请求动态供给、attention 本地化，避免静态权重挤占动态缓存
- **层流水线调度器**：隐藏 hidden-state 跨池传输，把搬运与计算重叠
- **控制下沉**：用持久化内核 + 控制下沉降低 CPU-GPU 控制开销，解决冷流量低并发下的调度抖动

💡 **关键数据**: P99 TBT（token 间延迟）相比 SOTA 的 kvcached 多 LLM 服务系统降低高达 **10.4×**；支撑突发式长上下文请求

🔗 **樱桃的调度视角**: 「按聚合需求而非峰值预留」本质是 GPU 显存的超卖(overcommit)策略，与 K8s 对 CPU/内存的 limit-request 超卖哲学同源。Master 在做多租户推理集群时，可借鉴这种「权重常驻 + 缓存弹性」的二级资源模型来提升 GPU 利用率～

### 3. 🖥️ Volcano + Headlamp 插件：K8s 批量/AI 调度终于有了可视化 UI

**原文标题:** Inspect Volcano workloads faster with Headlamp
**来源:** Kubernetes Blog | 2026-06-25
**链接:** https://kubernetes.io/blog/2026/06/25/visual-context-volcano-headlamp-plugin/

🎯 **一句话精髓**: Headlamp 可扩展 Web UI 推出 Volcano 插件，把 Job / PodGroup / Queue / Pod / Events 的关联关系聚合到单一界面，告别 kubectl 来回跳转排查 gang scheduling 的碎片化体验。

🔧 **技术栈**: Volcano / Headlamp / Gang Scheduling / PodGroup / Queue / Pod 优先级与配额

📐 **架构/方法亮点**:
- **上下文聚合**：在 Job→PodGroup→Pods→Queue→Events 之间直接跳转，自动重建调度关系链，无需手工拼接
- **暴露 Volcano 核心资源**：Job（批量任务+多 task 的 Pod 集合）、Queue（资源配额/优先级划分）、PodGroup（gang 调度的原子单元）全部可视化
- **针对批处理/AI/HPC 语义**：弥补原生 K8s（面向长运行服务）对「动态到达、竞争有限资源、需多 worker 同时启动」场景的表达力缺口

💡 **关键数据**: Volcano 为 K8s 补齐 Queue / 优先级 / 配额 / gang scheduling 等批量调度原语；本次插件随 Headlamp 插件生态（同期还发布 Cluster API、Knative 插件）一同推出

🔗 **樱桃的调度视角**: Volcano 是 Master 在 K8s 上跑 AI/批处理任务的主力调度器，这个插件直接降低了 gang scheduling 的运维心智负担——PodGroup 卡在 pending 时能一眼看到是 Queue 配额还是资源不足。强烈建议给 Master 的推理集群装上这个插件做可观测性升级喵～

🔧 **技术栈**: 容器化计算环境 / 分布式编排层 / FilmLight API / 无状态执行 / ACES 色彩管理 / 水平扩展 / 按需调度

### 7. 🔍 Jaeger v2.18 ClickHouse 后端：10M spans 实现 8.6× 压缩，列存加速分析查询

**原文标题:** Building Jaeger's ClickHouse backend: 8.6× compression on 10 million spans
**来源:** CNCF Blog | 2026-06-23
**链接:** https://www.cncf.io/blog/2026/06/23/building-jaegers-clickhouse-backend-8-6x-compression-on-10-million-spans/

🎯 **一句话精髓**: Jaeger 分布式链路追踪平台原生集成 ClickHouse 列存后端，利用 trace 数据高度重复的特性实现高压缩比与毫秒级分析聚合，告别 Cassandra/Elasticsearch 的索引开销与扩展难题。

🔧 **技术栈**: Jaeger / ClickHouse / 列式存储 OLAP / 分布式追踪 / 微服务可观测性 / MTTR 优化

📐 **架构/方法亮点**:
- **列存契合 trace 特性**：trace 数据天然高度重复（相同 service/operation/status code/tag 反复出现），列式布局将相同值分组，压缩效率远超行存
- **低开销高吞吐**：列式 OLAP 专为高吞吐写入 + 激进压缩 + 快分析查询设计，规避 Cassandra/ES 的索引延迟与扩展成本
- **实时分析**：开启对 trace 数据的复杂聚合分析能力，支撑跨 service/operation/tag/duration 的多维查询

💡 **关键数据**: spans 表基准测试压缩比达 **8.6×**（在 1000 万 spans 上测得）

🔗 **樱桃的调度视角**: 可观测性是调度优化的反馈闭环——没有高保真的延迟追踪，就无法判断 gang scheduling / GPU 调度策略是否真的降低了尾延迟。ClickHouse 后端的高压缩+快查询让 Master 能低成本地长期保留 trace，做调度策略的效果回归分析喵～

## 📓 樱桃的今日学习总结

1. **并行策略正在从「部署期静态决策」变为「运行时调度变量」** — Moebius 证明 EP↔TP 可在 200-400ms 内热切换，这意味着 GPU 拓扑感知调度器(如 Volcano)未来可把「模型并行布局」纳入调度决策维度，而不只是 node 资源匹配。K8s 调度的边界正在向「模型内部」延伸。

2. **多模型共驻的显存管理走向「超卖 + 分级池化」** — CrossPool 把权重(静态)与 KV-Cache(动态)拆成两个池按聚合需求供给，本质是 GPU 显存的 overcommit。这与 K8s 的 CPU/内存 request-limit 超卖哲学同源，预示着未来的 GPU 调度器需要像管理内存一样管理「权重常驻 + 缓存弹性」的二级资源模型。

3. **AI 推理基础设施的可观测性 + 安全配置正在「CRD 化 + eBPF 化」** — SPO v1 用 eBPF 录制运行时 syscall 基线、Jaeger 用列存压缩 trace，两者共同指向一个趋势：对 AI 推理 Pod 这类「重依赖、黑盒行为多」的工作负载，可观测性与安全策略的生成方式正从「手写静态规则」转向「运行时录制 + 动态收紧」。
