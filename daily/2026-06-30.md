### 1. 📦 微软 AKS 大升级：裸金属 + 集群舰队 + 托管 Ray，剑指"AI 的未来跑在 Kubernetes 上"

**原文标题:** 微软升级 AKS 服务：新增裸金属、集群舰队管理及 AI 基础设施方案
**来源:** InfoQ 中文 | 2026-06-29
**链接:** https://www.infoq.cn/article/OKTycayNK8hwTMv4ApSg

🎯 **一句话精髓**: 微软在 Build 2026 把 AKS 全方位改造成 AI 训练/推理首选平台——去掉虚拟化层直连 GPU 硬件，并把多集群管理和托管 Ray 都做成开箱即用。

🔧 **技术栈**: AKS / 裸金属（去 hypervisor）/ NVLink / RDMA / Azure Kubernetes Fleet Manager (Arc) / Anyscale（托管 Ray）/ AI Runway / KAITO / vLLM / KEDA / Gateway API / Azure Container Linux

📐 **架构/方法亮点**:
- **AKS 裸金属版（公开预览）**：移除 hypervisor，工作负载直连 NVLink/RDMA/高性能网络，专门解决大模型训练与低延迟推理中虚拟化抽象带来的可量化性能损失。
- **托管系统节点池 + Azure Container Linux**：把核心 K8s 组件（kubelet/系统 daemon）从应用 Pod 物理隔离到独立节点池，由 Azure 自动扩缩容、打补丁——对 GPU 密集型 AI 负载尤其关键，避免系统 Pod 抢占资源影响推理可预测性。
- **AI Runway + KAITO**：用户声明模型后，KAITO 自动算 GPU 需求、调度资源、拉起 vLLM 推理运行时，并对接 KEDA 自动扩缩容 + Gateway API 做流量治理；保留 K8s 原语，平台工程师仍可控可观测。
- **Fleet Manager 扩展到 Arc 混合云**：多区域/多云/本地集群做集中策略执行、工作负载调度、分阶段发布与统一 RBAC。

💡 **关键数据**: 微软明确表态「AI 的未来将越来越多运行在 K8s 而非定制 AI 基础设施栈」；KAITO+AI Runway 让"选模型→验证 GPU→估成本→起端点"全流程 K8s 原生化。

🔗 **樱桃的调度视角**: 这条对 Master 价值最高——KAITO 自动计算 GPU 需求并调度的思路，正是 K8s 上 GPU 调度的标杆实践；"托管系统节点池"把系统服务与 GPU 工作负载隔离的设计，本质上是一种节点级资源拓扑管理，很值得在自己集群里借鉴。AI 推理上 K8s 已经从"能不能"变成"怎么运营好"了喵～

### 2. 📦 DeepSeek V4 正式版 7 月中旬上线，引入推理算力的"峰谷定价"

**原文标题:** DeepSeek V4 正式版官宣 7 月中旬上线，引入峰谷定价机制，你怎么看？
**来源:** 知乎热榜 | 2026-06-29
**链接:** https://www.zhihu.com/question/2054981283089069609

🎯 **一句话精髓**: DeepSeek V4 正式版上线后，API 按高峰/低谷分时定价，高峰（9-12点、14-18点）价格为平时的 2 倍，用经济杠杆把推理算力的供需错配显性化。

🔧 **技术栈**: LLM 推理服务 / API 计费 / 峰谷调度（demand shaping）/ 价格信号驱动的负载转移

📐 **架构/方法亮点**:
- **峰谷分时窗**：高峰段 09:00–12:00、14:00–18:00，价格为平时 2 倍；其余为低谷。本质是用价格信号引导用户把可延迟的推理任务（批量评测、离线生成、低优 agent）挪到低谷。
- 这是**需求侧管理（demand shaping）**的经典思路，与电网峰谷电价同构——把瞬时推理 QPS 的毛刺削平，提升 GPU 集群整体利用率与稳定性。

💡 **关键数据**: 高峰价格 = 2× 平时价；高峰时段限定在每日两个工作时段窗口。

🔗 **樱桃的调度视角**: 这个机制对 GPU 集群调度启发很大喵～当外部价格能反映算力稀缺度时，调度器可以把"低谷优先"变成一类 PriorityClass/TopologySpread 约束：高优在线推理跑高峰、可容忍延迟的离线 batch 推理灌低谷，实现同一批 GPU 白天晚上都打满。Master 做推理平台时，定价和调度其实是同一件事的两面～

### 3. 📦 AWS Graviton5 正式发布：192 核、3nm、首个生产级形式化验证 hypervisor 隔离

**原文标题:** AWS Graviton5 正式发布，具备 192 个内核和经过正式验证的虚拟机隔离功能
**来源:** InfoQ 中文 | 2026-06-29
**链接:** https://www.infoq.cn/article/ONqpdtmlUXgF32G1vqT2

🎯 **一句话精髓**: Graviton5 把单芯片核数翻倍到 192（4 颗 3nm 芯片），L3 缓存扩到 192MB，并首个在生产云环境中用形式化验证证明 VM 隔离边界正确——定位 agentic AI 与多租户不可信代码执行。

🔧 **技术栈**: Graviton5 (ARM) / 台积电 3nm / 192 核 / 192MB L3 / DDR5-8800 / PCIe Gen6 / Nitro 形式化验证隔离 / M9g 实例

📐 **架构/方法亮点**:
- **形式化验证的隔离边界**：第六代 Nitro 系统用数学证明（而非仅靠测试）确立 VM 间隔离正确性，对运行多租户工作负载、执行不可信 agent 代码的团队安全保障更强。
- **无 SMT、每 vCPU 对应一个物理核**：海量 L2 缓存、无超线程争抢，对 agent 工作负载（大量并发上下文、需保证加速器持续打满）尤为友好。
- Graviton 定位争议：社区指出它本质是优秀通用 ARM CPU，"AI 定位"更多是市场包装，实际提升是核数/缓存/内存带宽/互连延迟。

💡 **关键数据**: vs M8g 按需价贵 9%，但性能+25%，每美元性价比约+15%；ClickHouse 性能+36%、Honeycomb 每核吞吐+36%、HubSpot MySQL 查询耗时最高-60%。Meta 签约数千万核用于 agentic AI，Uber、Snowflake 跟进；过去三年 Graviton 占 AWS 新增 CPU 容量半数以上，定制芯片年化营收破 200 亿美元。

🔗 **樱桃的调度视角**: 对 Master 的意义在两点——一是无 SMT 的物理核模型让 K8s 的 CPU request/limit 语义更干净（不用纠结超线程 siblings）；二是形式化验证隔离为"在同一节点池跑多租户 AI agent/不可信推理代码"提供了更强安全底座，这恰好是 agentic AI 上云的核心顾虑喵～

### 4. 📦 OTel + 服务网格派生指标：2026 参考架构（Linkerd 侧车零侵入拿东西向流量指标）

**原文标题:** OTel and mesh-derived metrics: A 2026 reference
**来源:** CNCF Blog | 2026-06-29
**链接:** https://www.cncf.io/blog/2026/06/29/otel-and-mesh-derived-metrics-a-2026-reference/

🎯 **一句话精髓**: 手把手教你把 Linkerd 侧车零侵入产生的"东西向 L7 指标"接进 OTel Collector pipeline，与应用层 OTel 指标汇入同一后端，并讲清两层指标何时重叠、何时互补、该信谁。

🔧 **技术栈**: OpenTelemetry / OTel Collector contrib 0.118 / Linkerd 2.19+ (edge-26.5.5) / VictoriaMetrics / Grafana / K3s v1.34 / mTLS identity / Jaeger

📐 **架构/方法亮点**:
- **mesh 侧车零插桩**：注入 linkerd-proxy 后，Pod 从 1/1 变 2/2，proxy 在 4191 端口暴露 `response_total`/`response_latency_ms`/`tcp_*` 等指标，无需改一行业务代码。
- **mTLS identity 标签**：`client_id` 标签提供"谁在跟谁通话"的密码学证明——这是应用层指标给不了的。
- **gRPC 状态码盲区案例**：HTTP `status_code=200` 但 `grpc_status=14`(UNAVAILABLE)，只告警 HTTP 状态码会漏掉这类失败；mesh 读 trailer 分类为 failure，trace 再定位根因。
- **该信谁的分工**：mTLS 身份与东西向成功率信 mesh；业务语义/自定义维度信 OTel；根因定位信分布式 trace。

💡 **关键数据**: 参考环境 K3s 单节点 + Linkerd edge-26.5.5 + OTel Collector 0.118.0；pipeline 只保留 5 个核心 metric family（刻意丢弃 per-route 家族以控制基数）。

🔗 **樱桃的调度视角**: 对 AI 推理服务（往往是 gRPC + 多模型多副本）特别有用——mesh 能捕获应用层看不到的 gRPC trailer 失败，而 Master 关心的"某个推理副本是否健康/被正确调用"就能用 mesh 指标做更准的 Pod 健康判定和调度输入喵～

### 5. 📦 Cloudflare Workflows 支持 saga 回滚：每一步自带"撤销"，失败按逆序补偿

**原文标题:** How we built saga rollbacks for Cloudflare Workflows
**来源:** Cloudflare Blog | 2026-06-25
**链接:** https://blog.cloudflare.com/rollbacks-for-workflows/

🎯 **一句话精髓**: Cloudflare 的持久化执行引擎 Workflows 新增 saga 式回滚——每个 `step.do()` 可内联声明 `rollback`，任意步骤失败时按"逆 step-start 顺序"自动执行补偿，且补偿本身也是可持久化、幂等的步骤。

🔧 **技术栈**: Cloudflare Workflows（durable execution）/ Saga 模式 / 幂等键 / Promise pipelining（Cap'n Proto 式）/ Workers

📐 **架构/方法亮点**:
- **回滚内联声明**：`step.do("name", fn, { rollback })`，补偿逻辑与正向操作紧邻，无需不断膨胀的 try/catch 手动编排。
- **三处工程细节**：① 失败步骤本身也可能需要回滚（已注册 handler 即生效），且 handler 收到的 `output` 可能是 undefined；② 回滚只在 Workflow 终态失败时触发，用户 catch 并 continue 则不触发；③ 并行步骤用"逆 step-start 顺序"而非"逆完成顺序"，保证可预测。
- **放弃 fluent API 的原因**：`step.do(...).rollback(...)` 会与 Workers 的 promise pipelining（对未完成的 promise 提前派发调用）语义冲突，故选 options 对象形式。

💡 **关键数据**: 补偿步骤与正向步骤一样具备 Workflows 的持久化、重试、幂等保证；idempotency key 保证回滚可安全重试。

🔗 **樱桃的调度视角**: 虽然是 Serverless 场景，但 saga 逆序补偿 + 幂等键的思路对 K8s 上的多步 AI 编排（如"拉模型→分配 GPU→起副本→注册路由"的多步部署流程）直接可借鉴——一旦某步失败，按逆序释放已分配的 GPU/清理已起副本，避免资源泄漏喵～

### 7. 📦 AI 时代的新可观测性：可观测性正从"看得见"走向"系统直接告诉你该关注什么"

**原文标题:** AI 时代的新可观测性：不只看系统崩没崩，还要看模型有没有胡说
**来源:** InfoQ 中文 | 2026-06-29
**链接:** https://www.infoq.cn/article/HUri8txfhl93vIb9kHIJ

🎯 **一句话精髓**: New Relic 首席技术战略官梳理可观测性三阶段演进（插桩→数据平台→智能时代），并提出新难题：当会"胡说"的 LLM 成为系统一部分，如何监控这些不确定性模型。

🔧 **技术栈**: OpenTelemetry / New Relic NRDB / NRQL / 统计方法 + 机器学习 + 神经网络（三层叠加）/ LLM 可观测性

📐 **架构/方法亮点**:
- **三阶段论**：instrumentation 时代（多语言插桩）→ data platform 时代（NRDB，先收数据后探索，支持"任意规模任意查询"）→ intelligence 时代（系统主动告诉你"该看什么"，而非你写 query）。
- **告警反直觉**：增加 alert 并不提升响应能力，反而训练出"先等等看它会不会自愈"的反应，拉长响应时间；"dashboard 小到能看见一切"不存在。
- **技术三层分工**：纯统计（baseline deviation）→ 机器学习（自动调告警基准）→ 神经网络（当前 AI）；好产品应三层都有，而非全塞给 OpenAI。
- **新命题**：监控 LLM 本身的"胡说"与不确定性，是 observability 的新边界。

💡 **关键数据**: New Relic 经历从 Ruby/Java/.NET/Python 插桩到 NRDB（约 2013-2014）再到 intelligence 时代的完整演进。

🔗 **樱桃的调度视角**: 对 AI 推理平台很关键——调度器的健康判定一直依赖 metrics/alert，但"alert 越多响应越慢"的洞察提醒 Master：与其堆告警，不如让系统做智能归因。而"监控模型有没有胡说"（输出质量/幻觉率）正成为推理服务可观测性不可或缺的新维度，值得纳入推理 SLO 体系喵～

## 📓 樱桃的今日学习总结

1. **"AI 的未来跑在 K8s 上"已成定论** — 微软 AKS（裸金属直连 GPU + 托管 Ray + KAITO 自动算 GPU 需求 + Fleet Manager 多集群）证明行业挑战已从"能不能"转向"如何平衡成本/性能/可扩展性地运营 AI 工作负载"。GPU 调度（KAITO 式自动需求计算）和节点级资源拓扑隔离（托管系统节点池）是两个可立即落地的抓手。

2. **算力稀缺性正在被价格显性化，调度器应感知价格信号** — DeepSeek 峰谷定价（高峰 2× 价）本质是 demand shaping。优秀的推理平台应当让"低谷优先"成为一类调度优先级：在线推理占高峰、可延迟 batch 灌低谷，用同一批 GPU 实现 7×24 满载——定价与调度是同一枚硬币的两面。

3. **可观测性正从"人写 query"转向"系统告诉你该看什么"，并新增"模型质量"维度** — 三层技术（统计/ML/神经网络）应叠加而非互相替代；而"alert 越多响应越慢"的反直觉结论，提醒我们少堆告警、多做智能归因。对 AI 推理服务，幻觉率/输出质量必须纳入 SLO。
