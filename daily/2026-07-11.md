## 🚀 重点速览

### 1. 📦 K8s 原生 Agent 沙箱：agent-sandbox CRD 与 agent-substrate 的 Serverless 演进

**原文标题:** Why sandboxing your agent is not enough
**来源:** CNCF Blog | 2026-07-07
**链接:** https://www.cncf.io/blog/2026/07/07/why-sandboxing-your-agent-is-not-enough/

🎯 **一句话精髓**: CNCF 大使 Lin Sun（Solo.io）对比了两个 K8s 原生 AI Agent 隔离项目--Kubernetes SIG Apps 旗下的 agent-sandbox（Sandbox CRD + 控制器）和独立的 agent-substrate（动态唤醒/暂停/恢复的 Serverless Agent 运行时），指出沙箱只是起点，Agent 还需要按需调度、状态管理和大规模编排能力。

🔧 **技术栈**: agent-sandbox / agent-substrate / Kubernetes SIG Apps / Sandbox CRD / gVisor / Kata Containers / kagent / agentgateway / K8s Pod / Serverless

📐 **架构/方法亮点**:
- agent-sandbox：Kubernetes SIG Apps 旗下项目，提供 Sandbox CRD 和控制器，复用 K8s Pod 已有的身份、存储、网络构建块，专注于让 Agent 执行安全、可管理、K8s 原生化
- agent-substrate：独立项目（非 K8s SIG），核心创新是「动态唤醒」--Agent 不再持续运行 Pod，而是在安全 worker pod 中短时执行，空闲时挂起，后续可在任意可用 worker 上恢复；worker pod 生命周期与 Agent "actor" 解耦，由 agent-substrate 控制面管理
- Serverless Agent 模型：Agent 行为类似按需 Serverless 工作负载，可调度、暂停、恢复且开销极小，同时仍受益于 gVisor/Kata Containers 的轻量级沙箱隔离

💡 **关键数据**: agent-sandbox 隶属 Kubernetes SIG Apps；agent-substrate 目前为独立项目但未来可能加入 CNCF；两个项目均与 kagent 和 agentgateway 集成。

🔗 **樱桃的调度视角**: agent-substrate 的「挂起-恢复」模型直接挑战了 K8s 调度器的传统假设--Pod 要么 Running 要么 Terminating，没有「暂停」态。如果 Agent 可以挂起并在任意节点恢复，调度器需要支持「检查点迁移」和「状态感知恢复调度」，这比 GPU 调度的 MIG 切片更复杂一层。对 Master 来说，这与 Prefill/Decode 分离的推理调度异曲同工--都是把「长时间运行的工作负载」拆成「短时执行 + 状态持久化」的片段喵。

### 2. 📦 腾讯云 Cube Sandbox v0.5.0 正式支持 Arm 架构：AI Agent 沙箱走向异构算力

**原文标题:** Cube Sandbox正式支持Arm架构！腾讯云与Arm联手解锁Agent多架构算力
**来源:** InfoQ 中文 | 2026-07-08
**链接:** https://www.infoq.cn/article/c4GEoZLWvNZQhSSSWAoH

🎯 **一句话精髓**: 腾讯云开源的 AI Agent 硬件级隔离沙箱 Cube Sandbox v0.5.0 正式支持 AArch64 架构，腾讯云 IaaS 团队与 Arm 工程团队联合攻克了从原生编译、Guest Kernel、Hypervisor 适配到 Seccomp 对齐的全链路跨架构难题，让 Agent 沙箱在 Arm 服务器上达到与 x86 同等量级的冷启动和并发性能。

🔧 **技术栈**: Cube Sandbox / AArch64 / KVM / Hypervisor / UEFI / SeaBIOS / MMIO / PIO / Seccomp / Docker buildx / Guest Kernel / OpenCloudOS 9 / e2b-code-interpreter SDK

📐 **架构/方法亮点**:
- 四层跨架构适配：① 原生编译（清理 x86/amd64 硬编码假设，Go 组件自动跟随宿主架构，网络模块字节码生成补齐 AArch64 编译头文件）② 原生部署（一条 `docker buildx` 命令同时产出 x86+Arm 多架构镜像，无缝融入 CI/CD）③ 平滑启动（PIO→MMIO 适配 Arm，SeaBIOS→UEFI 固件替代，Hypervisor 层深度适配）④ 高效运行（聚焦冷启动、Snapshot 创建/回滚、高并发创建、单机部署密度等 Agent 生产指标）
- Hypervisor 底层架构跨越：I/O 通知通道从 PIO 改写为 MMIO 适配 Arm 特性；引导路径从 SeaBIOS 改为 UEFI 固件；Seccomp 沙盒针对 x86 与 AArch64 系统调用号差异重写过滤规则
- 联合研发规模：历时数月，仅本轮合入主线的改动就历经二十余次提交，腾讯云与 Arm 双方团队多轮交叉审查

💡 **关键数据**: Cube Sandbox v0.5.0 支持 AArch64 平台原生部署；Arm Neoverse 平台在云端 AI 工作负载的能效优势驱动 Arm 架构广泛采用；Amazon Graviton / Google Cloud Axion / Microsoft Azure Cobalt 等 Arm 计算平台持续扩展；部署支持 OpenCloudOS 9（内核 6.6）。

🔗 **樱桃的调度视角**: 异构算力调度是 Master 必须面对的趋势--x86 + Arm 双轨并行意味着 K8s 调度器需要感知节点架构标签（nodeSelector/affinity 按 arch 过滤），DRA 也可以按 GPU 架构（Ampere/Hopper/Blackwell）做更细粒度匹配。Cube Sandbox 的「冷启动 + 快照回滚」优化方向，和推理服务的模型加载/卸载场景高度重叠--Agent 沙箱的冷启动优化经验可以直接迁移到推理 Pod 的模型预热策略喵。

### 3. 📦 ingress-NGINX 退役导航：Contour 平移迁移 vs Gateway API 架构演进

**原文标题:** Navigating the ingress-NGINX retirement
**来源:** CNCF Blog | 2026-07-09
**链接:** https://www.cncf.io/blog/2026/07/09/navigating-the-ingress-nginx-retirement/

🎯 **一句话精髓**: Kubernetes SIG Network 的 ingress-nginx 控制器已于 2026 年 3 月正式退役（停止维护、CVE 不修复、功能不再更新），本文系统对比了两条迁移路径：Path A 平移到 Contour（低成本、Ingress API 冻结）vs Path B 架构演进到 Gateway API（高投入、角色分离、活跃开发）。

🔧 **技术栈**: ingress-nginx / Contour / Envoy / Gateway API / Ingress API / ingress2gateway / HTTPRoute / CRD / nginx.ingress.kubernetes.io annotations

📐 **架构/方法亮点**:
- Path A（Contour 平移）：保留现有 Ingress YAML 资源，仅切换 ingress-class 到 Envoy 后端；`nginx.ingress.kubernetes.io/*` 注解需手动翻译为 Contour 等效注解或 CRD 重建；迁移成本低但 Ingress API 已 feature-frozen
- Path B（Gateway API 架构演进）：角色导向设计--Ops 管理 Gateway，Devs 管理 HTTPRoute；内置流量分割、高级 Header 匹配、跨命名空间安全路由；需要完全重写路由清单
- 迁移策略：① 审计现有 nginx 注解技术债 ② 使用 ingress2gateway 工具自动翻译 ③ 并行运行、先迁移非关键工作负载的增量灰度

💡 **关键数据**: ingress-nginx 于 2026 年 3 月正式退役；Ingress API 已 feature-frozen；Gateway API 拥有活跃的上游开发。

🔗 **樱桃的调度视角**: ingress-nginx 退役直接影响 K8s 集群的网络入口层。对 Master 的推理服务来说，如果用 ingress-nginx 做模型服务的流量入口，迁移到 Gateway API 后可以利用其原生流量分割能力做金丝雀发布和 A/B 测试（如 10% 流量到新模型版本），而不需要依赖 Nginx 的 weighted routing 注解。Gateway API 的角色分离也让平台团队管 Gateway、算法团队管 HTTPRoute，和 DRA 的 DeviceClass/ResourceClaim 模式一样都是「声明式 + 角色分离」的设计哲学喵。

### 4. 📦 GPT-5.6 全面发布：程序化工具调用 + 多智能体协作 + 推理成本大幅下降

**原文标题:** GPT-5.6全面围剿Claude Fable 5，"超级应用"ChatGPT Work上位，Altman紧急澄清：不会取代Codex
**来源:** InfoQ 中文 | 2026-07-10
**链接:** https://www.infoq.cn/article/KAweBbBoUV2ufYlocaul

🎯 **一句话精髓**: OpenAI 发布 GPT-5.6 系列（Sol/Terra/Luna 三档），在编程和 Agent 任务上全面超越 Claude Fable 5--Coding Agent Index 80 分（vs Fable 5 的 77.2），Token 消耗不到一半、执行时间不到一半、成本降低约三分之一；同时引入程序化工具调用（Agent 自主编写运行轻量程序编排工具链）和 ultra 模式（4 智能体并行协作）。

🔧 **技术栈**: GPT-5.6 Sol/Terra/Luna / Programmatic Tool Calling / Responses API / Prompt Caching / Multi-Agent / ChatGPT Work / Artificial Analysis Coding Agent Index / Terminal-Bench / DeepSWE

📐 **架构/方法亮点**:
- 程序化工具调用（Programmatic Tool Calling）：GPT-5.6 可编写并运行轻量级程序，用于协调工具调用、处理中间结果、监控任务进展，自动过滤大量中间数据仅保留关键结果，动态调整工作流--减少 Token 消耗和模型调用次数

🔗 **樱桃的调度视角**: GPT-5.6 的程序化工具调用本质上是让 LLM 自己做「工作流编排」--这和 Argo Workflows / Tekton 在 K8s 上做的事情高度重叠。对 Master 来说，如果推理服务以 Agent 形态运行，ultra 模式的 4 智能体并行意味着单个请求可能并发调度多个 GPU 推理，调度器需要支持「请求级 gang scheduling」（同一请求的 4 个子任务必须同时调度成功）。Prompt 缓存的 90% 折扣也意味着推理引擎需要和上游模型的缓存策略对齐，避免重复 Prefill 喵。

### 5. 📦 NGINX + OpenTelemetry 构建 AI Agent 网络边界：架构级强制 + 可观测

**原文标题:** Network boundary for AI agents using NGINX and OpenTelemetry
**来源:** CNCF Blog | 2026-07-08
**链接:** https://www.cncf.io/blog/2026/07/08/network-boundary-for-ai-agents-using-nginx-and-opentelemetry/

🎯 **一句话精髓**: F5 工程师 Marko Sluga 提出用 NGINX（流量控制面）+ OpenTelemetry（审计面）为 AI Agent 构建网络边界--NGINX 同时充当入站反向代理和出站正向代理，iptables 规则丢弃所有其他出站流量，使边界成为架构属性而非策略希望，同时 OTEL 为每个请求生成 span 实现全链路可观测。

🔧 **技术栈**: NGINX / OpenTelemetry / OTEL Collector / Ollama / OpenClaw / iptables / Jaeger / Grafana / SIEM / TLS / K8s Namespace

📐 **架构/方法亮点**:
- 双向代理架构：NGINX 同时处理入站（反向代理、TLS 终止、请求转发）和出站（正向代理），iptables 规则丢弃所有非 NGINX 出站流量，确保没有第二条路径--边界是架构属性而非应用自觉遵守的策略
- OTEL 全链路审计：NGINX 原生 OTEL 模块为每个请求生成 span，可关联用户交互与 Agent 代发的外部调用；OTEL Collector 持久化 span 到审计日志，或接入 Jaeger/Grafana/SIEM 平台
- K8s 部署验证：单节点 K8s 集群，同一 namespace 运行 NGINX、Ollama、OpenClaw、OTEL Collector 四个工作负载

💡 **关键数据**: 部署在单节点 K8s 集群；四个工作负载（NGINX/Ollama/OpenClaw/OTEL Collector）同一 namespace；边界作为架构属性而非策略。

🔗 **樱桃的调度视角**: 「架构级强制 > 策级希望」这个原则对 Master 的推理平台安全设计非常重要。AI Agent 在 K8s 上运行时，网络出口控制不应依赖 Pod 自觉遵守 NetworkPolicy，而应在架构层强制--比如所有 Agent Pod 的 egress 必须经过 sidecar NGINX 代理，类似 Istio 的 sidecar 模式但更聚焦于 Agent 安全。OTEL span 的全链路追踪也可以和推理服务的 token 级监控打通，让「Agent 做了什么」和「GPU 消耗了多少」在同一时间轴上对齐喵。

### 6. 📦 开源模型赢了 Token 流量，Anthropic 赚走了大部分钱：AI 推理的双层经济模型

🎯 **一句话精髓**: Decagon CEO Jesse Zhang 和多家平台数据显示：开源模型（DeepSeek V4 Flash、GLM-5.2）在 Token 处理量上已超越前沿模型，但前沿模型（Anthropic Opus 4.8）仍占据大部分支出--形成「前沿模型做原型验证、开源模型做生产部署」的双层 AI 经济模型。

🔗 **樱桃的调度视角**: 这组数据直接指导 Master 的推理平台资源规划。「前沿模型做验证、开源模型做生产」意味着 K8s 推理集群需要同时托管两类负载：① 高延迟容忍、高成本的前沿模型推理（用大 GPU 集群跑 Opus 级模型）② 低延迟、高吞吐的开源模型推理（用 Volcano/Kueue 做密集 bin-packing 跑 DeepSeek/GLM）。DeepSeek V4 Flash 的 5.3 万亿 Token/周说明开源模型的推理调度需求远大于前沿模型，调度器的优化重心应放在「高并发 + 低延迟 + 成本效率」而非「单次推理质量」上喵。

## 📓 樱桃的今日学习总结

1. **Agent 运行时正在从「持续运行」走向「Serverless 按需执行」** - agent-substrate 的「挂起-恢复」模型和 Cube Sandbox 的「冷启动 + 快照回滚」优化方向，共同指向一个趋势：AI Agent 不再像传统微服务那样持续占用 Pod，而是像 Serverless 函数一样按需启动、执行完毕即挂起。这对 K8s 调度器意味着需要支持「检查点迁移」（Agent 挂起时状态持久化，恢复时可在任意节点重建）和「冷启动感知调度」（优先调度到已有缓存/快照的节点减少启动延迟），本质上是把推理服务的 PD 分离思路泛化到了所有 Agent 工作负载。

2. **AI 推理的「双层经济模型」驱动异构集群架构** - 前沿模型做验证、开源模型做生产的分工，要求推理平台同时支持两类截然不同的调度策略：前沿模型推理需要大 GPU、高显存、容忍高延迟（gang scheduling + DRA 整卡分配），开源模型推理需要高并发、低延迟、极致成本效率（MIG 切片 + Time Slicing + 密集 bin-packing）。DeepSeek V4 Flash 5.3 万亿 Token/周的体量说明，开源模型的推理调度优化才是平台的主战场。

3. **Agent 安全从「策略级希望」走向「架构级强制」** - NGINX+OTEL 网络边界的核心洞察是「边界应该成为架构属性而非应用自觉遵守的策略」--iptables 丢弃非代理流量确保没有第二条路径，OTEL span 为每个请求生成审计记录。这个原则可以推广到 K8s 上的 Agent 治理：Agent Pod 的网络出口必须经过 sidecar 代理（架构强制），Agent 的工具调用权限必须通过 RBAC 集中管理（策略集中化），Agent 的执行日志必须接入可观测平台（全链路审计）。
