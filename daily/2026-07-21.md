## 🚀 重点速览

### 1. 📦 别再只盯吞吐：LLM 推理该用 goodput 衡量

**原文标题:** Why goodput matters more than throughput for LLM serving
**来源:** CNCF Blog | 2026-07-20
**链接:** https://www.cncf.io/blog/2026/07/20/why-goodput-matters-more-than-throughput-for-llm-serving/

🎯 **一句话精髓**: 吞吐量只数「出了多少盘菜」，goodput 才数「有多少食客满意离场」——只有同时满足 TTFT/TPOT 延迟 SLO 的请求才算 goodput；调 vLLM 时吞吐能一路涨，而服务其实正在悄悄变差。

🔧 **技术栈**: vLLM / Qwen2.5-7B / NVIDIA A10G (24GB) / Amazon EKS / GuideLLM / Prometheus / Grafana / NVIDIA DCGM exporter

📐 **架构/方法亮点**:
- **goodput 定义**: 每秒「完成且满足延迟目标」的请求数；LLM 的延迟目标通常是 TTFT（首 token 时间）与 TPOT（每输出 token 时间），任一超标即「冷盘」，计入吞吐但不计入 goodput
- **只调 3 个 vLLM 参数**: `gpu_memory_utilization`、`max_num_batched_tokens`（批 token 预算）、`max_num_seqs`（并发序列上限）——三者共同决定 KV cache 显存、每批工作量与并发请求数，是 operator 不动模型/硬件/集群就能安全改的旋钮
- **两种负载 + 窗口规则**: GuideLLM 的 rate sweep（渐进加压读延迟）对 throughput run（打满找天花板）；评分取连续 8 个稳定数据点而非单点，规避瞬时抖动

💡 **关键数据**: 三类 workload 的 SLO 痛点不同——chatbot（~485/121 tokens，紧 TTFT）、reasoning（~400/4000，TPOT）、agentic（~512/128 per-call，TTFT 累加）；优化目标为最大化 `prefill_token_throughput + decode_token_throughput`，约束平均 TTFT ≤ 1.5s；刻意选单卡 A10G（非 H100）只为让实验便宜可重复

🔗 **樱桃的调度视角**: 这条直击 Master 的推理调度——goodput 是比 QPS 更该盯的 SLO。若 K8s HPA/Volcano 只按吞吐扩容，会在延迟劣化时仍判定「健康」而错过扩容时机。和昨天云天励飞「Token 成本作为系统级 KPI」呼应：调度器未来要做「goodput 感知」的弹性决策，而不是资源利用率感知。

### 2. 📦 Grab Palana：给自治 AI Agent 建零信任 K8s 运行时

**原文标题:** Grab 构建安全的智能体 AI 工作负载平台（Grab Builds Secure Agentic AI Workload Platform）
**来源:** InfoQ 中文 | 2026-07-20
**链接:** https://www.infoq.cn/article/yQb5j83sxQhXB1EDJrS1

🎯 **一句话精髓**: Grab 为运行自治 AI Agent 构建 Palana——零信任隔离运行时，每个 Agent 独占 K8s 命名空间 + 严格 RBAC + 自定义网络策略，密钥经 Envoy+OPA 代理动态替换，控制平面带「断网开关」与外部回收器。

🔧 **技术栈**: Kubernetes / Namespace / RBAC / NetworkPolicy / HashiCorp Vault / Envoy / Open Policy Agent (OPA) / Custom Operator / CRD

📐 **架构/方法亮点**:
- **每 Agent 一个隔离单元**: 独占命名空间 + 隔离 service account + 自定义网络策略；持久化本地存储保存 Agent 状态/内存，支持长时间异步工作流
- **密钥解耦为两层**: 「Agent 可读凭据」vs「仅代理可用密钥」——敏感凭据（版本控制 PAT、模型网关 API key）存 HashiCorp Vault，Agent 容器只配占位令牌；发起外部调用时安全代理拦截、验证目标、动态把占位符替换为真实密钥，原始密钥从不进容器环境/内存/日志
- **集中式出口控制**: 所有出站 HTTP/HTTPS 经 Envoy 代理 + OPA 外部授权服务路由，MITM 证书终止做实时解密、头部评估、端点验证与令牌替换，并生成结构化审计跟踪；网络级「断网开关」可从控制平面直接禁用网络策略，外部 reaper trigger 空闲触发关机
- **CR + operator 管理生命周期**: 每个 Agent 建模为由自定义 K8s operator 调和的 CR，operator 动态配置命名空间/存储/网络策略/入口

💡 **关键数据**: 为 OpenClaw 等智能体框架搭原型后认识到需系统化基础设施方案；支持数百个并发 Agent 工作负载在生产集群中做编程化审计、更新与管理

🔗 **樱桃的调度视角**: 这是 Agent 时代 K8s 安全运行时的范本，与昨天 AWS Claude 网关（中心化凭证）互补。对 Master，Agent 不该直接拿长期 API key，而该走「占位令牌 + 代理动态替换」的零信任模式；用 CR + operator 管理 Agent 生命周期，正是 K8s 声明式对「非确定性行动体」的正确用法。

### 3. 📦 Headlamp 的 Kubeflow 插件：把 ML 的 CRD 搬进通用 K8s UI

**原文标题:** Operating AI/ML Workloads on Kubernetes: A Headlamp Plugin for Kubeflow
**来源:** Kubernetes Blog | 2026-07-14
**链接:** https://kubernetes.io/blog/2026/07/13/introducing-headlamp-plugin-for-kubeflow/

🎯 **一句话精髓**: Headlamp（K8s SIG UI，Apache 2.0）的 Kubeflow 插件直接读 K8s API server，把 Kubeflow 的 CRD（Notebook/Pipeline/Katib/TrainJob/Spark）收进通用 K8s UI，operator 排查 ML 故障不必再退回 kubectl。

🔧 **技术栈**: Headlamp / Kubernetes SIG UI / Kubeflow / CRD / Katib / TrainJob / TrainingRuntime / SparkApplication / ownerReferences

📐 **架构/方法亮点**:
- **覆盖五大组件族**: Notebooks（Notebook/Profile/PodDefault）、Pipelines（Pipeline/PipelineVersion/Run/RecurringRun/Experiment）、Katib（Experiment/Trial/Suggestion）、Training（TrainJob/TrainingRuntime/ClusterTrainingRuntime）、Spark（SparkApplication/ScheduledSparkApplication）
- **直读 API server**: 直接读 K8s API server，Pod conditions、`ImagePullBackOff`/`OOMKilled`、PVC 等待等 K8s 层故障原因一目了然；Pipelines 视图不查 Kubeflow Pipelines API 服务或后端数据库，即使该服务不可用也能查 pipeline 状态
- **关系图视图**: 注册 Headlamp map source，把资源渲染成图节点，按 `.metadata.ownerReferences` 画边；Notebook 详情显示 CPU/内存/GPU 的 requests/limits、volume mounts 及 backing 类型、sidecar、node tolerations

💡 **关键数据**: 插件按集群已安装的 Kubeflow API group 自动发现并只显示对应 section；Katib 视图显示调参算法、搜索空间、每个 Trial 实时状态、当前最优 Trial 的指标值与参数赋值

🔗 **樱桃的调度视角**: 对 Master 做 K8s 上 AI/ML 运维，这是「CRD-heavy 平台」的可视化范式——和 7 月 6 日报的 Volcano Headlamp 插件同源（同一套可扩展插件体系）。Kubeflow 的 `TrainJob`/`TrainingRuntime` 正是分布式训练调度资源，能在一个 UI 里同时看调度状态 + Pod 级故障，排障效率大增，是「调度可视化」的又一块拼图。

### 4. 📦 自定义指标 Exporter：给 HPA 接上业务信号

**原文标题:** Building a Custom Metrics Exporter for Kubernetes
**来源:** Kubernetes Blog | 2026-07-14
**链接:** https://kubernetes.io/blog/2026/07/14/custom-metrics-exporter-kubernetes/

🎯 **一句话精髓**: 手把手用 Go 写一个 Prometheus 自定义指标 exporter，把「队列深度/活跃连接/批处理耗时」等业务信号暴露到 `/metrics`，再接 HPA 做自动扩缩容——补上 K8s 内置 CPU/内存指标不够用的缺口。

🔧 **技术栈**: Go / Prometheus client_golang / Counter / Gauge / Histogram / HPA / /metrics endpoint / ServiceMonitor

📐 **架构/方法亮点**:
- **指标类型选择**: Counter（只增不减，总数）、Gauge（可升可降，队列深度/活跃连接）、Histogram（分布，请求延迟 p99/p50）；命名规范 `<namespace>_<name>_<unit>`
- **注册与采集**: Go client 用 `prometheus.MustRegister` 注册（重复注册会 panic，把配置错误暴露在启动期而非静默运行）；polling loop goroutine 周期刷新真实值，刷新间隔应短于 Prometheus scrape 间隔
- **三类典型指标**: `worker_jobs_processed_total`（counter，按 status 分）、`worker_queue_depth`（gauge）、`worker_job_duration_seconds`（histogram）

💡 **关键数据**: 大多数集群默认 scrape 间隔 15s；示例 polling 间隔 5s 以留足余量，确保每次 scrape 看到新鲜值

🔗 **樱桃的调度视角**: 自定义指标是 K8s 弹性调度的「信号源」——对 Master，AI 推理服务该用 goodput/队列深度等业务指标驱动 HPA，而不是 CPU 利用率（推理是 GPU-bound，CPU 利用率毫无意义）。结合第 1 条 goodput，完全可以把 goodput 做成 custom metric 接 HPA，实现「延迟感知的自动扩缩容」——这正是把第 1 条理论落地的工程抓手。

### 5. 📦 ProGlove 把 Lambda 扩到 100 万：每租户一账户 + 抖动调度

**原文标题:** 亚马逊云科技概要介绍了某客户如何将 Lambda 函数扩展至 100 万个（AWS - ProGlove scales Lambda functions to over 1 million）
**来源:** InfoQ 中文 | 2026-07-20
**链接:** https://www.infoq.cn/article/bkhkOrQYPuIN672g20hi

🎯 **一句话精髓**: 工业可穿戴厂商 ProGlove 把 SaaS 平台扩展到数千个专属客户账户、100 万+ 个 AWS Lambda 函数，靠「每租户一账户」+ CloudFormation StackSets 自动化 + 「缩减至零」策略，把每账户闲置成本压到 < 1 美元/月。

🔧 **技术栈**: AWS Lambda / 每租户一账户 / CloudFormation StackSets / AWS Organizations / Step Functions / DynamoDB / EventBridge / SQS / 抖动执行窗口 / 事件驱动触发

📐 **架构/方法亮点**:
- **租户隔离 + 自动化**: 每客户账户含 5-15 个 Lambda 函数，由 Step Functions 状态机协调——采集扫描器读数 → 持久化 DynamoDB → 发布业务事件到共享 EventBridge 总线供分析平台消费；AWS Organizations + Step Functions + StackSets 整合为单管道创建/更新每个账户
- **消灭「自酿 DDoS」**: 早期 cron 风格调度在所有账户同一时间触发同一函数造成「自酿 DDoS」；改用抖动执行窗口 + 事件驱动触发器平滑各区域负载，不依赖预留并发或预配置容量
- **可观测性成本治理**: 每个 Lambda 生成日志/指标，乘以数千账户几乎占满账单——把高优先级故障整合到中央死信队列、删除未用 SQS 队列

💡 **关键数据**: 100 万+ Lambda 函数、数千账户；账户数超 50 后运维摩擦显现；闲置成本 < 1 美元/账户/月；DoorDash 日超 1000 万次 API 调用走类似约束式自动化；Capital One 亦强调标准化部署/可观测/治理

🔗 **樱桃的调度视角**: 这是 Serverless 规模化的实战教训——「自酿 DDoS」（同步定时器）对 Master 做 K8s CronJob/Volcano 批调度同样警醒：大规模定时任务必须加 jitter 打散，避免 thundering herd。scale-to-zero + 抖动调度是降本关键，K8s 上可用 KEDA + 定时源 + 抖动实现类似效果。

### 6. 📦 KubeCon EU 实录：在银行里用开源搭出可预测的云原生平台

**原文标题:** 开源如何促进平台构建过程中的协作（How open source fosters collaboration during platform building）
**来源:** InfoQ 中文 | 2026-07-20
**链接:** https://www.infoq.cn/article/MgFehj2q2FJFrf6XvEco

🎯 **一句话精髓**: KubeCon EU 演讲「在银行中构建云原生文化」指出——平台是协作系统，工程师信任平台源于「行为可预测」而非功能多；开源提供了团队/供应商/工具间共享的标准与统一语言，采用 K8s 等于加入一个有明确规范的社区。

🔧 **技术栈**: Kubernetes / 平台工程 / 开源治理 / 标准化 / 自动化 / 「天才吧」开放式支持 / 认知负荷管理

📐 **架构/方法亮点**:
- **信任来自可预测性**: 无论部署到开发/测试/生产，都应有相同体验与预期；明确平台职责范围与 SLA，让各团队清楚「哪些由平台负责、自身运维从何处开始」
- **减轻认知负荷**: 不把 K8s 每项功能都暴露，而是提供合理默认值和有指导性的工作流，让开发者专注应用而非理解底层基础设施复杂性
- **「天才吧」+ 透明优先**: 跳出纯技术支持，开设与用户一起解决问题的开放式环节；不等到臻于完美才分享，让用户成为早期测试者和合作伙伴
- **社区驱动内化**: 开源社区记录决策过程、积极回馈、明确责任归属——团队把这些实践内化，工具要求的工作方式逐渐变成自己的

💡 **关键数据**: 开源在银行并非显而易见的选择（技术支持/责任归属有疑虑），但被视为「指南针」而非教条；采用 K8s = 加入有明确规范、预期和协作方式的社区

🔗 **樱桃的调度视角**: 对 Master 做内部平台，这条点出平台成功的核心是「可预测行为 + 降低认知负荷」——K8s 功能全但暴露太多会让应用团队无所适从。提供合理默认 + 引导式工作流（如 Backstage 黄金路径）比堆功能更能赢得信任；调度策略也该如此：给团队稳定可预期的调度行为，而不是频繁变动的优先级规则。

### 7. 📦 连续授权：把每次敏感操作都变成独立决策点

**原文标题:** 为敏感的云系统设计连续授权
**来源:** InfoQ 中文 | 2026-07-20
**链接:** https://www.infoq.cn/article/1KqNGCc9UamMzojQYs9h

🎯 **一句话精髓**: 大多数云系统只在登录时做一次授权决策，会话期间高风险操作缺乏持续校验——连续授权把每个敏感操作变成独立决策点，基于行为基线/位置/时间/数据敏感性实时评估「现在是否应该」而非「能否」。

🔧 **技术栈**: RBAC / 连续授权 / Policy Decision Point (PDP) / Risk Signal Aggregation Layer / 行为基线 / 设备指纹 / 影子模式 / Zero Trust / API 网关

📐 **架构/方法亮点**:
- **「能做」vs「应该做」**: RBAC 只判断角色能否调 API/查表，不评估当前是否应从该位置、针对该数据、以该数据量执行；例如客服账号能 `SELECT * FROM customers WHERE segment='high_value'` 检索远超实际需要的数据
- **PDP + 风险信号聚合层**: 策略决策点位于应用逻辑与数据访问间，类似 OAuth 资源服务器/API 网关但决策更丰富；风险信号聚合层后台异步更新行为画像，请求到达时评估是否偏离基线，环境信号（IP 段/设备指纹）以极低成本提供上下文
- **缓存分层 + 隐私保护审计**: 策略规则短期缓存、用户风险基线后台异步刷新、低风险操作复用临时决策；审计记录上下文化证据（哈希化用户 ID + 风险等级 + 决策）而非「用户 X 访问了病人 Y 的病历」，避免审计系统沦为新的敏感数据仓库
- **三阶段滚动发布**: 影子模式（评估记录不影响操作）→ 有限强制（要求理由/警告/临时覆盖）→ 全面强制（高风险阻断/升级/审批）

💡 **关键数据**: 生产模式最早且最有效的控制聚焦批量导出、跨租户访问、机器对机器访问；夜间批处理在预期窗口处理 10000 条记录为正常，同样负载在异常时间或访问更大数据集则触发调查

🔗 **樱桃的调度视角**: 这与第 2 条 Grab Palana 的「占位令牌 + 代理动态授权」是同一思想的不同层——都是把授权从静态前置转向运行时连续决策。对 Master 的 K8s 平台，准入控制（Kyverno/OPA）目前多是「创建时一次性校验」，连续授权启发我们：敏感操作（批量删 Pod、跨 namespace 访问 secret）应做运行时持续校验并结合行为基线检测异常。

## 📓 樱桃的今日学习总结

1. **AI 推理的度量标尺正在从「吞吐」转向「goodput」** - 今天 goodput 这条 + 自定义指标 exporter 这条合在一起读，指向一个清晰的工程闭环：推理服务该用 goodput/队列深度等业务指标驱动 HPA，而不是 CPU 利用率。K8s 调度器与弹性系统若仍按 QPS/利用率扩容，会在延迟劣化时误判「健康」。Master 关注的调度方向，正在从「资源利用率感知」进化为「SLO/goodput 感知」——把 goodput 接成 custom metric 驱动 HPA，就是落地这个思路的第一步。

2. **Agent 工作负载需要「运行时治理」而非「创建时校验」** - Grab Palana（零信任运行时 + 占位令牌动态替换）和连续授权（每个敏感操作独立决策点 + 行为基线）从不同层得出同一结论：Agent 是非确定性、自治的行动体，静态前置授权不够用，必须在运行时持续校验。在 K8s 上，这对应「CR + operator 管生命周期 + 准入/出口代理做运行时策略」的组合。Master 设计 Agent 平台时，「按需唤醒 + 沙箱隔离 + 运行时连续授权」应作为核心架构原则。

3. **大规模调度的两个隐形杀手：thundering herd 与认知负荷** - ProGlove 的「自酿 DDoS」（同步定时器）和 KubeCon EU 的「平台要可预测、降认知负荷」看似无关，实则都在讲同一件事：规模化系统的成败往往不在功能强弱，而在是否避免了「同步竞争」和「不可预期」。前者要求定时/批调度任务加 jitter 打散，后者要求调度策略给团队稳定可预期的行为。这两点对 Master 做 K8s 调度平台都是朴素但高频的坑。
