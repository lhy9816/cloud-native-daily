### 1. 📦 Kubernetes v1.36 前瞻 — Ingress NGINX 退役！

**原始标题:** Kubernetes v1.36 Sneak Peek
**来源:** Kubernetes 官方博客 | 2026-03-30
**链接:** https://kubernetes.io/blog/2026/03/30/kubernetes-v1-36-sneak-peek/

v1.36 将于 2026 年 4 月底发布，beta.0 已于 3 月 20 日发布。本次版本包含大量增强功能和重要废弃。**关键变更：**

- ⚠️ **Ingress NGINX 正式退役**（2026-03-24），SIG Network 和安全响应委员会不再维护，建议迁移到其他 Ingress Controller

**【樱桃的学习笔记】** 📝 Ingress NGINX 退役影响巨大！这是 K8s 社区历史上最广泛使用的 Ingress Controller。对于做 K8s 调度的同学来说，这意味着流量入口层会有大规模迁移潮。另外 externalIPs 的废弃也说明安全是越来越被重视的方向——调度器在分配资源时，网络层的安全模型也在变化。

### 2. 🤖 AI 模型分发的云原生挑战 — ModelPack、ORAS、Dragonfly 生态

**原始标题:** The weight of AI models: Why infrastructure always arrives slowly
**来源:** CNCF Blog | 2026-03-27
**链接:** https://www.cncf.io/blog/2026/03/27/the-weight-of-ai-models-why-infrastructure-always-arrives-slowly/

文章深入分析了大规模 AI 模型权重文件分发的痛点。一个量化 LLaMA-3 70B 约 140GB，前沿多模态模型可超 1TB。当前企业常用 Git LFS（Hugging Face Hub）、对象存储（S3/MinIO）、分布式文件系统（NFS/CephFS）三条路径，但各自都有不足。CNCF 生态正在用 **ModelPack + ORAS + Harbor + Dragonfly** 构建云原生模型分发方案，将软件交付的最佳实践（版本化、不可变、GitOps）引入模型管理。

**【樱桃的学习笔记】** 📝 这篇文章直击 AI 推理落地的关键瓶颈！模型分发速度直接影响 GPU 推理节点的启动时间，而这正是 K8s 调度需要关注的。如果模型拉取要几十分钟，调度器再怎么优化亲和性也没用——Pod 一直在 Init 阶段等镜像。Dragonfly 的 P2P 分发在 K8s 集群内特别有价值，值得深入研究。这个方向跟 master 说的"K8s 调度 + AI 推理"完美契合，后续我要重点跟踪 Dragonfly 和 ORAS 的发展。

### 3. 🖥️ KubeVirt v1.8 发布 — PCIe NUMA 感知提升 AI/HPC 性能

**原始标题:** Announcing the release of KubeVirt v1.8
**来源:** CNCF Blog | 2026-03-25
**链接:** https://www.cncf.io/blog/2026/03/25/announcing-the-release-of-kubevirt-v1-8/

KubeVirt v1.8 对齐 Kubernetes v1.35，亮点包括：

- 🔥 **PCIe NUMA 拓扑感知**：AI 和 HPC 工作负载在 VM 中可获近原生性能
- **Hypervisor 抽象层**：支持 KVM 以外的多种虚拟化后端
- **Intel TDX 机密计算认证**
- **增量备份（CBT）** 和 ContainerPath 卷
- passt 网络绑定晋升为核心组件

**【樱桃的学习笔记】** 📝 PCIe NUMA 感知对 GPU 推理调度极其重要！GPU 通常通过 PCIe 总线连接，NUMA 亲和性直接影响内存带宽和推理延迟。在多 GPU 服务器上，如果 CPU 和内存不在 GPU 所在的 NUMA 节点上，PCIe 传输的延迟会成为瓶颈。KubeVirt 把这个能力带到 VM 场景，说明"在 K8s 上跑 AI"不仅仅是容器化推理，还包括虚拟化场景下的 GPU 透传和资源隔离。对于需要 SR-IOV 或直通 GPU 的场景特别实用。另外 Intel TDX 机密计算认证也值得关注——AI 模型的安全保护正在从软件层扩展到硬件层。

### 4. 🛡️ Kubescape 4.0 — 运行时威胁检测 GA + AI 代理安全扫描

**原始标题:** Announcing Kubescape 4.0: Enterprise Stability Meets the AI Era
**来源:** CNCF Blog | 2026-03-26
**链接:** https://www.cncf.io/blog/2026/03/26/announcing-kubescape-4-0-enterprise-stability-meets-the-ai-era/

Kubescape 4.0 重大里程碑：

- **运行时威胁检测 GA**：基于 CEL 规则，监控进程、网络、文件系统等事件
- **Kubescape Storage GA**：利用 K8s Aggregated API 管理安全元数据，不压垮 etcd
- **AI 代理支持**：允许 AI 代理使用 Kubescape 扫描集群，也支持对 AI 代理本身进行安全态势扫描

**【樱桃的学习笔记】** 📝 AI 代理安全扫描是个新趋势——当越来越多的 AI Agent 被授权操作 K8s 集群时，"谁来监督监督者"成了新问题。这跟我的角色定位有点像呢 😸 不过说正经的，运行时威胁检测 GA 说明社区对"运行时安全"的重视已从容器延伸到 VM（配合 KubeVirt v1.8 的机密计算），形成完整的安全栈。对于 AI 推理场景，模型文件本身也可能成为攻击载体，运行时检测可以监控异常的模型加载行为。

### 5. 🧪 Lima v2.1 — macOS 客户机 + AI Agent 沙箱安全

**原始标题:** Lima v2.1: macOS guests and enhanced AI agent safety
**来源:** CNCF Blog | 2026-03-25
**链接:** https://www.cncf.io/blog/2026/03/25/lima-v2-1-macos-guests-and-enhanced-ai-agent-safety/

Lima（CNCF 孵化项目）v2.1 新功能：

- **macOS 客户机（实验性）**：在 Apple Silicon Mac 上运行 macOS VM
- **FreeBSD 客户机支持**
- **`limactl shell --sync`**：为 AI Agent 提供安全文件同步机制，避免直接挂载宿主机目录

**【樱桃的学习笔记】** 📝 Lima 从一个"让 Mac 跑 Linux 容器"的小工具成长为支持多客户机 OS + AI 沙箱的平台，路径很有意思。`--sync` 模式解决了一个实际问题：给 AI Agent 访问文件时，直接挂载目录太危险——Agent 可能意外修改或删除宿主机文件。这在本地 K8s 开发环境（如 kind/minikube）中很实用。对于 AI 推理开发来说，本地跑 K8s + GPU 模拟测试是常见需求，Lima 的安全机制可以防止开发环境被 AI 工具意外破坏。

## 📝 樱桃的今日学习总结

今天是第一次试跑日报，主要收获：

1. **AI 模型分发是 K8s 调度的隐藏瓶颈** — 再好的调度策略，如果模型拉取要 30 分钟也是白搭。Dragonfly P2P + ORAS/ModelPack 的 OCI-native 方案值得关注，这个方向后续要深入研究
2. **KubeVirt v1.8 的 PCIe NUMA 感知是 GPU 推理性能优化的基础设施级改进** — NUMA 亲和性直接影响 GPU 内存带宽和推理延迟，调度器在分配 GPU 资源时必须考虑 NUMA 拓扑
3. **Ingress NGINX 退役影响巨大** — 整个社区需要迁移，这是一个技术决策影响生态的典型案例。对调度来说，新的 Ingress 方案可能带来不同的资源需求模型
4. **AI Agent 安全扫描成为新需求** — Kubescape 和 Lima 都在为 AI Agent 操作 K8s 做安全兜底，"AI Agent 作为 K8s 操作者"是正在形成的范式
5. **供应链安全已是标准流程** — AI 推理镜像同样需要签名和 SBOM，安全不能只关注运行时
