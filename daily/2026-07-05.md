### 1. 📦 Kubernetes DRA 正式 GA：GPU 调度进入「声明式设备分配」时代

**原文标题:** Understanding dynamic resource allocation in Kubernetes
**来源:** CNCF Blog | 2026-07-01
**链接:** https://www.cncf.io/blog/2026/07/01/understanding-dynamic-resource-allocation-in-kubernetes/

🎯 **一句话精髓**: Dynamic Resource Allocation (DRA) 在 K8s v1.35 达到 GA，NVIDIA 将 dra-driver-nvidia-gpu 迁入 Kubernetes SIGs，GPU 分配从 Device Plugin 时代的「整卡占用」进化为声明式、可优先级回退的精细化调度。

🔧 **技术栈**: Kubernetes v1.35 DRA / ResourceClaim / ResourceClaimTemplate / ResourceSlice / DeviceClass / NVIDIA GPU Operator v26.3.1 / NVIDIA DRA Driver GPU v25.12.0 / Cluster API + OpenStack

📐 **架构/方法亮点**:
- **ResourceSlice 自动发现设备**：DRA driver 在每个 Node 上自动注册所有 GPU 设备，通过 `.spec.pool.generation` 和 `.spec.pool.resourceSliceCount` 让调度器判断设备列表是否完整；单个 Pool 超过 128 条目时自动拆分多个 ResourceSlice
- **firstAvailable 优先级回退**：通过 CEL 表达式 `device.attributes["gpu.nvidia.com"].productName == "NVIDIA RTX A5000"` 精确选择 GPU 型号，支持 ranked fallback（如优先 A5000，回退 T10），实现 Deployment 弹性扩缩容
- **GPU 共享与时间片切分**：Scenario I 演示两个 Container 共享一张 GPU；Scenario IV 启用 `TimeSlicingSettings` Feature Gate 实现 GPU Time Slicing，单卡可被多个 Pod 分时复用

💡 **关键数据**: 测试环境使用 CNTUG Infra Labs 的 Equinix 东京数据中心，包含 A5000 和 T10 两种 GPU 节点；DRA Driver GPU 版本 25.12.0，GPU Operator 版本 v26.3.1

🔗 **樱桃的调度视角**: 这是 Master 最该关注的内容！DRA GA 意味着 GPU 调度不再需要依赖 NVIDIA Device Plugin 的「粗粒度整卡分配」模式。通过 ResourceClaimTemplate + CEL 选择器，可以实现「按显存容量筛选」「按型号优先级回退」「多容器共享单卡」等精细化策略——这正是 AI 推理服务在 K8s 上做多模型混部的基础设施。建议尽快在测试环境验证 DRA + TimeSlicing 组合对推理吞吐的影响。

### 2. 📦 Apple 把机密推理搬上谷歌云：三层硬件信任 + Blackwell GPU 的 ZOA 模式

**原文标题:** 苹果首次将私有云计算平台扩展至谷歌云
**来源:** InfoQ 中文 | 2026-07-04
**链接:** https://www.infoq.cn/article/UoJtxVXj0d1QT1ftyjtd

🎯 **一句话精髓**: Apple 将最注重隐私的 Private Cloud Compute (PCC) 平台首次扩展到非自有基础设施（谷歌云），通过三层硬件信任机制（NVIDIA Blackwell 机密计算 + Intel TDX + Google Titan 芯片）实现端到端机密推理管道。

🔧 **技术栈:** NVIDIA Blackwell GPU / Intel TDX / Google Titan 芯片 / TEE（可信执行环境）/ Prompt Encryption SDK / Confidential G4 VM / DigiCert 第三方认证

📐 **架构/方法亮点**:
- **三层可信根设计**：基于 Blackwell GPU 的 NVIDIA 机密计算（GPU 层）、基于 CPU 的 Intel TDX（CPU 层）、Google Titan 芯片作为可信根（硬件层），任何单一供应商被入侵都不足以破坏验证链
- **加密账本独立追踪**：Apple 对所有谷歌云 PCC 硬件维护一份加密可验证的仅追加账本，独立追踪每个物理组件，而非依赖谷歌自身的证明
- **ZOA 模式定义**：提出云端推理三层级框架——Eyes on with retention（可视化+保留）、ZDR（零数据保留）、ZOA（零运维人员访问，安全隔离区+认证+加密），Apple PCC 采用最高级别的 ZOA 模式

💡 **关键数据**: 谷歌同步发布搭载 NVIDIA RTX PRO 6000 Blackwell GPU 的 Confidential G4 VM 预览版；开源 Prompt Encryption SDK 同步发布，使任何客户都能构建端到端加密推理管道

🔗 **樱桃的调度视角**: 这对 K8s 上 AI 推理的启发在于：当推理服务需要跨基础设施部署时，机密计算（Confidential Computing）正在从「H100 专属」走向更普及的 Blackwell GPU。如果 Master 的推理平台需要满足数据合规要求，TEE + 加密推理管道是值得关注的方向——尤其在多租户 K8s 集群中，机密计算可以隔离不同租户的推理数据。

### 3. 📦 Claude Code 砍掉 80% 系统提示词：Token 效率成 AI 工程新战场

**原文标题:** Claude Code 80%的提示词说删就删，Anthropic用Fable 5打了个样：AI行业的"降本"才刚刚开始
**来源:** InfoQ 中文 | 2026-07-03
**链接:** https://www.infoq.cn/article/GEkEm7rkUJfF8bdwTuBt

🎯 **一句话精髓**: Anthropic 删掉 Claude Code 80% 的系统提示词（从 65,000 token 降至极简），揭示了 AI 行业从「token maxing」炫耀转向「token apocalypse」成本治理的范式转变。

🔧 **技术栈:** Claude Code / Fable 5 / System Prompt 工程 / Caveman 插件 / Reasoning Trace 压缩 / Token 成本模型

📐 **架构/方法亮点**:
- **Prompt 债务概念**：System prompt 每次调用都被读入、计费、占上下文；Claude Code 一度膨胀到 65,000 token（关闭大部分功能后仍有 12,000 token），对比 Pi 启动上下文不到 1,000 token
- **Caveman 压缩模式**：开源插件通过去掉寒暄、模糊措辞、过渡语，减少 65%-75% 输出 token，不影响代码/命令/路径/URL 等精确内容；OpenAI 工程总监也贡献了代码
- **Reasoning Trace 压缩**：OpenAI 内部推理使用类工程速记语言（如 "Use core new nodes. Need infer. Try period."），压缩 reasoning token 的收益更大因为 agent 多步执行时前一步思考变成后一步输入

💡 **关键数据**: Anthropic 算力支出已达薪资支出的 2.3 倍（每位高级工程师 22.4 万美元年薪对应 51.5 万美元算力支出）；Uber 为每位工程师设每月 1,500 美元 token 上限；GPT-5.5 medium 仅用 2 万 token 达到惊人分数，而 Opus 4.8 用 5 万 token 得分反而更低

🔗 **樱桃的调度视角**: 这对 K8s 上部署 AI 推理服务的启示是——token 效率直接影响 GPU 利用率和成本。在多租户推理平台中，system prompt 的 token 占用会吃掉宝贵的上下文窗口和推理吞吐。Master 在设计推理服务的模型路由策略时，应将「token 效率」作为模型选型的关键维度，而非只看 benchmark 分数。

### 4. 📦 Cloudflare 修复 quiche 拥塞控制「死亡螺旋」：一行代码打破 CUBIC 恢复循环

**原文标题:** Cloudflare 如何解决了 quiche 中的一个拥塞漏洞
**来源:** InfoQ 中文 | 2026-07-01
**链接:** https://www.infoq.cn/article/CHON5sllKKAMBg309xiZ

🎯 **一句话精髓**: Cloudflare 发现其 QUIC 实现 quiche 的 CUBIC 拥塞控制算法在连接初始阶段高丢包时陷入「死亡螺旋」——拥塞窗口永远无法恢复，最终通过一行代码修复（从「上次发送时间」改为「最后 ACK 时间」计算空闲期）。

🔧 **技术栈:** QUIC / CUBIC 拥塞控制算法 / quiche (Rust) / Reno / 慢启动 / 拥塞规避 / 恢复状态机

📐 **架构/方法亮点**:
- **死亡螺旋根因**：在嘈杂的慢启动阶段，ACK 将传输中字节数降为零时触发空闲期优化；最小拥塞窗口为两个数据包时，空闲期优化变成「自我实现的预言」——将应用状态维持在恢复状态且恢复时间设在遥远未来
- **状态转换频率异常**：监控数据显示约 6.7 秒内发生 999 次状态转换（平均每 ~14ms 一次），与配置的 10ms RTT 过于接近，揭示异常循环
- **修复方案**：不再从上次发送数据时计算空闲时间，改为从收到最后一个 ACK 时计算——一行代码打破循环，测试通过率从 ~40% 恢复到 100%

💡 **关键数据**: 模拟测试中 60% 的用例未在 10 秒超时前完成（预期 4-5 秒）；用 Reno 替代 CUBIC 测试 100% 通过，成功将问题范围限定在 CUBIC；修复 commit 仅一行代码

🔗 **樱桃的调度视角**: 虽然这是网络协议层的调试，但其方法论对 K8s 可观测性有启发——当发现某项指标（如 Pod 间通信延迟）出现异常的周期性模式时，应像 Cloudflare 团队一样构建受控模拟环境，用监控数据定位状态机的异常转换。对于 K8s 上跑 gRPC/HTTP3 的推理服务，quiche 的这个修复直接影响长连接稳定性。

### 5. 📦 数据主权重塑云原生架构：K8s 作为主权控制平面 + OpenStack 底座

**原文标题:** How data sovereignty is changing cloud native infrastructure design
**来源:** CNCF Blog | 2026-07-03
**链接:** https://www.cncf.io/blog/2026/07/03/how-data-sovereignty-is-changing-cloud-native-infrastructure-design/

🎯 **一句话精髓**: 数据主权正在从「合规文档」变成「平台能力」——欧洲受监管企业正用 K8s + OpenStack + GitOps + Policy as Code 组合，通过架构而非合同来强制执行主权要求。

🔧 **技术栈:** Kubernetes / OpenStack (Ironic/Keystone/Neutron/Ceph) / GitOps / OPA/Gatekeeper / Kyverno / SBOM / Hardware Bill of Materials / 联邦学习

📐 **架构/方法亮点**:
- **K8s 作为主权控制平面**：Admission Controllers 在 Pod 调度前强制工作负载放置；Node affinity 确保工作负载只落在正确司法管辖区的 approved infrastructure；Policy as Code（OPA/Gatekeeper、Kyverno）将管辖权要求编码进集群
- **GitOps 多司法辖区管理**：Git 仓库包含共享配置和司法辖区特定 overlays，每个集群本地拉取并应用配置——无需中心化控制平面；每次变更可审查、版本控制、可审计
- **联邦学习主权架构**：主权 K8s 集群执行本地训练，只有聚合模型更新跨司法辖区移动——同样的策略、namespace 边界和治理控制成为分布式 AI 系统的基础

💡 **关键数据**: EU Cloud and AI Development Act (CADA) 于 2026 年 6 月提出四层主权框架；加拿大联邦政府已开始按数据驻留和司法管辖控制对云供应商评分

🔗 **樱桃的调度视角**: 对 Master 的启发是——K8s 的 Admission Controller + Node Affinity + Policy Engine 组合不仅是安全工具，更是「工作负载放置策略」的基础设施。在 GPU 集群中，同样的机制可以用于「数据驻留 GPU 节点」「合规推理只在特定区域节点执行」等场景。如果推理服务需要满足数据合规要求，Policy as Code 是必经之路。

🎯 **一句话精髓**: Argo CD v3.5 引入内部组件间 mTLS 加密、Git 源码签名验证（Source Integrity）和全新应用集 UI 预览功能，将 GitOps 供应链安全提升到新水平。

🔧 **技术栈:** Argo CD v3.5 / mTLS / GPG 签名 / Helm 4 / ApplicationSet / RBAC / Azure AD (Microsoft Graph API) / Tekton Chains

📐 **架构/方法亮点**:

- **Source Integrity 验证**：通过 `sourceIntegrity.required: true` 或 `argocd app set --source-integrity-required` 确保 Git 源在同步前具有有效签名

💡 **关键数据**: 对比 Flux v2.8（2026 年 3 月）通过 `spec.verify.mode: head` 原生支持 GPG 签名，Argo CD 在源码签名验证方面是在追赶 Flux；Rancher Fleet 使用 WebSocket 代理而非 gRPC 因此不需要内部 mTLS

🔗 **樱桃的调度视角**: 对于在 K8s 上通过 GitOps 管理 AI 推理服务配置的团队，Argo CD v3.5 的 Source Integrity 功能可以防止篡改后的模型配置被部署到生产。结合 SBOM 和镜像签名，可以构建从「代码提交→配置渲染→镜像构建→集群部署」的完整供应链验证链。

### 7. 📦 Kepler 重写：放弃 eBPF 回归 /proc，功耗监控精度大幅提升

**原文标题:** Kepler, re-architected: Improved power accuracy and a community call to action!
**来源:** CNCF Blog | 2026-06-30
**链接:** https://www.cncf.io/blog/2026/06/30/kepler-re-architected-improved-power-accuracy-and-a-community-call-to-action/

🎯 **一句话精髓**: CNCF 沙箱项目 Kepler 完成架构重写——放弃 eBPF（需 CAP_BPF/CAP_SYSADMIN 权限且精度不足），改用标准 /proc 和 /sys 读取，功耗归因精度从「多 kW 级尖峰」提升到「毫瓦级偏差」。

🔧 **技术栈:** Kepler / eBPF（已弃用） / /proc + /sys / RAPL / IPMI / Prometheus / Grafana / Helm / Kubernetes Operator

📐 **架构/方法亮点**:
- **eBPF→/proc 转型**：原架构依赖 eBPF 捕获利用率信号，造成大多数用户报告问题；eBPF 漏掉短生命周期进程导致功耗低估。新架构用只读 /proc 和 /sys（Linux 通用），显著降低权限要求和配置复杂度
- **动态功率结构发现**：旧版本假设硬编码功率结构（如 RAPL = core + DRAM），但实际硬件拓扑差异大导致归因到不存在的 ground truth；新版运行时动态发现主机功率计结构
- **精度验证**：CERN 测试中新 Kepler 的 `kepler_node_cpu_watts` 指标紧密切合 IPMI 负载曲线，消除了旧版本超出 IPMI 真值的 multi-kW 尖峰；Red Hat 测试显示进程级功耗归因偏差仅几毫瓦

💡 **关键数据**: 测试覆盖率提升至 90%；数据中心占 2024 年全球电力需求的 1.5%，IEA 预测到 2030 年将翻倍至约 945 TWh，AI 工作负载是主要驱动力

🔗 **樱桃的调度视角**: Kepler 重写对 GPU 集群的功耗感知调度意义重大。在 AI 推理场景中，GPU 功耗远高于 CPU，如果能精确归因到 Pod 级别，就可以实现「功耗感知调度」——在电力预算受限时优先调度低功耗推理任务。放弃 eBPF 降低权限门槛也意味着 Kepler 可以更容易部署在托管 K8s（如 EKS/GKE）上。

## 📓 樱桃的今日学习总结
