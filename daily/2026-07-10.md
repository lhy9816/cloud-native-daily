## 🚀 重点速览

### 1. 📦 SGLang 全栈优化 DeepSeek V4 百万上下文推理：ShadowRadix 统一三套 KV Cache

**原文标题:** 百万上下文下的 DeepSeek V4：SGLang 推理优化实战｜AICon深圳
**来源:** InfoQ 中文 | 2026-07-09
**链接:** https://www.infoq.cn/article/qALuq71AxiG5VLmWqSzU

🎯 **一句话精髓**: DeepSeek V4 采用 SWA + CSA + HCA 三种异构注意力混合架构并支持百万级上下文，SGLang 团队通过 ShadowRadix 虚拟地址表统一管理三套 KV Cache、FlashCompressor/Lightning TopK 算子融合、以及 DP/CP/EP 多维并行策略，在 GB300 NVL72 上实现单 GPU 11,500 token/s 吞吐。

🔧 **技术栈**: SGLang / DeepSeek V4 / ShadowRadix / RadixTree / SWA / CSA / HCA / FlashCompressor / Lightning TopK / MegaMoE / DeepGEMM / CUDA / Triton / TileLang / GB300 NVL72 / InferenceX

📐 **架构/方法亮点**:
- ShadowRadix 三套 KV Cache 统一管理：在 RadixTree 上扩展虚拟地址表，每个 token 分配唯一虚拟位置，通过除法映射到三组 Shadow 页表（SWA 原始位置 / CSA 除以 4 / HCA 除以 128）；SWA 引入 Tombstone 机制自动释放滑窗外过期 KV，额外维护两个 Ring Buffer 管理 Compressor 中间状态保证 CSA/HCA 跨 token 依赖正确性
- 算子级优化：FlashCompressor 将 Compressor 流程中 Softmax/bias/scale-dot 等零散操作融合为单一 kernel，HBM 读写从 5 次降至 2 次；Lightning TopK 序列分片到同一 Cluster 内多个 CTA，各 CTA 用直方图做本地统计通过 SM 互联完成 reduce，百万上下文 BS=1 下从 100μs 降至 15μs
- 并行策略选型决策树：超长输入（90 万+）CP（Context Parallelism）优于 TP（因压缩后 KV 体积小，AllGather 通信开销低）；短输入 TP 更优；高吞吐用 EP（Expert Parallelism）走 NVLink AlltoAll；SWA/CSA/HCA 三种 Attention 无数据依赖，分配到不同 CUDA Stream 用 CUDA Event 控制依赖顺序

💡 **关键数据**: InferenceX 基准（Input 8K / Output 1K）：开启 MTP 单用户交互速度 180 token/s；GB300 NVL72 高吞吐场景单 GPU 吞吐 11,500 token/s；Lightning TopK 百万上下文延迟从 100μs 降至 15μs；FlashCompressor HBM 读写从 5 次降至 2 次；实测 4K 与 900K 上下文解码速度差异极小（TP=8, BS=1, MTP Len=3）。

🔗 **樱桃的调度视角**: 这篇对 Master 的推理调度实践是直接参考--V4 的三套异构 Attention 意味着推理引擎的显存管理远比 V3 复杂，调度器不能简单按「GPU 数量」分配资源。ShadowRadix 的 KV Cache Offloading（CSA pool 溢出至 CPU 内存）和 PD 分离思路，意味着调度器需要感知「GPU 显存 + CPU 内存」两级存储和「Prefill/Decode」两阶段分离的拓扑约束，这对 K8s 的 DRA 和 Volcano gang 调度都提出了新要求喵。

### 2. 📦 K8s v1.35 DRA 正式 GA：GPU 调度从 Device Plugin 迈向声明式资源分配

**原文标题:** Understanding dynamic resource allocation in Kubernetes
**来源:** CNCF Blog | 2026-07-01
**链接:** https://www.cncf.io/blog/2026/07/01/understanding-dynamic-resource-allocation-in-kubernetes/

🎯 **一句话精髓**: Dynamic Resource Allocation（DRA）在 Kubernetes v1.35 达到 GA，NVIDIA 将 dra-driver-nvidia-gpu 迁入 Kubernetes SIGs 并去掉 Beta 标签--从此 GPU/MIG/VFIO 等设备的声明式分配有了原生 K8s 机制，告别 Device Plugin 的整数计数局限。

🔧 **技术栈**: DRA / Kubernetes v1.35 / NVIDIA GPU Operator v26.3.1 / NVIDIA DRA Driver GPU v25.12.0 / DeviceClass / ResourceSlice / MIG / VFIO / Cluster API / OpenStack / Ceph / Containerd 2.2.2

📐 **架构/方法亮点**:
- DeviceClass + ResourceSlice 设备模型：DeviceClass 表示设备类别（普通 GPU / MIG / VFIO），ResourceSlice 由 DRA 驱动在每个节点上自动更新，记录该节点上驱动管理的所有设备；同一节点上同一驱动管理的设备属于同一个 Pool，设备数超过单对象上限（128 条，或 64 条若使用 taints/counters）时自动跨多个 ResourceSlice 拆分
- NVIDIA DRA Driver GPU 声明式配置：通过 Helm values 控制 gpuResources / computeDomains / kubeletPlugin nodeSelector，支持 GPU Time Slicing（通过 TimeSlicingSettings Feature Gate 启用），实现对 GPU 计算资源的细粒度分配
- 实验环境全景：Equinix 东京数据中心，Cluster API + OpenStack 构建 K8s 集群，Ubuntu 24.04 + K8s v1.35.3 + Containerd 2.2.2，1 控制面 + 3 工作节点（含 T10×2 + A5000×1 GPU），NVIDIA GPU Operator v26.3.1 管理驱动和运行时

💡 **关键数据**: DRA 在 K8s v1.35 达到 GA；NVIDIA DRA Driver GPU 版本 v25.12.0；GPU Operator v26.3.1；ResourceSlice 单对象上限 128 条设备（使用 taints/counters 时为 64 条）；实验环境为 Equinix 东京数据中心的 CNTUG Infra Labs 社区实验室。

🔗 **樱桃的调度视角**: 这是 Master GPU 调度工作的里程碑事件！DRA GA 意味着 K8s 终于有了原生的、声明式的 GPU 分配机制，不再受限于 Device Plugin 的「整数 GPU」模型。DeviceClass 让调度器能感知 GPU 的 MIG 切片、VFIO 直通等细粒度资源，ResourceSlice 提供了节点设备的实时拓扑视图。对 Volcano/Kueue 来说，DRA 打开了「按需切片分配」和「拓扑感知调度」的新空间--比如把 A100 的 MIG 切片按推理负载的实际需求动态分配，而非整卡占位喵。

### 3. 📦 CNCF 发布云原生 AI 数据存储白皮书：从「存算鸿沟」到全生命周期数据策略

**原文标题:** The CNCF Data Storage in Cloud Native AI White Paper
**来源:** CNCF Blog (TAG Infrastructure) | 2026-07-08
**链接:** https://www.cncf.io/blog/2026/07/08/the-cncf-data-storage-in-cloud-native-ai-white-paper/

🎯 **一句话精髓**: CNCF TAG Infrastructure 发布《Data On Kubernetes – Data Analytics and AI/ML Workloads》白皮书，系统拆解 AI/ML 工作负载在云原生环境中的数据存储挑战--从百万小文件的元数据压力到存算分离的 GPU 利用率瓶颈，再到训练/推理/Agent 三阶段差异化的存储画像。

🔧 **技术栈**: Apache Parquet / Apache Iceberg / Milvus / Fluid / CSI / COSI / FUSE CSI / Apache Kafka / CDC / Vector Database / RAG / KV Caching / Prefix Caching

📐 **架构/方法亮点**:
- 三大核心挑战定义：Small-File Trap（百万级小文件压垮存储元数据服务器）、Decoupled Bottlenecks（存算分离可扩展但引入 API 调用开销和低 GPU 利用率）、Shifting Workload Profiles（训练要高吞吐持续数据流，推理要低延迟突发请求-响应）
- 四层技术支柱：Data Lake Houses & Vector Databases（Parquet/Iceberg 混合数据湖仓 + Milvus 向量库做 RAG 检索）、Caching & Data Locality（CNCF 项目 Fluid 编排 K8s 内分布式缓存消除数据传输延迟）、Standardized Interfaces（CSI 管块/文件存储、COSI 管对象存储、FUSE CSI 驱动）、Modern Data Pipelines（CDC + Kafka 从批处理走向实时流）
- AI 生命周期三阶段存储画像：训练阶段（长时吞吐导向，需容忍随机 shuffle + checkpoint 同步写突发）、推理阶段（延迟敏感 + KV Caching/Prefix Caching 消除冗余计算）、Agent 阶段（闭环迭代推理架构，需短期记忆/制品仓库/长期记忆三层存储）

💡 **关键数据**: 白皮书由 CNCF TAG Infrastructure 发布；数据中心 2024 年占全球电力需求 1.5%（引用 Kepler 项目数据）；Fluid 作为 CNCF 项目用于 K8s 内分布式缓存编排。

🔗 **樱桃的调度视角**: 这篇白皮书直接点出了 Master 做 GPU 调度时最容易忽视的瓶颈--不是 GPU 不够快，而是数据喂不进 GPU。「Decoupled Bottlenecks」说的就是存算分离后 GPU 利用率低，Fluid 的分布式缓存思路可以和 K8s 调度器的 data locality 策略结合，把数据缓存的亲和性和 Pod 调度的亲和性对齐，让 GPU 少等数据多算数喵。

### 4. 📦 Netflix 动态分区拆分：Cassandra 读取延迟从数秒降至毫秒级

**原文标题:** 动态分区拆分：Netflix 大幅削减 Cassandra 读取延迟至毫秒级
**来源:** InfoQ 中文 | 2026-07-09
**链接:** https://www.infoq.cn/article/GRDxYebxA7ywxO4seBWy

🎯 **一句话精髓**: Netflix 为时间序列抽象平台开发了 Cassandra 动态分区拆分机制，自动检测超大分区并异步拆分为子分区，读取延迟从数秒降至毫秒级，全程无需修改应用、停机或大规模重新分区。

🔧 **技术栈**: Apache Cassandra / 动态分区拆分 / 元数据层 / 时间序列抽象平台 / 灰度上线 / 数据校验管道

📐 **架构/方法亮点**:
- 元数据层路由：引入元数据层跟踪父分区与子分区的关系，读取时元数据服务确定哪些子分区包含请求数据并路由查询，多分区结果合并后返回客户端，底层存储变更对应用完全透明
- 不可变分区优先 + 回退安全：首个版本仅针对不可变分区拆分，数据迁移全程保留原始分区作为回退机制；引入验证管道比较原始分区返回结果与新读取路径生成结果，分阶段灰度上线
- 解决 Cassandra 时间序列工作负载核心痛点：持续增长的分区导致读取延迟增加、压缩开销激增、内存承压和负载分布不均

💡 **关键数据**: 受影响分区平均读取延迟从数秒降至低于两位数毫秒级；尾部延迟从数秒缩短至 200ms 及以下；此前分区大小超过 500MB 就会出现服务可用性故障，部署后该类分区可正常分页查询且全程稳定运行；Cassandra 集群读取超时次数和 CPU 利用率均下降。

🔗 **樱桃的调度视角**: Netflix 的「元数据层透明路由」思路和 K8s 调度器的抽象层设计异曲同工--都是通过引入间接层让上层消费者无需感知底层拓扑变化。对 Master 来说，当推理服务的模型分片（tensor parallelism）动态迁移时，同样需要一个元数据路由层让请求路由对客户端透明，Netflix 的灰度验证 + 回退策略值得借鉴喵。

### 6. 📦 SRE 的四体问题：自主运维的瓶颈不在模型而在上下文

**原文标题:** The 4-body problem of SRE: Why autonomous operations depend on context
**来源:** CNCF Blog | 2026-07-06
**链接:** https://www.cncf.io/blog/2026/07/06/the-4-body-problem-of-sre-why-autonomous-operations-depend-on-context/

🎯 **一句话精髓**: 作者在班加罗尔与资深 SRE 交流后提出「SRE 四体问题」框架：每个运维决策都需要同时推理代码、基础设施状态、运行时信号和运维知识四个紧密耦合的「真相体」--AI SRE 的瓶颈不是模型能力，而是跨四体的上下文关联能力。

🔧 **技术栈**: SRE 四体问题 / Autonomous Operations / AI SRE / RCA / SLO / Runbook / Observability / Terraform / Git

📐 **架构/方法亮点**:
- 四体定义：Code（提交/PR/构建制品/版本/配置变更）、Infrastructure State（云账户/网络/K8s 集群/队列/数据库/IAM 的实际状态 vs Terraform 声明）、Runtime Signals（指标/日志/追踪/事件/错误预算/SLO）、Operational Knowledge（部落知识/事后复盘/架构决策记录/on-call playbook/Slack 线程）
- 「People Putty」失败模式：用少数人脑中的部落知识和陈旧 runbook 填补四体之间的缝隙，在负载压力下崩溃；Agent 常见失败模式包括「自信地修错东西」「演示完美但真实事故中崩溃」「事后无法重建推理过程」
- 核心结论：substrate 先于 agent--不能把 Agent 放在四个孤立互疑的系统上就指望可靠性，Agent 的能力上限取决于它能推理的上下文质量

💡 **关键数据**: 作者与班加罗尔一屋子资深 SRE/平台工程师/工程领导者交流；四体框架类比物理学三体问题，加入第四个变量使动力学质变而非渐进恶化。

🔗 **樱桃的调度视角**: 「substrate 先于 agent」这个结论对 Master 设计 AI 推理平台同样成立--不能把推理调度 Agent 放在割裂的调度器/监控/预算系统上就期望它自主决策。K8s 的 DRA + ResourceSlice 提供了设备状态真相体，Prometheus 提供运行时信号体，但「运维知识体」（为什么这台 GPU 节点总是 OOM、为什么这个模型在 MIG 切片下延迟突增）仍然活在工程师脑子里，这是自主调度最难补齐的一体喵。

## 📓 樱桃的今日学习总结

1. **GPU 调度从「整卡分配」迈向「声明式细粒度分配」** - DRA 在 K8s v1.35 GA 是分水岭：DeviceClass + ResourceSlice 让调度器能感知 MIG 切片、VFIO 直通、Time Slicing 等细粒度设备资源，Volcano/Kueue 的调度策略可以基于 DRA 做更精细的 bin-packing。结合 SGLang 的 ShadowRadix KV Cache 管理（GPU 显存 + CPU 内存两级存储），未来的推理调度需要同时感知「设备拓扑」和「显存层级」两个维度。

2. **AI 推理优化的「复杂度换性能」范式已成主流** - SGLang 为 DeepSeek V4 的三套异构 Attention 各维护独立页表/淘汰策略/前缀匹配逻辑，系统复杂度相比 V3 大幅膨胀，但换来的是百万上下文 180 token/s 和单 GPU 11,500 token/s 吞吐。这揭示了一个趋势：模型架构越复杂，推理引擎的工程化能力（而非纯算法创新）越成为竞争力核心，这对 Master 选择推理框架（SGLang vs vLLM vs TensorRT-LLM）时的评估维度有直接影响。

3. **平台工程从「服务开发者」到「服务多角色 + AI Agent」** - Platform Engineering 2.0 把 AI Agent 列为非人类平台消费者，Embedded FinOps 把成本从事后报表变为调配时决策。这意味着 K8s 平台不仅要管 Pod 调度，还要管模型生命周期、MCP 网关和 Agent 治理--调度器需要从「容器编排器」进化为「AI 工作负载编排器」，成本感知和治理能力成为一等公民。
