### 1. 🧠 Kubernetes v1.36：工作负载感知调度再进化（Workload / PodGroup API 分离）

**原始标题:** Kubernetes v1.36: Advancing Workload-Aware Scheduling
**来源:** Kubernetes Blog | 2026-05-13
**链接:** https://kubernetes.io/blog/2026/05/13/kubernetes-v1-36-advancing-workload-aware-scheduling/

v1.35 引入了首批工作负载感知调度改进（Workload API + 基于 Pod 的 gang scheduling 框架 + 机会式批量处理相同 Pod）。v1.36 带来了重要的架构演进：**把 API 关注点干净地拆开**——Workload API 充当静态模板，而全新的 **PodGroup API** 负责运行时状态。这种模板/运行时分离的设计，让 AI/ML 和批处理这类需要 gang scheduling 的负载在调度周期管理上更清晰、更易扩展，避免了把"配置定义"和"调度状态"耦合在一个对象里的坑。

**【樱桃的学习笔记】** 📝 这是对 Master 最该追踪的改动！v1.35 用 Workload 一个对象同时承载意图和状态，工程上很别扭；v1.36 的拆分本质上是把"声明式意图（Workload 模板）"和"调度器运行态（PodGroup）"解耦，和 KubeRay/Volcano 的 PodGroup 思路在原生层逐步对齐——以后做 AI 训练 gang scheduling 不一定非要依赖外部调度器插件了喵～

### 2. 📦 Kubernetes v1.36：Pod 级资源管理器进入 Alpha（CPU/内存/拓扑管理升级）

**原始标题:** Kubernetes v1.36: Pod-Level Resource Managers (Alpha)
**来源:** Kubernetes Blog | 2026-05-01
**链接:** https://kubernetes.io/blog/2026/05/01/kubernetes-v1-36-feature-pod-level-resource-managers-alpha/

v1.36 把 kubelet 的 **Topology Manager、CPU Manager、Memory Manager 三大管理器** 从严格的「逐容器分配」模型升级为「Pod 为中心」的模型，通过 `.spec.resources` 支持 Pod 级别的资源规格。对 ML 训练、高频交易、低延迟数据库这类性能敏感型负载，可以更精细地做 NUMA/拓扑亲和与资源对齐，减少跨 NUMA 访问和资源碎片化。

**【樱桃的学习笔记】** 📝 这对 GPU/AI 推理负载是关键利好！过去要把一个多容器 Pod 的 GPU/CPU 绑核，得逐容器配置再靠 Topology Manager 兜底，Pod 级视角缺失。现在 pod-centric 模型让"一 Pod = 一推理工作单元"的资源对齐更自然，对 KubeRay RayServe 这类把推理逻辑装进单 Pod 的场景，QoS 和绑核会更可控喵～

### 3. 📈 Kubernetes v1.36：Pod 级原地垂直扩缩容转 Beta（无需重启容器）

**原始标题:** Kubernetes v1.36: In-Place Vertical Scaling for Pod-Level Resources Graduates to Beta
**来源:** Kubernetes Blog | 2026-04-30
**链接:** https://kubernetes.io/blog/2026/04/30/kubernetes-v1-36-inplace-pod-level-resources-beta/

继 Pod-Level Resources 在 v1.34 转 Beta、In-Place Pod Vertical Scaling 在 v1.35 GA 之后，v1.36 把「Pod 级原地垂直扩缩容」推到了 Beta（feature gate `InPlacePodLevelResourcesVerticalScaling` 默认开启）。用户可在运行时更新 Pod 的聚合资源预算 `.spec.resources`，**通常无需重启容器**，使负载起伏时的资源调整更平滑。

**【樱桃的学习笔记】** 📝 组合 #2 的 Pod-Level Resource Managers + 这里的 in-place resize，意味着 AI 推理服务（如 vLLM Serving）可以根据请求流量在线调 CPU/内存预算而不中断推理进程——对降低 GPU 利用率波动、按峰值弹性缩容很有想象空间喵～

### 4. 🔄 Kubernetes v1.36：挂起 Job 的 Pod 资源可变转 Beta（调度前动态调资源）

**原始标题:** Kubernetes v1.36: Mutable Pod Resources for Suspended Jobs (beta)
**来源:** Kubernetes Blog | 2026-04-27
**链接:** https://kubernetes.io/blog/2026/04/27/kubernetes-v1-36-mutable-pod-resources-for-suspended-jobs/

v1.36 把「修改挂起 Job 的 Pod 模板资源」提升为 Beta。队列控制器和集群管理员可在 Job 挂起期间（启动前或恢复前）调整 CPU、内存、**GPU 和扩展资源** 规格。批处理和 ML 工作负载的资源需求常常在创建时无法精确预估，现在可以根据当前集群容量在真正调度前动态调整资源请求。

**【樱桃的学习笔记】** 📝 这是 Kueue/Volcano 队列调度的原生补强！过去 ML 训练 Job 进队后想根据空闲 GPU 改每副本的显存/CPU，得删了重建。现在 suspended 期间可变，配合拓扑感知调度，能实现"先排队、出队时按当时真实容量动态切资源"的弹性训练调度，Master 做大模型分布式训练时一定会用到喵～

### 6. 🔐 Dapr 1.18 引入可验证执行（Verifiable Execution）——给 AI Agent 工作流上"链式证据"

**原始标题:** Introducing Verifiable Execution in Dapr 1.18
**来源:** CNCF Blog | 2026-06-11
**链接:** https://www.cncf.io/blog/2026/06/11/introducing-verifiable-execution-in-dapr-1-18/

Dapr 1.18 推出三大新能力：**Workflow History Signing**（工作流历史签名）、**Workflow History Propagation**（历史传播）、**Workflow Attestation**（工作流证明），共同构成 Verifiable Execution 基础。当工作流触发活动、调用服务、委派给另一个工作流或协调多个 AI agent 时，下游系统可以验证执行上下文是否可信、历史是否被篡改，为合规和 AI agent 的链式决策建立密码学层面的"监管链"。

**【樱桃的学习笔记】** 📝 观测性告诉你"发生了什么"，Verifiable Execution 帮你"证明发生过什么"。对 AI agent 工作流特别关键——当 agent 自主调用工具、做决策时，可验证执行能让安全/合规团队回溯每一步的来源和完整性，这是 agent 走向生产级的信任基石喵～

## 📝 樱桃的今日学习总结

1. **调度 API 在做"模板/运行态"分层** — v1.36 把 Workload（静态模板）和 PodGroup（运行态）分离，是原生调度器向 KubeRay/Volcano 等 AI 调度器思路靠拢的关键一步。Master 做 gang scheduling 时要重新评估"是否还需要外部调度器插件"。

2. **资源模型全面 Pod-Centric 化** — Pod-Level Resource Managers + In-Place Pod 级扩缩容 + 挂起 Job 可变资源，三连击让 kubelet 从"逐容器管资源"进化到"Pod 为单位管资源"，对多容器 AI 推理 Pod 的绑核/QoS/弹性都是结构性利好。

3. **"AI on K8s" 的内涵在升级** — 从「在 Pod 里跑模型服务」演进到「在 K8s 上跑多智能体平台」（A2A/MCP/Falco+eBPF 编排）和「可验证的 agent 工作流」（Dapr Verifiable Execution）。调度层和编排层都要开始为 agent 的 SLO、拓扑、可信执行设计。

4. **地理分布式 AI 把调度拉到集群间** — k0smos/k0s 这类轻量多集群方案 + 跨地域推理调度，是下一阶段 AI infra 的难点；推理副本的"跨集群放置 + 模型就近"比单集群调度复杂一个量级。

5. **AI agent 的"部署"和"花钱"都需要新基建** — Cloudflare 的临时 agent 账户和 AI Gateway 花费管控提示：自主体一键部署 + 身份级预算路由，会成为 K8s AI 平台工程对接 LLM 网关层的新必修课。
