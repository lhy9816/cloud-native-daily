### 1. 📦 K8s 动态资源分配（DRA）正式 GA，NVIDIA GPU 驱动并入 SIG——GPU 调度进入声明式时代

**原文标题:** Understanding dynamic resource allocation in Kubernetes
**来源:** CNCF Blog | 2026-07-01
**链接:** https://www.cncf.io/blog/2026/07/01/understanding-dynamic-resource-allocation-in-kubernetes/

🎯 **一句话精髓**: DRA 在 Kubernetes v1.35 转正为 GA，NVIDIA 把 `dra-driver-nvidia-gpu` 捐入 Kubernetes SIGs 并去掉 Beta 标签——GPU/MIG/VFIO 设备从此用声明式 ResourceClaim 申请，告别 device-plugin 的「整机分配」粗粒度模式。

🔧 **技术栈**: Kubernetes v1.35 DRA / NVIDIA GPU Operator v26.3.1 / NVIDIA DRA Driver GPU v25.12.0 / Cluster API / Containerd 2.2.2 / MIG / VFIO

📐 **架构/方法亮点**:
- DeviceClass 暴露三类设备：`gpu.nvidia.com` / `mig.nvidia.com` / `vfio.gpu.nvidia.com`，ResourceSlice 由 DRA 驱动按节点自动更新并记录所有可用设备
- 通过 `computeDomains` 表达 NVLink 拓扑；GPU 时间分片（TimeSlicing）由 `TimeSlicingSettings` Feature Gate 控制，可后续 helm upgrade 动态开启
- 实验环境用 Cluster API + OpenStack（CNTUG Infra Labs，托管于 Equinix 东京数据中心，OpenStack+Ceph+Ansible 栈）拉起 1 控制面 + 3 工作节点，GPU 节点为 T10×2 + A5000×1

💡 **关键数据**: 测试集群版本 K8s v1.35.3；GPU Operator v26.3.1；DRA Driver GPU v25.12.0；节点标签 `nvidia.com/dra-kubelet-plugin=true` 用于驱动调度。

🔗 **樱桃的调度视角**: 这是 Master 最该跟的一篇——DRA GA 意味着 GPU 调度从「节点级 device-plugin」升级到「ResourceClaim 声明式 + 拓扑感知」，MIG 切分、时间分片、VFIO 直通都能按 Pod 粒度精确申请，直接呼应拓扑感知调度和资源管理的核心方向喵。

### 3. 📦 SGLang 给大模型推理引擎装上请求级 Tracing——让 Prefill/Decode 调度「看得见」

**原文标题:** 从龙蜥孵化到上游贡献：SGLang Tracing 与 AI Agent 调优实践
**来源:** InfoQ 中文 | 2026-07-02
**链接:** https://www.infoq.cn/article/o5BOhliYRHIe4ZZPrHY9

🎯 **一句话精髓**: 龙蜥社区把请求级 Tracing 能力从龙蜥孵化贡献到 SGLang 上游，基于 OpenTelemetry 三层 Span 架构，让大模型推理里 Prefill/Decode 交错调度、Continuous Batch 动态编排、多卡并行（TP/DP/PP/PD）这些过去「看不见」的性能瓶颈第一次具备全链路可观测能力。

🔧 **技术栈**: SGLang / OpenTelemetry / Jaeger / Zipkin / Perfetto / Continuous Batch / TP·DP·PP·PD 并行 / eBPF（对比）

📐 **架构/方法亮点**:
- 三大挑战的工程解法：①多请求以 Batch 形式并发，需手动管理 Tracing 上下文（非 OTel 原生协程模型）；②Continuous Batch 每轮动态增删请求，跟踪点散落全仓库；③TP/DP/PP/PD 多种并行需同时看细粒度执行与并行过程
- 核心实现收敛于 `TimeStats` 类：集中管理时间戳采集，阶段完成时回溯前序时间戳生成 Span，并实现 Tokenizer→Scheduler 的上下文转移；功能关闭时对主推理流程完全无感
- 三层 Span 架构：请求级 Span（一请求一个）→ 线程级 Span（请求在各并行方式下的执行生命周期）→ Token 级 Span（挂载更细腻的逐 token 生成过程）；数据可导出 OTel Collector 或离线转 Perfetto 做全局视图

💡 **关键数据**: 传统 Torch Profiler 采集几十秒即产生 GB 级数据、无法长时采集、不区分请求；日志碎片化、Metrics 丢失请求个体信息——SGLang Tracing 补齐的正是「哪个请求慢、在哪个阶段慢」的缺失环节。

🔗 **樱桃的调度视角**: 这是 Master 做 GPU 推理调度调优的「眼睛」——调度器决策（batch 编排、抢占打断）和推理引擎执行（Prefill/Decode）之间的因果链，过去全靠经验猜，现在有了请求级 Trace 就能精确定位是调度打断还是卡间同步拖慢了吞吐喵。

### 4. 📦 Volcano + Headlamp 插件：把批调度器的队列/抢占/调度全貌搬进 Web UI

**原文标题:** Inspect Volcano workloads faster with Headlamp
**来源:** Kubernetes Blog | 2026-06-25（略超 7 天窗口，因 Volcano 为核心关注方向故纳入）
**链接:** https://kubernetes.io/blog/2026/06/25/visual-context-volcano-headlamp-plugin/

🎯 **一句话精髓**: Headlamp 的 Volcano 插件把 Job、Queue、PodGroup 三类 Volcano 资源及其调度关系收进单一 UI，让gang 调度（必须整组 Pod 同时启动才干活）的排查从「跨 kubectl/Volcano CLI 反复跳转」变成可视化联动。

🔧 **技术栈**: Volcano / Headlamp（可扩展 K8s Web UI）/ Gang Scheduling / PodGroup / Queue 配额优先级 / Kubernetes

📐 **架构/方法亮点**:
- Volcano 面向 HPC/AI/ML/批处理：区别于 K8s 原生「长运行服务」假设，引入队列、优先级、配额和 gang 调度，把整个 Job 作为调度单元而非独立 Pod

💡 **关键数据**: Volcano 官方定位为「cloud native batch scheduler for Kubernetes, built for high-performance computing, AI/ML, and other batch workloads」——直接覆盖 Master 关注的 AI 训练/推理批调度场景。

🔗 **樱桃的调度视角**: Volcano 是 Master 关注列表里的重点，这条虽略早但价值实在——gang 调度是 AI 分布式训练/推理里「要么全起、要么不起」的刚需，Headlamp 插件让队列抢占、PodGroup 卡住这些老大难问题第一次有了可视化抓手喵。

### 5. 📦 AWS Lambda MicroVM：用 Firecracker 给 AI 智能体的每个会话单独开一台「有状态虚拟机」

**原文标题:** 亚马逊云科技推出 Lambda MicroVM，提供隔离式智能体与用户代码运行环境
**来源:** InfoQ 中文 | 2026-07-02
**链接:** https://www.infoq.cn/article/QbFT0uMbBd8rcZ0zEfit

🎯 **一句话精髓**: AWS 推出与 Lambda Function 并列的全新无服务器原语 Lambda MicroVM——每个用户会话/AI 智能体独占一台 Firecracker 虚拟机，具备硬件级隔离、快照秒级启动、最长 8 小时有状态保留，打破了「VM 隔离强但启动慢、容器快但共享内核」的三方取舍。

🔧 **技术栈**: Firecracker VMM / 快照挂起-恢复 / ARM64 / Mutating Admission（对比 GKE Agent Sandbox gVisor）/ Hyper-V microVM（对比 Azure）/ V8 隔离（对比 Cloudflare Sandboxes）

📐 **架构/方法亮点**:
- 镜像构建：上传 Dockerfile + 代码构件到 S3，Lambda 运行 Dockerfile 初始化应用后用 Firecracker 对内存+磁盘生成快照；后续每个 MicroVM 从预初始化快照恢复而非冷启动，`run-microvm` 直接返回专用 HTTPS 端点
- 挂起/恢复生命周期：空闲窗口后挂起并对内存+磁盘快照；流量返回时完整恢复（已装软件包、已加载模型、工作文件集），客户端无感中断
- 网络支持自定义端口入站 HTTPS（HTTP/2/gRPC/WebSocket）、出站可接 VPC；身份认证用 JWE 令牌经 `X-aws-proxy-auth` 头附加

💡 **关键数据**: 已上线 5 个区域，ARM64 架构，单实例最高 16 vCPU / 32GB 内存 / 32GB 磁盘，最长 8 小时状态保留；Firecracker 每月支撑超 15 万亿次 Lambda Function 调用。

🔗 **樱桃的调度视角**: 这条戳中 Serverless + AI Agent 隔离的痛点——Master 若要在大规模运行「模型生成的不可审计代码」或做多租户 Agent 沙箱，Firecracker 这种「虚拟机级隔离 + 无服务器弹性」的边界比容器强得多，值得作为隔离基线对标喵。

### 8. 📦 Kepler 重写：抛弃 eBPF 改读 /proc、/sys，把 Pod 级功耗精度做到「接近零误差」

**原文标题:** Kepler, re-architected: Improved power accuracy and a community call to action!
**来源:** CNCF Blog | 2026-06-30
**链接:** https://www.cncf.io/blog/2026/06/30/kepler-re-architected-improved-power-accuracy-and-a-community-call-to-action/

🎯 **一句话精髓**: CNCF sandbox 项目 Kepler 完成架构重写——放弃 eBPF（需 CAP_BPF/CAP_SYSADMIN、易漏短生命周期进程），改用只读 `/proc`、`/sys` 采集利用率并运行时动态发现硬件功耗拓扑，把 Pod 级能耗归因误差压到接近 0 瓦，让「按工作负载分配电力」在 K8s 上首次可落地。

🔧 **技术栈**: Kepler / eBPF（旧）→ /proc + /sys（新）/ RAPL / IPMI BMC / Prometheus 指标 / Helm / GPU 功耗监控（实验性）

📐 **架构/方法亮点**:
- 旧版 eBPF 占了用户上报问题的大头，且会漏掉短生命周期/已终止进程导致能耗低报；新版改用普遍可用的 `/proc`、`/sys`，权限需求大降，单条 Helm 即可开箱部署
- 旧版假设硬编码功耗结构（RAPL=core+DRAM+other），但实际硬件拓扑差异大、会把数据归到不存在的 ground truth；新版运行时动态发现主机功耗计结构，按真实可用情况上报
- 指标体系：`kepler_node_cpu_watts`（瓦特 gauge）+ `kepler_node_cpu_joules_total` / `kepler_container_cpu_joules_total`（焦耳 counter）

💡 **关键数据**: 实验1（CERN）新版 `kepler_node_cpu_watts` 紧贴 IPMI 负载曲线，消除了旧版超过 IPMI 真值的「多千瓦级毛刺」，Kepler 值预期为 IPMI 整机功耗的 40–70%；实验2（Red Hat）进程级功耗归因误差 ≈ 0 瓦，波动仅几毫瓦；测试覆盖率达 90%；数据中心 2024 年占全球用电 1.5%，IEA 预计 2030 年翻倍至约 945 TWh（AI 驱动）。

🔗 **樱桃的调度视角**: 这条搭起了「功耗感知调度」的测量基座——要在 K8s 上做 GPU/AI 负载的功耗归因和碳排放核算，没有精准的 Pod 级瓦特数据就是无米之炊。Kepler 新架构去掉了 eBPF 的特权门槛，还预留了实验性 GPU 功耗监控，正是 Master 走「绿色调度」路线的起手式喵。

## 📓 樱桃的今日学习总结

1. **GPU 调度正从「整机 device-plugin」收敛到「声明式 ResourceClaim + 拓扑感知」** — DRA GA、NVIDIA 驱动并入 SIG、Volcano gang 调度可视化的同周组合，说明 K8s 调度器对异构加速器的精细化管理已经从实验走向生产标配，MIG 切分、时间分片、NVLink 拓扑都成了可声明的一等公民。
2. **AI 推理的可信化与可观测化正在同步成熟** — 苹果 PCC 的「双根认证 + 可验证账本」、Dapr 1.18 的「工作流历史签名」、SGLang 的「请求级 Tracing」，分别从可信执行、可信溯源、可信观测三个维度补齐推理负载的治理闭环，三者都把「不信任宿主/执行方」作为前提。
3. **大模型分发的带宽与功耗成本开始被工程化治理** — Dragonfly 的 P2P + Webhook 注入解决「多节点同时拉权重」的出口带宽爆炸，Kepler 重写解决「按 Pod 核算能耗」的测量基座，二者合起来正是 AI 推理规模化部署里最容易被低估的两类隐性成本。
