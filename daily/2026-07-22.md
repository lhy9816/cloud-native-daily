## 🚀 重点速览

### 1. 📦 HAMi 升级为 CNCF 孵化项目：K8s 上的 GPU 虚拟化调度中间件标准化

**原文标题:** HAMi becomes a CNCF incubating project
**来源:** CNCF Blog | 2026-07-16
**链接:** https://www.cncf.io/blog/2026/07/15/hami-becomes-a-cncf-incubating-project/

🎯 **一句话精髓**: CNCF TOC 投票接纳 HAMi 为孵化项目——它是个开源云原生 GPU 虚拟化中间件，能在 K8s 上把物理 GPU/NPU/DCU/MLU 按显存、算力核或设备数切分，硬隔离 + binpack/spread/拓扑感知调度，且不动应用代码和现有资源清单。

🔧 **技术栈**: Kubernetes / Mutating Webhook / Scheduler Extender / Device Plugin / HAMi-Core / Volcano / Koordinator / Kueue / KAI-scheduler / llm-d / Prometheus / Grafana

📐 **架构/方法亮点**:
- **五大组件分工**: Mutating Webhook（拦截 Pod 提交、重写调度字段与资源请求）/ Scheduler Extender（binpack/spread/拓扑感知过滤-评分-绑定）/ Device Plugins（厂商特定，注册加速器并分配分数设备资源）/ HAMi-Core（容器内虚拟化层，硬限制 GPU 显存与算力，拦截 NVIDIA 原生 CUDA 驱动）/ HAMi-WebUI + Prometheus 可观测层
- **多厂商单一接口**: 区别于绑定单一厂商生态的 device-plugin 工具，HAMi 用一致接口覆盖 GPU/NPU/DCU/MLU 等异构加速器，切分维度 = 显存 / 算力核 / 设备数，硬运行时隔离
- **生态打通 + 路线图**: 已与 Volcano（批式 AI 调度）、Koordinator（GPU 共享）集成并兼容默认 K8s 调度器路径；路线图含 gang-scheduling、preemption、autoscaling、DRA 消费监控，并要与 KAI-scheduler/Kueue/llm-d/Volcano 深度协作

💡 **关键数据**: 550+ 贡献组织、5 份独立 CNCF 案例研究；DaoCloud 在 10+ 数据中心跨 **1 万+ GPU** 部署；招商银行大规模管理异构加速器；约 3500 GitHub stars、550+ forks；**2687 contributors（YoY +43%）**；16 个 release，稳定版 v2.9.0；2024-08-21 进入 Sandbox

🔗 **樱桃的调度视角**: 这是对 Master 最直接相关的一条——HAMi 解决「整卡分给只用零头的任务」的碎片化浪费，正是 GPU 调度核心痛点。Scheduler Extender + binpack/spread/拓扑感知 + gang-scheduling 路线图，与 Volcano/Kueue 生态打通，意味着 K8s 上的 GPU 共享调度栈正在标准化。Master 做推理调度可把 HAMi 当「设备级切分 + 调度扩展」的底层基础设施。

### 2. 📦 AI Agent 该不该用 Pod 做部署单元？kagent 引入 agent-substrate 控制面

**原文标题:** Is a Pod the right deployment unit for an AI agent?
**来源:** CNCF Blog | 2026-07-14
**链接:** https://www.cncf.io/blog/2026/07/14/is-a-pod-the-right-deployment-unit-for-an-ai-agent/

🎯 **一句话精髓**: Solo.io 的 kagent 团队反思——每个 Agent 独占 Pod/Service/ServiceAccount 虽解决了隔离/身份/可观测，但 Agent 是短生命周期、突发、可派生子 Agent 的，为每个潜在 Agent 常驻 Pod 太浪费；于是引入 agent-substrate，在 K8s 之上加一层控制平面管理 AI actor 的生命周期与放置。

🔧 **技术栈**: kagent / Kubernetes Pod·Service·ServiceAccount / agent-substrate / WorkerPool / ActorTemplate / gVisor (runsc) / ate.dev/v1alpha1 CRD / 快照恢复

📐 **架构/方法亮点**:
- **Pod 方案的收益与痛点**: Pod 给进程/容器隔离、ServiceAccount 给身份、现有 NetworkPolicy/准入策略/可观测全复用；但 Agent 非持续可用（被分配任务才唤醒、执行几秒几分钟就空闲），常驻 Pod 浪费；Agent 还会动态派生子 Agent 并行、模拟用户、暂停等人审——生命周期以秒/分钟计而非天
- **agent-substrate 抽象**: 在 K8s 之上加控制平面，K8s 继续管 Pod/Service/网络/存储/计算，substrate 管 AI actor 生命周期与放置到 worker；**WorkerPool≈NodePool、Worker≈Node、ActorTemplate≈PodTemplate**（声明式 spec）
- **gVisor 沙箱 + 快照**: ActorTemplate 内含 runsc 配置作为 gVisor 执行入口，配 pause image + snapshotsConfig（快照位置）实现 actor 快启/恢复

💡 **关键数据**: 默认 WorkerPool replicas: 3；ateom-gvisor:v0.0.6；gVisor nightly 2026-06-02 runsc（文章为概念/架构反思，无性能基准数字）

🔗 **樱桃的调度视角**: 这条直击 Master 的调度神经——「Pod 是否是 Agent 的正确生命周期抽象」是个真问题。短生命周期 + 突发 + 可派生子 Agent 的工作负载更像 Volcano/Kueue 的批式 job 或 serverless，而非 Deployment。agent-substrate 把「actor 调度」从「pod 调度」解耦，WorkerPool/ActorTemplate 模式值得 Master 借鉴设计 Agent 调度层。

### 3. 📦 K8s 上自建 vLLM 推理栈：PVC 缓存模型权重 + DRBD 复制存储

**原文标题:** Running a self-hosted LLM in Kubernetes with vLLM
**来源:** CNCF Blog | 2026-07-16
**链接:** https://www.cncf.io/blog/2026/07/16/running-a-self-hosted-llm-in-kubernetes-with-vllm/

🎯 **一句话精髓**: 手把手在 K8s 实验环境用 vLLM + LINSTOR（DRBD 复制块存储）搭自托管 LLM 推理栈——PVC 缓存模型权重只下载一次、Secret 管 HF token、Deployment+Service 暴露 OpenAI 兼容 API，让「自托管高频/敏感负载 + 托管 API」的混合架构落地。

🔧 **技术栈**: vLLM / Kubernetes / LINSTOR (DRBD) / CSI / Piraeus Operator / PersistentVolumeClaim / Hugging Face / Llama-3.2-1B-Instruct / OpenAI 兼容 REST API

📐 **架构/方法亮点**:
- **三资源部署**: PVC（模型存储，linstor-csi-lvm-thin-r2，50Gi，两副本冗余）+ Secret（HF token）+ Deployment/Service（推理服务）
- **模型只下一次**: mountPath=`/root/.cache/huggingface` 由 PV 支撑，模型权重下载一次后 Pod 重启跳过下载；DRBD 复制块存储保证大模型权重在节点故障下存活
- **OpenAI 兼容性**: vLLM 暴露 OpenAI 兼容 REST API，已对接 OpenAI SDK 的代码（LangChain/LlamaIndex）只需改 URL 即可指向自托管实例——这是混合架构可行的关键

💡 **关键数据**: 模型 meta-llama/Llama-3.2-1B-Instruct（1B 参数，轻量到可在 CPU 跑，适合无 GPU 的实验环境）；PVC 50Gi；linstor-csi-lvm-thin-r2 两副本（thin-provisioned LVM + DRBD 复制）

🔗 **樱桃的调度视角**: 对 Master 这是「推理上 K8s」的最小可行栈——PVC 缓存权重解决冷启动重复下载、CSI 复制存储保证权重高可用。生产规模还要叠加 HAMi（GPU 切分）+ Volcano/Kueue（排队/批调度）+ goodput 感知 HPA（昨天那条），这套基础件是拼图的地基。

### 4. 📦 Agentic Enterprise 时代的平台工程：Agent 成为平台的一等消费者

**原文标题:** Platform engineering for the agentic enterprise: Managing applications, resources, and AI agents
**来源:** CNCF Blog | 2026-07-21
**链接:** https://www.cncf.io/blog/2026/07/21/platform-engineering-for-the-agentic-enterprise-managing-applications-resources-and-ai-agents/

🎯 **一句话精髓**: 平台工程正从「服务人类开发者」演进到「同时服务人和 AI Agent」——IDP 不再只管应用，还要把 resources（数据库/消息/对象存储/AI 模型/密钥/K8s 集群）和 AI agents 都作为一等平台对象，统一身份、权限、审计与治理。

🔧 **技术栈**: Internal Developer Platform (IDP) / Kubernetes / GitOps / MCP server / RBAC / 策略引擎 / 可观测性 / 审计链路

📐 **架构/方法亮点**:
- **三类一等对象**: Applications（业务逻辑/API/事件）、Resources（数据库/消息/对象存储/AI 模型/API/身份/密钥/K8s 集群/SaaS——各自有生命周期/所有权/治理/安全策略）、AI Agents（推理上下文、调用工具、与人协作、自动化运维的新一类软件 actor）
- **两类一等消费者**: 人类（portal/CLI/GitOps）和 AI Agent（MCP server），各自独立身份、scoped 权限、审计链路，但受一致安全控制/策略边界/运维护栏约束
- **平台职责扩展**: 不只是开发平台，而是开发者/平台工程师/SRE/AI Agent 共同构建/运维/治理企业软件的运营底座——平台不能再假设每次交互都来自人

💡 **关键数据**: 文章为概念框架（无具体性能数字），核心论点是「Agentic Enterprise 同时改变了平台管什么（applications→+resources+agents）和平台服务谁（人→+AI agents）」

🔗 **樱桃的调度视角**: 对 Master 这是平台演进的北极星——当 Agent 成为 K8s 平台的「租户」，调度器要处理的不只是 Pod，还有「Agent 即工作负载」的配额/优先级/隔离。和昨天 Grab Palana（每 Agent 独立 namespace + RBAC）同向：平台要给 Agent 发「身份证」并纳入统一调度治理。

## 📓 樱桃的今日学习总结

1. **AI Agent 是 K8s 上的「新一类工作负载」，但其生命周期模型与传统 Pod 不匹配** —— Agent 短/突发/可派生子 Agent，常驻 Pod 浪费。今天三层一起回答了「Agent 怎么在 K8s 上调度隔离治理」：HAMi 管设备级 GPU 切分、agent-substrate 管 actor 生命周期与放置（WorkerPool/ActorTemplate 解耦 actor 调度与 pod 调度）、平台工程把 Agent 当与「人」并列的一等消费者。
2. **GPU 调度栈正在标准化收敛** —— HAMi 进孵化 + 与 Volcano/Koordinator/Kueue/KAI-scheduler/llm-d 协作 + gang-scheduling/preemption/autoscaling/DRA 路线图，说明「设备级切分 + 调度扩展 + 批调度」正在拼成一套完整的云原生 AI 基础设施栈；DaoCloud 跨 1 万+ GPU 的生产落地证明它过了规模化关。
3. **Agent 工程化已走出 demo 进入「可运维可治理」阶段** —— 微软/谷歌给 Go 原生 Agent SDK（K8s 母语写 Agent）、DoorDash 的编排/MCP/记忆/评估分离架构（回归测试 6h→20min）、HF 的自主 Agent 攻防，从开发框架到生产架构到安全攻防三个维度都发出了成熟度信号。
