## 🚀 重点速览

### 1. 📦 FlashQLA：Qwen 线性注意力算子库，前向 2-3 倍、反向 2 倍加速

**原文标题:** 面向 Qwen 系列模型线性注意力的高性能优化实践｜AICon深圳
**来源:** InfoQ 中文 | 2026-07-12
**链接:** https://www.infoq.cn/article/wg5h3JBMvs5ZkoP0azha

🎯 **一句话精髓**: 阿里巴巴 Qwen 团队算法工程师章程瑞东提出 FlashQLA——基于 TileLang 的高性能线性注意力算子库，专门解决 Qwen-3.5 混合注意力结构中 GDN（Gated Delta Network）线性注意力层在训练与推理 Prefill 阶段的访存瓶颈与并行度受限问题。

🔧 **技术栈**: Qwen-3.5 / Gated Delta Network (GDN) / FlashQLA / TileLang / FLA Triton kernel / Hopper GPU / Chunked Prefill / warp-specialization / 门控衰减 / 卡内序列并行

📐 **架构/方法亮点**:
- 两大瓶颈定位：① 多步算子需反复读写 HBM，访存瓶颈严重；② 状态递推的串行性使算子并行度受限于注意力头数，在长序列、小 batch 或 TP 场景下 GPU 利用率低下
- 融合算子拆分：将 GDN Chunked Prefill 的前向与反向流程各拆分为两个融合算子，通过数据复用降低访存开销，并在算子内通过 warp-specialization 实现不同计算的重叠
- 基于门控衰减的自动化卡内序列并行，并建立数学模型动态调节并行粒度；针对具有指数衰减特性的注意力头，舍弃昂贵的修正矩阵计算，改用轻量级预热获得精确状态

💡 **关键数据**: 在 Hopper 显卡上，FlashQLA 相比 FLA Triton kernel 实现前向 2–3 倍、反向 2 倍加速，且在 TP 增大时加速比持续提升；本文字数 1821 字（约 6 分钟阅读），为 AICon 深圳 8 月 21-22 日演讲预告。

🔗 **樱桃的调度视角**: 线性注意力（GDN）的核心特征是「状态递推的串行性」——这直接冲击了 GPU 调度里的两个维度：访存带宽（HBM 读写次数）和并行度（warp/CTP 利用率）。FlashQLA 的「卡内序列并行 + 数学模型动态调节并行粒度」本质是在单卡内做细粒度的算子调度，和推理引擎层面的 Prefill/Decode 分离、CP/TP 选择是同一类问题——都是在串行依赖和并行收益之间找最优切分点。对 Master 的启示：当推理集群里跑 Qwen-3.5 这类线性注意力模型时，GPU 利用率瓶颈不再在显存容量而在算子并行度，调度器的资源画像需要从「显存占用」扩展到「算子可并行度」喵。

### 2. 📦 ShadowRadix：DeepSeek V4 三套异构 KV Cache 的统一管理，百万上下文推理 180 token/s

**原文标题:** 百万上下文下的 DeepSeek V4：SGLang 推理优化实战｜AICon深圳
**来源:** InfoQ 中文 | 2026-07-09
**链接:** https://www.infoq.cn/article/qALuq71AxiG5VLmWqSzU

🎯 **一句话精髓**: SGLang（RadixArk）工程师杨雨豪分享了 DeepSeek V4 百万上下文推理的全栈优化——核心挑战是 SWA/CSA/HCA 三种异构注意力共存、每种有独立 KV Cache 生命周期，团队通过 ShadowRadix 虚拟页表、FlashCompressor 算子融合、Lightning TopK 和 DP/TP/CP/EP 并行策略组合实现 Day-0 支持与持续性能提升。

🔧 **技术栈**: DeepSeek V4 / SGLang / RadixTree / ShadowRadix / SWA（滑窗128）/ CSA（4:1压缩+TopK稀疏筛选）/ HCA（128:1纯压缩）/ FlashCompressor / Lightning TopK / MegaMoE / DeepGEMM / Context Parallelism / Expert Parallelism / GB300 NVL72 / CUDA Graph / 投机采样 MTP

📐 **架构/方法亮点**:
- ShadowRadix 虚拟地址映射：在 RadixTree 上扩展虚拟地址表，每个 token 分配唯一虚拟位置，通过除法映射到三组 Shadow 页表（SWA 原始位置 / CSA 除以 4 / HCA 除以 128）；SWA 引入 Tombstone 机制自动释放滑窗外过期 KV
- 算子级优化：FlashCompressor 将 Compressor 流程中 Softmax/bias/scale-dot 等零散操作融合为单一 kernel，HBM 读写从 5 次降至 2 次；Lightning TopK 将序列分片到同 Cluster 内多 CTA，用直方图本地统计 + SM 互联（而非 HBM）完成 reduce
- 并行策略无通解：超长输入（90 万+）CP 优于 TP（压缩后 KV 体积小、AllGather 通信开销低），短输入 TP 更优，高吞吐用 EP；SWA/CSA/HCA 三种 Attention 无数据依赖，分配到不同 CUDA Stream + CUDA Event 控制依赖实现多流并行
- 投机采样边界：大 Batch（512/1024）下投机采样收益下降，因 GPU 瓶颈从访存转为计算——投机采样本质优化的是访存

💡 **关键数据**: InferenceX 基准（Input 8K / Output 1K）开启 MTP 单用户交互速度 180 token/s；GB300 NVL72 高吞吐场景单 GPU 吞吐 11,500 token/s；Lightning TopK 百万上下文 BS=1 下从 100μs 降至 15μs；4K 与 900K 上下文解码速度差异极小（TP=8, BS=1, MTP Len=3）。

🔗 **樱桃的调度视角**: 这篇对 Master 价值极高——ShadowRadix 的「虚拟地址表 + 分组除法映射」是处理异构 KV Cache 的通用思路，本质上是给三种生命周期不同的资源各建一套独立的页表和淘汰策略。这和 K8s DRA 给不同设备类型建独立 ResourceClaim 的思路高度同构：当集群里同时存在推理（memory-bound decode）、训练（compute-bound + checkpoint 写突发）、Agent（短长期记忆混存）时，调度器也需要类似的「虚拟资源抽象 + 分组策略」。CP（超长输入优于 TP）和 EP（依赖 NVLink 带宽）的选择说明：并行策略是「拓扑感知」的——这恰恰是拓扑感知调度（Topology Aware Scheduling）在推理层面的体现，调度器必须知道 GPU 间是 NVLink 还是 PCIe 才能选对策略喵。

### 3. 📦 CNCF 云原生 AI 数据存储白皮书：按 AI 生命周期刻画存储画像

**原文标题:** The CNCF Data Storage in Cloud Native AI White Paper
**来源:** CNCF Blog / TAG Infrastructure | 2026-07-08
**链接:** https://www.cncf.io/report-whitepaper/2026/07/08/the-cncf-data-storage-in-cloud-native-ai-white-paper/

🎯 **一句话精髓**: CNCF TAG Infrastructure 发布云原生 AI 数据存储白皮书，将云原生 AI 数据生态分为数据湖仓/向量库、缓存与数据本地性、标准化接口（CSI/COSI）、现代数据管道四层，并按训练/推理/Agentic AI 三个生命周期阶段刻画差异化的存储画像，给出可落地的架构模式。

🔧 **技术栈**: Apache Parquet / Apache Iceberg / Milvus / RAG / Fluid（CNCF）/ CSI（Container Storage Interface）/ COSI（Container Object Storage Interface）/ FUSE CSI / CDC（Change Data Capture）/ Apache Kafka / KV Caching / Prefix Caching

📐 **架构/方法亮点**:
- 数据湖仓与向量库融合：用 Parquet/Iceberg 等开放格式将集中式系统演进为混合数据湖仓；向量库（如 Milvus）处理高维 embedding 用于相似搜索和 RAG
- 数据本地性消除传输延迟：白皮书重点推荐 CNCF 项目 Fluid 在 Kubernetes 内编排分布式缓存，让数据贴近计算
- 标准化接口打通存储层：CSI 管块/文件存储、COSI 管对象存储、cloud-native FUSE CSI driver 统一访问；现代数据管道用 CDC + Kafka 从批处理走向实时流式
- AI 生命周期三阶段存储画像：① 训练（长时吞吐导向，最大化 GPU 利用率，需容忍 shuffle 的非顺序访问 + checkpoint 同步写突发）；② 推理（延迟敏感、流量 spiky、需快速模型加载，依赖 KV/Prefix Caching）；③ Agentic AI（闭环迭代推理，需短期记忆/可变状态/append-only 事件历史、artifact 仓库、长期记忆）

💡 **关键数据**: 白皮书由 TAG Infrastructure 社区产出；明确点名 Fluid 作为 K8s 内分布式缓存编排的 CNCF 项目；将 Agentic AI 列为独立存储画像阶段（区别于训练/推理）。

🔗 **樱桃的调度视角**: 这篇白皮书对 Master 的直接价值是「存储画像驱动调度」——训练阶段的 checkpoint 同步写突发、推理阶段的快速模型加载、Agentic 阶段的可变状态 + 事件历史，三者的存储 IO 模式完全不同。Fluid 作为 CNCF 项目把「数据缓存」做成 K8s 原生资源，意味着调度器可以把数据本地性当成一个可调度的维度（类似拓扑感知调度把 NUMA/机架拓扑纳入考量）。Agentic AI 被单独列为存储画像阶段尤其值得关注：Agent 的短期记忆（可变状态）和长期记忆（持久化 session）混存，意味着 Agent Pod 的存储需求是「有状态 + 高频小写 + 长尾读」，这和传统无状态微服务完全不同，调度器需要感知这种存储画像差异喵。

### 4. 📦 ingress-nginx 控制器即将 EOL：迁移 Contour 还是升级 Gateway API？

**原文标题:** Navigating the ingress-NGINX retirement
**来源:** CNCF Blog | 2026-07-09
**链接:** https://www.cncf.io/blog/2026/07/09/navigating-the-ingress-nginx-retirement/

🎯 **一句话精髓**: 社区维护的 ingress-nginx 控制器即将停止维护（未修补 CVE、功能更新和社区支持全面停止），文章系统对比了两条迁移路径——「lift-and-shift」换到 Contour（保持 Ingress API）或借机演进到 Gateway API，并给出审计-工具化-增量上线的迁移策略。

🔧 **技术栈**: ingress-nginx controller / Kubernetes Ingress API / Gateway API / Contour / Envoy / HTTPRoute / ingress2gateway / CRD / nginx.ingress.kubernetes.io 注解

📐 **架构/方法亮点**:
- 关键澄清：被退役的是社区维护的 ingress-nginx controller，而非 Ingress API 本身（Ingress API 仍受支持但已 feature-frozen）；EOL 意味着未修补 CVE 和社区支持停止，带来严重运维风险
- Path A「lift and shift」：保留现有 Ingress YAML，把底层 ingress class 换到 Envoy-based 的 Contour；但所有 nginx.ingress.kubernetes.io/* 专有注解会失效，需手动翻译为 Contour 注解或 CRD
- Path B 架构演进：迁移到 Gateway API（upstream-backed 的 Ingress 继任者），角色导向设计显式分离基础设施关注点与应用路由，标准化流量分割、高级 header 匹配、安全跨 namespace 路由
- 迁移三步法：① 审计现有技术债（盘点 nginx 专有注解）② 用 ingress2gateway 自动化翻译 ③ 增量上线（并行运行、先迁移非关键工作负载）

💡 **关键数据**: Path A 迁移成本低-中（翻译注解）、未来保障低（Ingress API feature-frozen）；Path B 迁移成本高（完全重写路由 manifest）、未来保障高（active upstream）；Path A 为单 owner 运维，Path B 角色化（ops 管 Gateway、devs 管 HTTPRoute）。

🔗 **樱桃的调度视角**: 这篇对 Master 是一个「预警 + 决策框架」。如果推理/AI 服务的流量入口用的是 ingress-nginx，EOL 后的 CVE 风险会直接暴露推理面。樱桃的建议是借这次 EOL 直接评估 Gateway API——AI 推理服务普遍需要流量分割（金丝雀模型版本）、header 匹配（按模型路由）和跨 namespace 路由（模型仓库与推理 Pod 分离），这些都是 Gateway API 原生支持而 Ingress 靠注解硬撑的能力。如果时间紧迫，Path A（Contour 平移）作为过渡也是合理的，但要明确这只是「买时间」而非终点喵。

### 5. 📦 用 NGINX + OpenTelemetry 给 AI Agent 造网络边界：强制且可观测

**原文标题:** Network boundary for AI agents using NGINX and OpenTelemetry
**来源:** CNCF Blog / NGINX（作者 epower）| 2026-07-08
**链接:** https://www.cncf.io/blog/2026/07/08/network-boundary-for-ai-agents-using-nginx-and-opentelemetry/

🎯 **一句话精髓**: 受「不知道那 agent 到底在干什么」的顾虑启发，作者用两个云原生环境里已成熟的组件构建 AI Agent 网络边界——NGINX 作为流量控制面（同时做入站 reverse proxy 和出站 forward proxy，iptables 封死旁路）、OpenTelemetry 作为审计面（每请求 emit span），实现「边界是架构属性而非希望应用遵守的策略」。

🔧 **技术栈**: NGINX forward/reverse proxy / OpenTelemetry / OTel Collector / iptables / Ollama / OpenClaw / Jaeger / Grafana / SIEM / Kubernetes / NVIDIA GPU / NGINX Ingress Controller (NIC) / NGINX Gateway Fabric (NGF)

📐 **架构/方法亮点**:
- 双向代理 + 无旁路：NGINX 入站做 reverse proxy 终止 TLS 转发请求，出站做 forward proxy 让每个 agent 请求必经；iptables 规则 drop 所有其他 egress 流量，确保「没有第二条路径」——边界成为架构属性而非策略
- 可观测即审计：NGINX 原生 OpenTelemetry module 为每个请求 emit 一个 OTEL span，OTel Collector 持久化到审计日志或喂给 Jaeger/Grafana/SIEM，实现「流量控制 + 审计」一体
- 验证部署：单节点 K8s 集群同 namespace 跑 NGINX、Ollama、OpenClaw、OTel Collector 四个 workload，配消费级 NVIDIA GPU，设计可从 edge 镜像到企业级 AI 基础设施
- 局限坦诚：只控制/观察网络行为不理解 agent intent，限制通信不保证决策正确安全，是纵深防御的一层而非全部

💡 **关键数据**: 开源代码在 OpenClaw Network Boundary 仓库（提供 K8s 部署代码）；NGINX config map 可精细到只放行 nginx.org 和 duckduckgo.com；NIC 与 NGF 将逐步继承核心 NGINX 的 forward proxy 能力。

🔗 **樱桃的调度视角**: 这篇直接回应了 Master 在 K8s 上跑 Agent 推理时的安全痛点——Agent 有工具调用能力意味着它会主动发起出站请求，传统 NetworkPolicy 只做 L3/L4 放行无法做应用感知的细粒度控制。NGINX 做 forward proxy + iptables 封旁路的「无第二条路径」设计很巧妙，本质是把 egress 网关做成强制的网络拓扑约束。OpenTelemetry span 审计面则让 Agent 的每一次外部调用都可追溯——这对推理集群的合规审计很重要。樱桃的延伸思考：这套模式和 service mesh 的 egress gateway 思路一致，但更轻量，适合 Master 先在单集群内小范围验证 Agent 的网络行为画像，再决定是否升级到全 mesh 方案喵。

### 6. 📦 Netflix Cassandra 动态分区拆分：读取延迟从数秒降到毫秒级

**原文标题:** 动态分区拆分：Netflix 大幅削减 Cassandra 读取延迟至毫秒级
**来源:** InfoQ 中文 | 2026-07-09
**链接:** https://www.infoq.cn/article/GRDxYebxA7ywxO4seBWy

🎯 **一句话精髓**: Netflix 为其时间序列抽象平台开发了适用于 Apache Cassandra 的动态分区拆分机制——自动检测超大分区并异步拆分为子分区，通过元数据层路由查询，在无需修改应用、不停机、不大规模重新分区的前提下，将读取延迟从数秒降到毫秒级。

🔧 **技术栈**: Apache Cassandra / 时间序列抽象平台 / 动态分区拆分 / 元数据层（父-子分区映射）/ 异步拆分管道 / 不可变分区 / 灰度上线 / 一致性校验管道

📐 **架构/方法亮点**:
- 元数据层路由：引入元数据层跟踪父-子分区关系，读取时元数据服务确定哪些子分区含请求数据并路由查询，多分区结果合并返回；底层存储变更对应用完全透明（仍查逻辑分区）
- 运维安全设计：分布式分区拆分会带来并发写入/数据迁移/一致性难题，首版仅针对不可变分区实现拆分，数据迁移全程保留原始分区作为回退机制
- 分阶段灰度 + 校验：验证管道比较原始分区结果 vs 新读取路径结果，一致性达标后才逐步增加生产流量；采用「缩小变更影响范围 + 增量式校验」思路降低上线风险

💡 **关键数据**: 此前分区超 500MB 就出现服务可用性故障的业务部署后可正常分页查询；平均读取延迟从数秒降到低于两位数毫秒级；尾部延迟从数秒缩短至 200ms 及以下；Cassandra 集群读取超时次数、CPU 利用率下降，线程排队阻塞大幅减少。原文：https://www.infoq.com/news/2026/07/netflix-cassandra-partition/

🔗 **樱桃的调度视角**: 这篇表面是数据基础设施，但对 Master 的调度视角有两点启发：① 「分区大小不均导致提前扩展」和 K8s 调度里的负载不均是同一类问题——一个热点分片失衡会迫使集群提前扩容，调度器的负载均衡必须感知分片级热度而非只看节点级平均；② 「保留原始分区作为回退 + 灰度校验」的迁移策略，和 K8s 上做有状态工作负载（如推理模型版本灰度、向量库分片再平衡）的滚动迁移完全同构——先在新路径跑影子读取校验一致性，再切流量，是任何有状态系统迁移的安全范式喵。

## 📓 樱桃的今日学习总结

1. **复杂注意力架构正在让推理引擎的「KV Cache 管理」从单一系统走向多系统并存** - DeepSeek V4 的 ShadowRadix 为 SWA/CSA/HCA 三种注意力各维护一套独立的页表、淘汰策略和前缀匹配逻辑，Qwen 的 GDN 也需要专门的 FlashQLA 算子库。这说明当模型架构变复杂（线性注意力、混合稀疏注意力），推理引擎的缓存管理不再是单一 RadixTree 能搞定的，而是「虚拟地址抽象 + 分组策略」的多系统设计。代价是系统复杂度膨胀、联调和一致性维护成本高——这是复杂模型落地的工程必然。

2. **推理的并行策略是「拓扑感知」的，没有通解只有组合空间** - DeepSeek V4 的实践明确指出：超长输入用 CP（因压缩后 KV 体积小、AllGather 通信开销低）、短输入用 TP、高吞吐用 EP，且运行时动态切换并行策略不可行（涉及 CUDA Graph 重建、元数据重建）。这给调度器的启示是：推理调度不能只看显存余量，必须感知 GPU 间互联拓扑（NVLink vs PCIe）和输入长度分布，才能选对并行策略——这正是拓扑感知调度在推理层面的体现。

3. **AI 工作负载的「存储画像」正在按生命周期分化，调度器需要感知存储差异** - CNCF 白皮书把训练（checkpoint 同步写突发）、推理（快速模型加载 + KV/Prefix Caching）、Agentic AI（可变状态 + 事件历史 + 长期记忆）列为三种截然不同的存储画像。加上 Netflix Cassandra 的「分区热度不均导致提前扩展」，共同指向一个调度原则：有状态 AI 工作负载的调度必须感知存储 IO 模式和分片热度，而非像无状态微服务那样只看 CPU/内存平均水位。
