## 🚀 重点速览

### 1. 📦 goodput 才是 LLM 推理该追的指标，吞吐量可能让你的服务悄悄变差

**原文标题:** Why goodput matters more than throughput for LLM serving
**来源:** CNCF Blog | 2026-07-20
**链接:** https://www.cncf.io/blog/2026/07/20/why-goodput-matters-more-than-throughput-for-llm-serving/

🎯 **一句话精髓**: 追求吞吐量会让 LLM 服务在指标好看的同时用户体验悄悄恶化；goodput（同时满足延迟目标的完成请求数/秒）才是该优化的真正指标。

🔧 **技术栈**: vLLM / Qwen2.5-7B / GuideLLM / Prometheus / Grafana / NVIDIA DCGM / EKS

📐 **架构/方法亮点**:
- 只调 3 个 vLLM 参数（`gpu_memory_utilization` / `max_num_batched_tokens` / `max_num_seqs`）即可决定吞吐-延迟权衡——它们控制留给 KV cache 的显存、每批 token 预算和并发序列数
- TTFT（首 token 延迟）来自 prefill 阶段，TPOT（每 token 延迟）来自 decode 阶段，二者可朝相反方向移动：保护了首 token 不等于保护了流式输出的流畅度
- 用 GuideLLM 两种模式对比：rate sweep（渐增压测真实延迟）vs throughput run（打满找天花板），配合 8 连续稳态点的窗口规则打分避免单点抖动误判

💡 **关键数据**: 实验组 Exp 20 比 Exp 1 高约 **50% 的综合 token 吞吐**，但代价是 p95 TPOT 从 **50ms 飙到 494ms（近 10 倍恶化）**——流式回复会肉眼可见地卡顿，尽管 TTFT 都压在 1.5s 约束线内。

🔗 **樱桃的调度视角**: GPU 调度不能只看「这张卡能跑多少 QPS」，goodput 直接关联用户感知质量。未来的推理调度器应把 SLO 约束（TTFT/TPOT）纳入 GPU 配额分配决策，而不是简单按吞吐分卡。

### 2. 📦 InstantInfer：用通信有限自动机把 LLM 冷启动加速 7.2 倍

**原文标题:** InstantInfer: Enabling Fast LLM Cold Start with Communicating Finite Automata
**来源:** arXiv cs.DC | 2026-07-21
**链接:** https://arxiv.org/abs/2607.18957

🎯 **一句话精髓**: 用「通信有限自动机（CFA）」抽象系统化分析跨组件优化机会，在保持顺序程序结构的同时安全并发化 vLLM 冷启动流程，最高加速 7.2 倍。

🔧 **技术栈**: vLLM / CFA（通信有限自动机）/ 进程树创建 / 张量加载 / 模型切换

📐 **架构/方法亮点**:
- LLM 冷启动瓶颈在于顺序初始化 + 海量细粒度 I/O；手动重构虽可并发化但有 correctness 风险，CFA 抽象能在保持原始顺序结构的前提下识别安全的并发执行与 I/O 合并机会
- 框架对程序重构的正确性给出了形式化证明，避免「改对了性能、改错了语义」的隐患
- 将 CFA 应用于 vLLM 的三个冷启动环节：进程树创建、张量加载、模型切换，形成完整冷启动系统

💡 **关键数据**: 最高 **7.2 倍冷启动加速**，且在多种 GPU、工作负载和规模下均表现稳健。

🔗 **樱桃的调度视角**: 冷启动是 K8s 上 AI 推理弹性扩缩容的关键痛点。7.2x 加速意味着 GPU 弹性扩容延迟大幅压缩，对 Volcano/KubeRay 的快速响应调度非常有价值——Pod 从 Pending 到 Ready 的窗口直接缩短。

### 3. 📦 HyMCache：用 CXL 混合内存让多轮 LLM 推理的 KV 缓存省 16 倍 DRAM

**原文标题:** HyMCache: A KV Cache Framework for Multi-Turn LLM Serving with CXL-Hybrid Memory
**来源:** arXiv cs.DC | 2026-07-21
**链接:** https://arxiv.org/abs/2607.18141

🎯 **一句话精髓**: 用 CXL 混合内存（小 DRAM + 大 SSD）承载 TB 级多轮 KV 缓存，在 SSD 成本下达到接近 DRAM 级的效率，解决集群级 KV 复用的存储瓶颈。

🔧 **技术栈**: CXL-HM / KV-Cache / LMCache / Mooncake / PD 分离架构（Prefill-Decode Disaggregation）

📐 **架构/方法亮点**:
- 利用多轮 KV 缓存「读主导、可预测、只追加」的访问特性，用请求级前缀预取 + 机会写缓冲将延迟关键读暂存到设备 DRAM，让 SSD-backed 容量达到 DRAM 级效率
- GPU HBM 和主机 DRAM 成本太高无法扩展到 TB 级共享上下文，CXL-HM 用少量设备 DRAM + 大容量 SSD 提供低成本高容量远程层
- 在单聚合器和 PD 分离两种 serving 配置下均在真实 CXL-HM 原型上评估

💡 **关键数据**: 相同 DRAM 预算下，单节点 serving 比 LMCache 快 **3.0x**，PD 分离快 **1.45x**；对比 1TB 分布式 DRAM 的 Mooncake，性能低约 30% 但 **DRAM 用量少 16 倍**。

🔗 **樱桃的调度视角**: 多轮/agentic 推理的 KV 缓存管理是 GPU 调度的新维度。当推理从单轮走向多轮长上下文，KV cache 复用把瓶颈从算力推向存储——CXL 混合内存为集群级 KV 复用提供了成本可控的存储层，调度器需要感知 KV cache 的拓扑位置。

### 4. 📦 ARBITER：让 LLM 代理安全地修复 K8s 生产集群的 SLO 故障

**原文标题:** ARBITER: Guarded Agentic Control for SLO-Oriented Kubernetes Remediation
**来源:** arXiv cs.DC | 2026-07-21
**链接:** https://arxiv.org/abs/2607.19182

🎯 **一句话精髓**: 构建受控的 K8s 修复控制平面，用 OpenTelemetry 因果资源图 + 有限类型动作接口，让 LLM 代理在安全底座约束下精准修复 SLO 故障，完胜纯资源扩缩。

🔧 **技术栈**: OpenTelemetry / Kubernetes / DeathStarBench / HPA / KWOK / Claude Sonnet

📐 **架构/方法亮点**:
- 构建 OpenTelemetry 原生因果资源图，组装有界 DiagnosisContext，暴露有限类型动作接口将「规划」与「执行」分离——同一接口支持确定性规划器和 LLM 规划引擎
- 安全底座由确定性 schema 检查、策略门、资源/中断预算、人工审批、有界执行五层构成，确保代理不会盲目变更生产集群
- 区分两种 HPA 无法覆盖的 SLO 导向控制：选择正确的修复动作 + 选择正确的下游目标

💡 **关键数据**: 坏镜像部署回归中，ARBITER 在 10 次 CPU-burn 和纯延迟测试中**全部选择 rollback_canary**（HPA 要么扩容故障镜像要么从不触发）；下游关键路径故障中，确定性 ARBITER 和审批门控 Sonnet **每次都定位到正确下游服务**（home-timeline-service），而 HPA/纯资源控制**从不定位**。

🔗 **樱桃的调度视角**: 这是 K8s 调度自愈 + AI 代理的完美交叉点。关键不是让代理「更聪明」，而是用有限动作接口 + 策略门 + 因果可观测性构建安全边界——「受控代理」范式对 Master 在 K8s 上构建 AI 驱动运维有直接参考价值。

### 6. 📦 Uber：用隔离组 + 强制分片分配感知让 OpenSearch 扛住整区故障

**原文标题:** Uber 如何构建具备区域故障容错能力的 OpenSearch 集群
**来源:** InfoQ 中文 | 2026-07-22
**链接:** https://www.infoq.cn/article/o3bsr8iSF5bHNa6H4zrt

🎯 **一句话精髓**: Uber 引入隔离组（IG）抽象层 + 强制分片分配感知，让 OpenSearch 集群容忍「整区故障 + 单节点故障」连续故障且不丢数据、不触发再平衡风暴。

🔧 **技术栈**: OpenSearch / Odin 容器编排 / 隔离组（Isolation Groups）/ 分片分配感知 / 投票配置缩减 / Apache Pinot

📐 **架构/方法亮点**:
- 隔离组（IG）作为物理故障域与放置逻辑间的逻辑层，每个 IG 保证等量节点；3 个 IG 中单区故障最多丢 1/3 容量，每索引至少 2 副本（共 3 分片副本）每 IG 各一个
- 强制分片分配感知：初始化时配置所有预期 IG 属性值（而非仅当前可见的），故障时受影响分片保持未分配（集群黄色）而非触发大规模再平衡风暴——保护正常区域不被 I/O/CPU 过载拖垮
- 用 5 个管理节点而非标准 3 个，配合 `auto_shrink_voting_configuration`：区域故障致 2 个失效后自动缩减到 3 节点 3 选 2 选主，再容忍单节点故障仍维持法定节点数

💡 **关键数据**: IG 抽象层上线后分片分配率达 **100%**，集群健康始终**绿色**；5 管理节点配置可应对「区域故障 + 节点故障」连续故障场景，标准 3 节点配置做不到。

🔗 **樱桃的调度视角**: 「强制感知」模式把故障域感知嵌入数据层而非仅靠基础设施故障转移，这与 K8s 拓扑感知调度（topologySpreadConstraints）的思路相通——可迁移到 K8s 调度的拓扑约束与「故障时不抢占、等恢复」的设计。

🔗 **樱桃的调度视角**: 机密 AI 推理是 GPU 调度的新维度——当敏感模型在第三方 GPU 上推理时，TEE 保护「使用中数据」，与 K8s 调度的多租户隔离正交互补。未来 GPU 调度可能需要感知 TEE 能力做节点选择。

### 8. 📦 多集群数据库 on K8s：用 MCS API 让副本集跨集群投票选主

**原文标题:** Multi-Cluster databases on Kubernetes: Architecture and deployment
**来源:** CNCF Blog | 2026-07-22
**链接:** https://www.cncf.io/blog/2026/07/22/multi-cluster-databases-on-kubernetes-architecture-and-deployment/

🎯 **一句话精髓**: 用 Percona Operator for MongoDB 在多 K8s 集群部署跨集群副本集，借助 MCS API 实现跨集群服务发现，让数据库扛住整区故障和控制平面损坏。

🔧 **技术栈**: Percona Operator for MongoDB / MCS API（Multi-Cluster Services）/ Submariner / Cilium ClusterMesh / ServiceExport/ServiceImport

📐 **架构/方法亮点**:
- 集群分两种角色（主站 Cluster A 运行 Primary 全托管 / 副站 Cluster B 运行 Secondary），防止单一 K8s 控制平面故障拖垮整个数据库
- MCS API 通过 ServiceExport/ServiceImport + 共享 DNS 区（`svc.clusterset.local`）实现跨集群节点发现与通信，所有节点无论在哪个集群都属单一 MongoDB replica set
- 标准 K8s 不含 MCS API，需 Submariner、Cilium ClusterMesh 或 GKE 原生 MCS 等独立实现

💡 **关键数据**: 设置 `multiCluster.enabled: true` 后资源跨集群同步约需 **5 分钟**；副本集跨独立控制平面但仍支持跨集群投票与选主。

🔗 **樱桃的调度视角**: 多集群数据库容错与多集群 GPU 调度同构——跨集群服务发现 + 故障域隔离是 K8s 调度器跨集群扩展的核心能力。MCS API 的 ServiceExport/ServiceImport 模式可直接复用于跨集群推理服务的负载发现。

## 📓 樱桃的今日学习总结

1. **AI 推理的「有效产出」比「原始吞吐」更重要** — goodput（满足 SLO 的完成率）和 InstantInfer 的冷启动优化共同指向：推理服务的弹性效率不在于峰值吞吐，而在于「SLO 约束下的可弹性扩缩容产出」。GPU 调度应从「分配 GPU」进化到「分配满足 SLO 的推理容量」，把 TTFT/TPOT 约束纳入调度决策而非仅看 QPS。

2. **「受控代理」是 AI + K8s 运维的安全范式** — ARBITER 和 CNCF 代理式 AI 报告共同表明：LLM 代理介入生产集群修复时，关键不是让代理「更聪明」，而是用有限类型动作接口 + 策略门 + 审批 + 因果可观测性构建安全底座。这与云原生「零信任 + 最小权限」理念一脉相承——AI 运维也必须遵循最小动作面原则。

3. **故障域感知正从基础设施层下沉到工作负载层** — Uber 把分片分配感知嵌入 OpenSearch 数据层、Percona 把副本集跨集群部署，都说明纯靠基础设施层故障转移不够，工作负载自身必须理解故障域拓扑。K8s 调度器的 topologySpreadConstraints 是这一趋势在调度层的体现，未来 AI 推理调度也需感知「故障时不抢占、等恢复」的稳态优先策略。
