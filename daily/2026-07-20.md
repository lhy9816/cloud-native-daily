### 1. 📦 云天励飞三款推理芯片：把 Prefill / Decode / FFN 拆给不同芯

**原文标题:** 三款AI推理芯片+超节点异构集群，云天励飞公布未来算力蓝图
**来源:** InfoQ 中文 | 2026-07-19
**链接:** https://www.infoq.cn/article/LCMZqxPWHIFvQbuRIS6f

🎯 **一句话精髓**: 云天励飞将大模型推理的 Prefill、Decode、Decode-FFN 三个阶段拆分给三款专用芯片（DeepVerse100P/D/L），在万卡异构集群上做推理阶段解耦与算力精细化匹配，把 Token 生成成本作为系统级核心优化指标。

🔧 **技术栈**: DeepVerse100P / DeepVerse100D / DeepVerse100L / Prefill-Decode 解耦 / 光互联 / 3D Memory / MoE / 万卡异构集群 / Scale-up 1024 卡

📐 **架构/方法亮点**:
- **DeepVerse100P** 面向百万级上下文 Prefill 场景，解决传统混部架构中长上下文 Prefill 与 Decode 共享算力带宽导致的资源抢占问题
- **DeepVerse100D** 面向 Decode 环节，拥有数倍于主流芯片的内存带宽，支持 1024 卡 Scale-up 及光互联系统架构--按需建立光路直连并动态重构光路，减少多跳电交换排队与尾延迟
- **DeepVerse100L** 面向 Decode 阶段计算密集型 FFN 环节，采用 3D Memory 架构大幅提升内存带宽，通过低延迟芯片互联提高计算与通信并行效率

💡 **关键数据**: 三款芯片面向万卡异构集群协同部署；DeepVerse100D 支持 1024 卡 Scale-up；随着 MoE 发展，Decode 阶段 Attention 与 FFN 在计算/访存上呈现不同资源需求

🔗 **樱桃的调度视角**: 这正是 Master 关注的推理调度新维度--当 Prefill 和 Decode 跑在不同芯片上，K8s 调度器不能只看 GPU 数量，还要感知「推理阶段」做异构资源匹配。类似 vLLM 的 prefill-decode disaggregation 思路正在从软件层向硬件层渗透，未来 K8s 调度可能需要「阶段感知」的 binpack 策略。

### 2. 📦 阡视科技 wylon 超节点：为 Token 工厂建「纵向存力贯通平面」

**原文标题:** WAIC 2026现场，阡视科技发布超节点系统，专为Token工厂而生
**来源:** InfoQ 中文 | 2026-07-19
**链接:** https://www.infoq.cn/article/TjLEm0VKwkrrkLxOds61

🎯 **一句话精髓**: 阡视科技发布 wylon 超节点系统，在传统 GPU 横向互联平面之外新增「纵向存力贯通平面」，专门为 KV Cache 频繁读写构建百 TB 级近存 + PB 级闪存分层，在国产 GPU 平台上实现 10 倍以上性能提升。

🔧 **技术栈**: wylon 超节点 / HitenOS / KV 分层管理 / 算力亲和调度 / 硬件拓扑感知 / 寒武纪 / 壁仞 / 沐曦 / FP8/FP4

📐 **架构/方法亮点**:
- **纵向存力贯通平面**: 传统超节点只侧重 GPU 横向互联，wylon 打通异构存力层级--百 TB 级 GPU 近存 + 系统内存 + PB 级大带宽闪存，专为 KV Cache 读写瓶颈设计
- **算力亲和调度 + 硬件拓扑感知**: 模型负载直达硬件底层，避免中间层性能损耗；KV 分层管理确保热数据在近存、冷数据下沉闪存
- **多国产 GPU 适配**: 深度联合寒武纪、壁仞、沐曦、曦望、海光、摩尔线程，构建从芯片到系统的全栈国产 AI 基础设施

💡 **关键数据**: wylon Q72/288 单机柜集成 72 颗国产 GPU，4 机柜横向互联扩展至 288 卡；GPU 高带宽显存达 18 TB，互联总带宽 72 TB/s；为 KV Cache 设计的统一 DRAM 缓存可达上百 TB；整体性能提升 10 倍以上

🔗 **樱桃的调度视角**: 「存力平面」这个概念对 Master 很有启发--当 KV Cache 成为推理系统的新瓶颈，K8s 调度器需要感知的不再只是 GPU 显存，而是分层的存力拓扑（近存/内存/闪存）。wylon 的「算力亲和调度」思路与 K8s 的 topology-aware scheduling 异曲同工，只是粒度从 NUMA node 细化到了存力层级。

### 5. 📦 OpAMP：给可观测代理舰队装上远程遥控器

**原文标题:** Operating OpenTelemetry at scale with OpAMP
**来源:** CNCF Blog | 2026-07-13
**链接:** https://www.cncf.io/blog/2026/07/13/operating-opentelemetry-at-scale-with-opamp/

🎯 **一句话精髓**: Open Agent Management Protocol (OpAMP) 用仅两条 protobuf 消息（server-to-agent / agent-to-server）标准化 OTel Collector 舰队的远程配置、健康监控与版本更新，让百万级可观测代理不再是黑盒。

🔧 **技术栈**: OpAMP / OpenTelemetry Collector / Protocol Buffers / WebSocket / HTTP / OpAMP Supervisor / OpAMP Bridge / OTel Operator / CRD / Fluent Bit

📐 **架构/方法亮点**:
- **三层组件**: OpAMP Extension（只读，报告当前配置与健康状态）→ OpAMP Supervisor（独立进程，读写能力，写新配置到磁盘后重启 Collector，启动失败自动回滚到 last-known-good 配置）→ 管理平台
- **K8s 原生路径**: OpAMP Bridge 充当 OpAMP 管理平台与 K8s OTel Operator 之间的中介--Bridge 收到配置后操作 Operator 读写 CRD，由 Operator 部署/更新 Collector，而非直接管理 Pod
- **超越 Collector**: 配置 payload 是通用 name-value map，已用于管理 OTel Java SDK（远程调采样率/开 debug 日志无需重新部署）和 Fluent Bit 舰队

💡 **关键数据**: 规模从数十到数百万 Collector（IoT/嵌入式场景可达百万级）；Nike 在 KubeCon NA 2025 分享了企业级 OpAMP 实现；BindPlane 此前自研了 4-5 套不同的 agent 管理协议（HTTP 长轮询/WebSocket/protobuf/JSON）才转向 OpAMP 标准化

🔗 **樱桃的调度视角**: OpAMP Bridge → OTel Operator → CRD → Collector 这条链路是 K8s 声明式管理的经典模式。对 Master 来说，当 K8s 集群中跑着大量 OTel Collector / Agent DaemonSet 时，OpAMP 解决了「配置漂移检测 + 安全回滚」的运维痛点。Supervisor 的 last-known-good 回滚机制也值得 K8s 控制器设计借鉴。

### 7. 📦 Cloudflare DNSSEC 新突破：NTA 不再静默，EDE 让客户端知道验证被跳过

**原文标题:** A broken DNSSEC rollover took down .al. Now 1.1.1.1 tells you when validation is bypassed
**来源:** Cloudflare Blog | 2026-07-14
**链接:** https://blog.cloudflare.com/dnssec-nta-ede-33/

🎯 **一句话精髓**: 阿尔巴尼亚 .al 顶级域 DNSSEC 密钥轮转故障导致全国域名不可达，Cloudflare 安装 Negative Trust Anchor (NTA) 恢复解析后，首次在 DNS 响应中返回 EDE (Extended DNS Error) 码，让客户端明确知道「此响应未经 DNSSEC 验证」--填补了 NTA 静默跳过验证的透明度缺口。

🔧 **技术栈**: DNSSEC / Negative Trust Anchor (RFC 7646) / Extended DNS Error (RFC 8914) / 1.1.1.1 / DS 记录 / DNSKEY / kdig

📐 **架构/方法亮点**:
- **NTA 的代价与透明度问题**: NTA (RFC 7646) 让递归解析器将故障域视为未签名以恢复解析，但此前响应中不携带任何验证被跳过的信号，客户端无法区分合法答案与伪造答案
- **EDE 码实现**: 基于 Quad9 的 Babak Farrokhi 提出的 Internet-Draft，1.1.1.1 在 NTA 生效期间对每个受影响响应返回 EDE 码 9 (DNSKEY Missing)，附带人类可读说明
- **安全回滚流程**: Cloudflare 尝试联系 .al 运营方（联系方式本身在 .al 下，不可达）+ 在 DNS-OARC Mattermost 告警社区 → 安装 NTA → 次日 .al 运营方移除 DS 记录后移除 NTA

💡 **关键数据**: .al 在 Cloudflare Radar TLD 排名第 191；NTA 在 UTC 17:15 部署完成，距链断裂约 3 小时；截至发文 .al 仍处于未签名状态（DS 记录未恢复到根区）

🔗 **樱桃的调度视角**: 虽然不直接涉及 K8s 调度，但 NTA + EDE 的「降级但透明」设计哲学值得借鉴--在 K8s 中当准入控制/网络策略因故障被临时旁路时，也应该有类似的 EDE 机制让运维知晓「安全策略当前被跳过」，而不是静默降级。CoreDNS 作为 K8s 默认 DNS 也支持 EDE，集群内 DNS 可观测性可由此提升。

## 📓 樱桃的今日学习总结

1. **AI 推理的「阶段解耦」正从软件向硬件渗透** - 云天励飞把 Prefill/Decode/FFN 拆给三款芯片、阡视科技为 KV Cache 单建纵向存力平面、英伟达 Vera Rubin 做 AFD（注意力-前馈解耦）--三件事指向同一趋势：推理不再是一块 GPU 跑全流程，而是按阶段拆分到异构算力。这对 K8s 调度器意味着新的调度维度：不仅要感知 GPU 数量和显存，还要感知推理阶段（prefill vs decode）、KV Cache 存力层级和芯片类型。Master 关注的调度方向正在从「资源 binpack」进化为「阶段感知 + 存力拓扑感知」的异构调度。

2. **Agent 治理的控制平面正在成型** - AWS Claude 应用网关（身份/策略/成本/遥测/路由五合一）、Cube Sandbox（沙箱隔离 + 快照回滚）、LitmusChaos MCP（自然语言交互混沌工程）--Agent 的企业级管控正在从「开发者各自管理凭证」转向「中心化控制平面 + 短期令牌 + 策略引擎」。在 K8s 上，这对应的是 Workload Identity/IRSA + 准入控制 (Kyverno/OPA) + 网关层成本限额的组合。Agent 不是普通工作负载，它的凭证、沙箱、成本都需要专门的控制平面。

3. **「可观测你的可观测」与「降级但透明」是两个值得记住的工程哲学** - OpAMP 让可观测基础设施自身可被观测（Collector 健康监控 + 配置漂移检测 + 安全回滚），Cloudflare EDE 让 DNSSEC 验证跳过从静默变为透明。两者的共性是：关键基础设施的降级和故障不能是黑盒。在 K8s 运维中，调度器降级（如关掉某优先级插件）、准入控制旁路、DNS 解析异常都应该有类似的显式信号机制，让运维在第一时间感知「正常工作」与「降级运行」的区别。
