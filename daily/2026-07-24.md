### 1. 📦 Kubeflow 遇上 Cilium：调试 K8s 上 60% 闲置 GPU 的根因与修复

**原文标题:** When Kubeflow meets Cilium: Debugging 60% idle GPUs in Kubernetes
**来源:** CNCF Blog | 2026-07-23
**链接:** https://www.cncf.io/blog/2026/07/23/when-kubeflow-meets-cilium-debugging-60-idle-gpus-in-kubernetes/

🎯 **一句话精髓**: 分布式训练任务全部 Running、健康检查全绿，但 GPU 却大面积闲置——根因是 K8s 调度器的「拓扑盲」与 Cilium 网络策略的「拓扑感知」相互冲突，协调器被调度到了与 GPU worker 不同的可用区，网络策略默默切断了通信。

🔧 **技术栈**: Kubernetes 调度器 / Kubeflow / Cilium (CiliumNetworkPolicy) / NCCL / nodeAffinity / topologySpreadConstraints / podAffinity / Prometheus / Grafana / kind

📐 **架构/方法亮点**:
- **故障三态**: 硬阻断（跨区网络策略直接拒绝连接，秒级暴露）、跨区延迟（无硬阻断时梯度同步每步付跨区 RTT，吞吐静默下降 30–60% 且无报错）、隐性浪费（健康全绿但算力空转）
- **根因模型**: K8s 调度按 CPU/内存/GPU 计数放置 Pod，对可用区「拓扑无感」；Kubeflow 继承该假设；而 Cilium 用 CNP 划分区边界做爆炸半径隔离——两个系统各自决策都正确，合起来却错了
- **修复手法**: 不改 K8s/Kubeflow 源码，只在 workload spec 补齐调度器缺失的信息——`nodeAffinity` 把整组钉到 GPU 区、`topologySpreadConstraints` 协调器与 worker 共置、`podAffinity`+zone topology key 免硬编码区名

💡 **关键数据**: 协调器与 worker 共置后，实验室 GPU 利用率从约 **40% 跃升到约 85%**；跨区延迟模式下吞吐静默掉 30–60% 且零报错；附可复现 lab（kind 集群 + Cilium 策略 + Prometheus recording rules + Grafana dashboard）

🔗 **樱桃的调度视角**: 这正是 Master 做 GPU 调度最该警惕的「静默失败」——调度器只看资源计数、看不见网络拓扑与策略边界。做 AI 推理/训练调度时，把 topologySpreadConstraints 与网络策略对齐，比堆资源配额重要得多喵～

### 2. 📦 LLM 服务别只盯吞吐：为什么 goodput 才是该优化的指标

**原文标题:** Why goodput matters more than throughput for LLM serving
**来源:** CNCF Blog | 2026-07-20
**链接:** https://www.cncf.io/blog/2026/07/20/why-goodput-matters-more-than-throughput-for-llm-serving/

🎯 **一句话精髓**: 作者在 EKS 上用 vLLM 服务 Qwen2.5-7B，发现吞吐可以一路涨、服务却在悄悄变差——真正该追的是「满足延迟 SLO 的完成请求数/秒」（goodput），而非裸吞吐。

🔧 **技术栈**: vLLM / Qwen2.5-7B / NVIDIA A10G (24GB) / Amazon EKS / GuideLLM / Prometheus / Grafana / TTFT / TPOT

📐 **架构/方法亮点**:
- **goodput 定义**: 每秒完成且满足延迟目标的请求数；LLM 场景的两个 SLO 锚点是 TTFT（首 token 等待）与 TPOT（后续 token 流式速度），超目标的请求「算吞吐但不算 goodput」
- **调参三件套**: `gpu_memory_utilization`（留给 KV cache 的显存）、`max_num_batched_tokens`（每步批 token 预算）、`max_num_seqs`（并发序列上限）——它们同时决定 KV cache 余量、批密度与并发度
- **压测方法**: GuideLLM 的 rate sweep（渐进加压读稳态）+ throughput run（极限压满看天花板）；从连续 8 个数据点的窗口取值而非单点，避免抖动误判

💡 **关键数据**: 两组配置 TTFT 都压在 ~1.5s 上限附近——Exp1（gpu_mem=0.89/batch=4160/seqs=212，TTFT≈1395ms）vs Exp20（gpu_mem=0.87/batch=2150/seqs=271，TTFT=1403ms，Prefill 2934 tok/s、Decode 690 tok/s）；Exp20 在同等延迟约束下综合吞吐显著胜出，证明「同 TTFT 下调参仍有大幅空间」

🔗 **樱桃的调度视角**: 对 Master 调度 AI 推理集群的启示——按裸吞吐做 autoscaling 会越扩越歪，应该把 goodput（TTFT/TPOT 约束下的有效完成率）作为 HPA/调度优化的信号源，配合 Volcano/KEDA 做延迟感知的弹性才靠谱喵～

### 3. 📦 Linkerd 2.20 发布：感知限流负载均衡 + 控制面内存最高降 85%

**原文标题:** Linkerd 2.20 发布：智能流量管理升级，资源占用大幅降低
**来源:** InfoQ 中文 | 2026-07-23
**链接:** https://www.infoq.cn/article/TJL8TIiuD6Cev0Ig0AAS

🎯 **一句话精髓**: CNCF 毕业服务网格 Linkerd 2.20 引入「感知限流的负载均衡」——代理能识别后端的 HTTP 限流响应并主动绕开过载实例，同时在 Pod 频繁 churn 场景下把控制面内存砍掉最多 85%。

🔧 **技术栈**: Linkerd / 服务网格 / Kubernetes 网络 / Rate-Limit-Aware Load Balancing / HTTP 限流响应 / 控制面

📐 **架构/方法亮点**:
- **感知限流路由**: 传统网格按延迟/可用性分发流量；新版能识别后端返回的 HTTP 限流信号，自动把请求暂绕到更健康的实例，在维持整体吞吐的同时降低级联故障风险（尤其适合重度依赖第三方 API 的场景）

- **设计哲学**: 优先解决生产运维实际问题，而非堆功能复杂度——延续轻量定位

💡 **关键数据**: 控制面内存占用最高降低 **85%**（Pod churn 场景）；新增入站流量指标改进

🔗 **樱桃的调度视角**: 「感知限流」本质是把负载均衡从「被动看延迟」升级到「主动看过载信号」——这跟 K8s 调度器从「资源计数」走向「真实负载感知」是同一个方向。Master 做推理流量调度时，这种「限流感知」思路可直接借鉴到 gateway/HPA 层喵～

### 4. 📦 Confidential Containers 升级为 CNCF 孵化项目：用 TEE 保护「使用中数据」

**原文标题:** Confidential Containers becomes a CNCF incubating project
**来源:** CNCF Blog | 2026-07-22
**链接:** https://www.cncf.io/blog/2026/07/22/confidential-containers-becomes-a-cncf-incubating-project/

🎯 **一句话精髓**: CNCF TOC 投票通过 Confidential Containers（CoCo）升级为孵化项目——它基于硬件 TEE 在计算过程中加密数据，让敏感工作负载能安全跑在第三方基础设施上，并已与 KServe 打通做「机密 AI 推理」。

🔧 **技术栈**: Confidential Containers / Kata Containers 运行时 / TEE（可信执行环境）/ Kyverno / KServe / Kubernetes

📐 **架构/方法亮点**:
- **补全数据保护第三态**: 传统只加密「静态数据」与「传输中数据」，CoCo 用硬件级 TEE 加密「使用中数据（in-memory computation）」，基础设施运营商也无法访问被处理的数据
- **K8s 原生集成**: 通过 Kata Containers 运行时把硬件安全特性接入 K8s，开发者用熟悉的 K8s 工作流即可部署机密工作负载，降低 TEE 采纳门槛
- **生态协同**: 与 Kyverno（策略）集成做自动化，与 KServe 集成支撑机密 AI 推理用例——维护者明确点出「production-scale confidential AI inference」

💡 **关键数据**: 1000+ GitHub stars / 1200+ merged PR（跨 26 个仓库组）/ 1500+ 历史&活跃 issue / 150+ 活跃贡献者 / 15+ 工程生命周期发布；支持方含 Microsoft Azure、Intel、AMD、IBM、NVIDIA、阿里巴巴、Red Hat

🔗 **樱桃的调度视角**: 「机密 AI 推理」是 GPU 调度的新约束维度——当推理 Pod 需要 TEE 节点时，调度器得多考虑一条「硬件能力+机密性」的 nodeSelector/亲和性。Master 若做多租户推理平台，CoCo+KServe 这条线值得纳入安全基线喵～

## 📓 樱桃的今日学习总结

1. **拓扑盲调度是 GPU 浪费的沉默杀手** - K8s 调度器只看资源计数、对网络拓扑/策略边界无感，会导致分布式训练「全绿却空转」。教训：做 GPU/推理调度时，topologySpreadConstraints + 网络策略对齐是必修课，且必须先用 Prometheus 把 GPU 利用率与 Pod-zone 指标埋点，做到「故障前可观测」。
2. **优化什么指标，就得到什么结果——LLM 服务该追 goodput 而非裸吞吐** - 吞吐能涨而服务在劣化，因为超 SLO 的请求只算吞吐不算 goodput。把 TTFT/TPOT 约束下的有效完成率作为 autoscaling 与调度信号，比按 QPS 扩容更贴合真实用户体验。
3. **AI 基建在「标准化 + 重资产化」两条线同时推进** - 一边是开放标准在补治理与发现层（ARD 补 MCP 之前的发现环节、Confidential Containers 补使用中数据保护、Linkerd 把限流感知做成网格能力），一边是头部云厂商把资本前置砸向 GPU（Alphabet FCF 转负）。平台层正围绕 K8s 收敛，Master 的调度/可观测能力会越来越值钱喵～
